# 2607.0001 Tarmac UI Kit — standalone design-sync export

**Date:** 2026-07-01
**Status:** approved
**Author:** EP Lin
**Design doc:** [2607.0001](../../docs/designs/2607.0001_tarmac_ui_kit_design_sync_export.md)
**Issue:** #47

## Context

Designs explored in Claude Design use generic look-alike components, so what comes back drifts from Tarmac's real UI and needs a manual re-implementation pass before it's shippable. The fix is to hand the design agent our *actual* compiled presentational components + Breeze tokens, uploaded to a Claude Design project by `/design-sync`.

Scoping (in the design doc) already verified the target slice is import-clean: the 8 live `desktop/src/ui/` components (all of `ui/` except `TitleBarChip`, which is dead code and dropped) plus `desktop/src/cards/{CardShell,DocCard}.tsx` reach into no `ipc/`, `board/`, `cards/TerminalCard`, `@tauri-apps/*`, or `xterm` code. Their only non-React edges are pure `kit/*.ts` modules, `theme.ts`, and `marked`. So this is a **packaging** job, not a decoupling job.

Three things block a standalone render today: (1) the components exist only inside the Vite app graph, styled by a 1060-line `desktop/src/theme.css` entangled with board-engine + terminal rules; (2) the chrome font (`IBM Plex Mono`) is assumed system-installed and is never shipped; (3) three runtime CSS-var writers (`BoardEngine`'s `--zoom/--world-tx/--world-ty`, `docCardVars()`'s `--card-w/--card-h`, and the `.board` ancestor) don't exist when a component renders in isolation.

## Motivation

Every generated design currently costs a manual re-implementation pass against the real components. This kit buys that down by making the design agent compose with our real compiled parts. The load-bearing bet: **preview fidelity is the whole point** — if a standalone preview renders wrong (collapsed prose, wrong font, zoomed/translated frame), that error propagates into every design the agent generates, which is worse than not syncing at all. The acceptance gate therefore verifies concrete render fidelity (DocCard prose at non-zero width, IBM Plex Mono present), not just "it renders."

## Proposed Solution

Ship the five decisions (a)–(e) from the design doc. This is frontend-only: no daemon, socket, `Msg`, or conformance-vector change. The app's `tsc && vite build` → `desktop/dist/` and Tauri's `frontendDist: "../dist"` are left completely untouched; the kit builds to a **separate** `desktop/dist-kit/`.

### Overview

A new, standalone esbuild script emits a self-contained browser bundle (`tarmac-kit.js` IIFE exposing `window.TarmacKit`), one compiled stylesheet (`tarmac-kit.css`), the vendored fonts, and one `@dsCard`-marked preview HTML per component into `desktop/dist-kit/`. To keep exactly one source of truth for CSS, `theme.css` is split into `@import` partials by closure membership; the kit imports only the token+card+chrome subset (plus a kit-only preview-defaults partial), while the app's `theme.css` imports **all** partials and renders byte-identically.

### Key Components

- **(a) Build target — `desktop/scripts/build-kit.mjs`.** A Node script calling esbuild's JS API. Wired to a new `npm run build:kit` (in `desktop/package.json` scripts) and a new `.PHONY: kit` target in the root `Makefile` — **never** folded into `app`. Adds `esbuild` as an explicit `devDependency`. Emits a single IIFE `dist-kit/tarmac-kit.js` exposing `window.TarmacKit = { …8 ui components…, CardShell, DocCard, mount }` with **React, react-dom, and marked inlined** (not external). Also compiles the kit stylesheet, copies fonts, and writes the per-component preview HTML (see (e)).

- **(b) CSS split — `desktop/src/theme/*.css` partials.** Every rule moves out of `theme.css` into partials; `theme.css` becomes an ordered list of `@import`s **in the original top-to-bottom order** so cascade + specificity ties resolve identically. Partitions:
  - `tokens.css` — the ~30 design tokens, `--oversample-k`, the three engine-var *defaults* (`--zoom:1; --world-tx:0px; --world-ty:0px;`) already at `:root` today, plus the IBM Plex Mono `@font-face` from (c).
  - `card.css` — `.card` + state classes, `.card-header` children, `.card-handle*`, `.repo-dot` (kept unscoped, as today), and all `.doc-card`/`.doc-*` prose rules.
  - `chrome.css` — the 8 in-scope `ui/` component rules, including `.dock-pane`/`.dock-header`/`.dock-body` **box** styling.
  - `app-only.css` — board-engine/world-layer/edge rules, the JetBrains `@font-face`, and every terminal/reparent rule: both `.term-host` definitions (base + `.dock-body .term-host` override), `.card.docked`, `.term-raster-*`, `.term-dock-ghost`.
  - The kit stylesheet is the compiled closure of `tokens.css + card.css + chrome.css + preview-defaults.css`; the app's `theme.css` `@import`s **all** partials (tokens, card, chrome, app-only) and **never** preview-defaults.

- **(c) Font bundle — `desktop/public/fonts/`.** Vendor IBM Plex Mono Regular (400) and Bold (700) as `.woff2` (OFL, redistributable). Add their `@font-face` to `tokens.css` with `src: url('/fonts/ibm-plex-mono-*.woff2')`. Living in the shared token partial means both the kit *and* the shipping app gain the font (closing the latent silent-fallback-to-SFMono bug). JetBrains Mono is **not** bundled into the kit (no in-scope component renders a terminal surface).

- **(d) Preview neutralizations.**
  1. Engine vars: **no change beyond the split** — the `:root` `1`/`0px`/`0px` defaults ride along in `tokens.css`, so a preview with no `BoardEngine` renders unzoomed/untranslated.
  2. DocCard prose collapse: a **kit-only** partial `desktop/src/theme/preview-defaults.css` sets `:root { --card-w: 360px; --card-h: 520px; --card-x: 0px; --card-y: 0px; }`. Imported by the kit stylesheet **only**, never by `theme.css`. Without it, `--card-w/--card-h` have no `:root` fallback, `.doc-prose-sizer` width resolves to `auto`, and prose collapses.
  3. DocCard needs `.board` + `marked`: `marked` is inlined by esbuild; the DocCard preview HTML wraps the mount in `<div class="board"><div id="mount"></div></div>` so `DocCard`'s `closest('.board')` finds an ancestor (and, with nothing mutating `--zoom`, its MutationObserver correctly never fires). **No source-component edits** for (d).

- **(e) Kit output for `/design-sync`.** `npm run build:kit` writes `desktop/dist-kit/`:
  ```
  dist-kit/
    tarmac-kit.js                 # IIFE → window.TarmacKit (components + mount), React+marked inlined
    tarmac-kit.css                # compiled @import closure: tokens + card + chrome + preview-defaults
    fonts/                        # ibm-plex-mono-{regular,bold}.woff2
    components/<name>/index.html  # one per component; first line = <!-- @dsCard group="…" -->
  ```
  Each `components/<name>/index.html` is tiny + declarative: first-line `@dsCard` marker, then `<link rel=stylesheet href=../../tarmac-kit.css>` + `<script src=../../tarmac-kit.js>`, a mount node (wrapped in `.board` for DocCard), and one `TarmacKit.mount("<name>", <sampleProps>, el)` call. Sample props are hand-authored fixtures from each component's documented props interface (the real bulk of the work).

  The **10** components and their groups:
  - `Cards`: `CardShell`, `DocCard`
  - `Overlays`: `BoardSwitcher`, `ToastOverlay`, `OffscreenHints`, `MinimapOverlay`, `CycleHud`
  - `Chrome`: `StatusBar`, `ZoomControl`, `DockPane`

### Data Flow

1. `make kit` → `npm run build:kit` → `node desktop/scripts/build-kit.mjs`.
2. esbuild bundles the component entry (React/react-dom/marked inlined) → `dist-kit/tarmac-kit.js` exposing `window.TarmacKit`.
3. The kit stylesheet (`@import` closure of tokens+card+chrome+preview-defaults) is compiled/emitted → `dist-kit/tarmac-kit.css`.
4. Fonts copied → `dist-kit/fonts/`.
5. For each of the 10 components, a preview `index.html` is written under `dist-kit/components/<name>/` with the correct first-line `@dsCard group=…` marker and a `TarmacKit.mount(...)` call over its fixture props.
6. `/design-sync` consumes `dist-kit/`; the app's self-check compiles `_ds_manifest.json` from the `@dsCard` first-line markers; the kit uploads to a Claude Design project.

### Interface Contract

- **`window.TarmacKit`** (from `tarmac-kit.js`): an object with keys for all 8 in-scope `ui/` components + `CardShell` + `DocCard`, plus `mount`.
- **`TarmacKit.mount(name: string, props: object, el: HTMLElement): void`** — renders component `name` with `props` into `el` (React root under the hood). Called once per preview HTML.
- **Preview marker (fixed DesignSync contract):** each `components/<name>/index.html`'s **first line** is exactly `<!-- @dsCard group="Cards|Overlays|Chrome" -->` (well-formed HTML comment, `group` attribute one of the three group names). The app self-check compiles `_ds_manifest.json` from these first-line markers; explicit `register_assets` is legacy and not used.
- **Output tree** (verbatim): `dist-kit/tarmac-kit.js`, `dist-kit/tarmac-kit.css`, `dist-kit/fonts/ibm-plex-mono-{regular,bold}.woff2`, and `dist-kit/components/<name>/index.html` for each of the 10 components.
- **Additive-only to the app:** no `desktop/src/*.tsx` component source is edited (fixture files are new); `theme.css` becomes `@import`-only and preserves original rule order; nothing is written into `desktop/dist/`.

## Alternatives Considered

### Vite library mode (`build.lib`)
Rejected: designed for ESM/CJS npm packages consumed by another bundler, not a single global-exposing browser file with React inlined; mode-gating it into `vite.config.ts` risks perturbing the app build we must leave alone. A standalone esbuild script is a clean, separate seam.

### Duplicate token/card CSS into a kit-only stylesheet
Rejected: two copies of `.card`/tokens is exactly the drift the project exists to kill. The `@import`-partial split gives the kit its subset with zero duplication and one definition per shared selector.

### Add `--card-w/--card-h` defaults to shared `tokens.css`
Rejected in favor of a kit-only `preview-defaults.css`: keeping the neutralizing fallbacks out of the app's `:root` is the strongest guarantee that "existing app rendering unaffected" holds literally; previews are the only consumer that needs them.

### Runtime harness that fakes a `BoardEngine` to drive `--zoom`
Rejected: over-engineered. Static previews want a fixed, correct frame, not animation; the `:root` defaults already deliver that.

## Acceptance Scenarios

### Happy Path
- **S1 — build emits the full tree:** Given a clean checkout with deps installed, when `npm run build:kit` (and `make kit`) runs, then `desktop/dist-kit/` contains `tarmac-kit.js`, `tarmac-kit.css`, `fonts/ibm-plex-mono-regular.woff2`, `fonts/ibm-plex-mono-bold.woff2`, and `components/<name>/index.html` for all 10 in-scope components (`CardShell`, `DocCard`, `BoardSwitcher`, `ToastOverlay`, `OffscreenHints`, `MinimapOverlay`, `CycleHud`, `StatusBar`, `ZoomControl`, `DockPane`) — and no others.
- **S2 — global surface:** Given `tarmac-kit.js` loaded in a bare page, then `window.TarmacKit` exposes a callable `mount` plus a key for each of the 10 components; React/react-dom/marked are inlined (the page needs no import map and no second `<script>`).
- **S3 — every preview mounts from props with no daemon:** Given each `components/<name>/index.html` opened as a static file (no daemon, no socket, no Tauri), when the page loads, then the component renders its fixture props without console errors and without any network call to a socket/IPC endpoint.
- **S4 — DocCard prose renders at non-zero width:** Given `components/DocCard/index.html` (mount wrapped in `.board`) opened statically, when it renders, then the `.doc-prose`/prose container has a computed width > 0 (the `preview-defaults.css` `--card-w: 360px`/`--card-h: 520px` fallbacks apply, so `.doc-prose-sizer` does not collapse to `auto`/0), and the markdown is parsed by the inlined `marked` (prose text nodes present).
- **S5 — chrome font is real IBM Plex Mono:** Given the kit loaded statically, then `tokens.css` contains an `@font-face` for `IBM Plex Mono` with `src` pointing at `/fonts/ibm-plex-mono-*.woff2`, and those `.woff2` files exist under `dist-kit/fonts/` — so chrome text uses the bundled font, not the SFMono fallback.

### Edge Cases
- **S6 — kit stylesheet closure is tokens+card+chrome+preview-defaults only:** Given the compiled `dist-kit/tarmac-kit.css`, when its full `@import` closure is inspected, then it contains **zero** board-engine, world-layer, edge, or terminal selectors — specifically no `.term-host`, `.card.docked`, `.term-raster-*`, `.term-dock-ghost`, `.world`, or provenance-edge rules, and no JetBrains `@font-face`. (The `app-only.css` partial is not in the kit closure.)
- **S7 — each preview's first line is a well-formed `@dsCard` marker:** Given each `components/<name>/index.html`, when its first line is read, then it matches exactly `<!-- @dsCard group="G" -->` where `G` is the component's assigned group (`Cards` for CardShell/DocCard; `Overlays` for BoardSwitcher/ToastOverlay/OffscreenHints/MinimapOverlay/CycleHud; `Chrome` for StatusBar/ZoomControl/DockPane).
- **S8 — DockPane preview renders empty pane chrome:** Given `components/DockPane/index.html`, when it renders, then the `.dock-pane`/`.dock-header`/`.dock-body` box chrome is present and the `.dock-body` is an empty node (no reparented xterm host, no React children) — matching its documented imperative contract.
- **S9 — MinimapOverlay preview is correct out of the box:** Given `components/MinimapOverlay/index.html`, when it renders, then its SVG fills come from the bundled `theme.ts` `palette` (the default palette travels in the IIFE), so the minimap is correct with no CSS-var override.

### Error / Regression Scenarios
- **S10 — app render is byte-identical after the CSS split:** Given the split `theme.css` (now `@import`-only, partials in original order), when the app builds and renders, then the emitted app cascade is equivalent to before the split — no rule moved order, no shared selector duplicated, and `theme.css` imports `tokens+card+chrome+app-only` but **not** `preview-defaults.css`.
- **S11 — `make` still passes; existing tests unchanged:** Given the branch, when `make` (and `make app`, `make core`) runs, then it passes; `desktop`'s existing Vitest suite and the app-cargo (`desktop/src-tauri`) + `core` cargo tests all pass unchanged; nothing is written into `desktop/dist/` by the kit build.
- **S12 — `build:kit` is never invoked by `make app`/`make core`:** Given the Makefile, when `make app` or `make core` runs, then `build-kit.mjs` does not execute and `dist-kit/` is not produced — the kit is only built by the dedicated `make kit` / `npm run build:kit` entry point.
- **S13 — TitleBarChip is not exported:** Given `window.TarmacKit` and `dist-kit/components/`, then there is no `TitleBarChip` key and no `components/TitleBarChip/` directory (dead code with no CSS is deliberately dropped).

### Intentional App-Facing Change (decision (c))
- **S14 — the shipping app also gains real IBM Plex Mono:** Given the branch, when `make app` builds, then `desktop/public/fonts/` contains `ibm-plex-mono-regular.woff2` and `ibm-plex-mono-bold.woff2`, the app's `theme.css` cascade includes the `IBM Plex Mono` `@font-face` (inherited from the shared `tokens.css`) with `src` → `/fonts/ibm-plex-mono-*.woff2`, and those two files are emitted into `desktop/dist/fonts/` by the app build — so the shipping app renders chrome in real Plex instead of the silent `SFMono` fallback. This is the **one** deliberate app-facing rendering change (an additive `@font-face` + two font files); it is distinct from, and does not violate, the byte-identical CSS-split guarantee of S10 (the split alone changes nothing; the font is a separate, additive decision).

## For the Implementing Agent

> **Your job:** make every acceptance scenario above pass with tests/checks that would *fail if the behavior were wrong*. A green result that passes for the wrong reason does not satisfy this contract — `/verify` will hunt for vacuous checks by asking, of each behavior, "what is the smallest change that breaks this, and would any check catch it?"

Notes for this repo (per MEMORY: the React/Tauri app shell is not unit-tested — only pure `kit/`/Rust logic is):

- **S1, S2, S6, S7, S13** are the machine-checkable core. Prefer a small Node/Vitest check (or a `build-kit`-adjacent assertion) that runs `build:kit` and asserts the output tree, the `window.TarmacKit` keys, the CSS `@import` closure contents (no terminal/board selectors, no JetBrains face), and each preview's exact first-line marker. These must be non-vacuous: assert the *actual* emitted files/strings, not hard-coded copies.
- **S3, S4, S5, S8, S9** are render-fidelity gates. If a headless render harness is impractical, verify by opening the static previews (manual-QA is acceptable per repo convention for the app shell) — but S4 (DocCard prose width > 0) and S5 (IBM Plex Mono `@font-face` + font files present) are the #1-risk gates and must be explicitly confirmed, not assumed.
- **S10, S11, S12** protect the untouched-app constraint: split `theme.css` into `@import`-only partials preserving original order; keep the kit build out of `make app`/`make core`; ensure `make` + existing suites stay green and nothing lands in `desktop/dist/`.
- Do **not** edit any in-scope component `.tsx` source (fixtures are new files). Do **not** `cargo fmt`. Match surrounding code style. `make` is the source of truth.

## Definition of Done

Done is when `/verify` passes against this spec (criteria mirror the design doc's "Success criteria (observable)"):

- [ ] `make` still passes; all existing tests (`desktop` Vitest + app-cargo + `core` cargo) pass unchanged, and no component source is edited except new fixture files. The CSS split is `@import`-only and preserves cascade order, so the split alone leaves app rendering unchanged. The **only** intentional app-facing change is the additive IBM Plex Mono `@font-face` + two `.woff2` files from decision (c) (which fixes the latent SFMono fallback and adds ~120 KB to the app bundle) — there is no other change to `make app` output. — S10, S11, S14
- [ ] `npm run build:kit` and `make kit` emit `dist-kit/` containing `tarmac-kit.js`, `tarmac-kit.css`, `fonts/ibm-plex-mono-{regular,bold}.woff2`, and one `@dsCard`-marked `components/<name>/index.html` for all 10 in-scope components (and no `TitleBarChip`). — S1, S7, S13
- [ ] The kit stylesheet's `@import` closure contains **only** tokens + card + chrome + preview-defaults — no board-engine or terminal rules (no `.term-host`/`.card.docked`/`.term-raster-*`/`.world`/edge selectors, no JetBrains `@font-face`). — S6
- [ ] `window.TarmacKit` exposes the 10 components + `mount`, with React/react-dom/marked inlined (one self-contained `<script>`). — S2
- [ ] Each in-scope component renders standalone from fixture props with **no live daemon**, correct IBM Plex Mono chrome, and correct tokens — **including DocCard prose at non-zero width** (via `preview-defaults.css` `--card-w/--card-h`, not 0) and MinimapOverlay's bundled-`palette` fills. — S3, S4, S5, S8, S9
- [ ] The kit build is **not** wired into `make app`/`make core` and writes nothing into `desktop/dist/`. — S11, S12
- [ ] `/design-sync` consumes `dist-kit/`, the app self-check compiles `_ds_manifest.json` from the `@dsCard` markers, and the kit uploads to a Claude Design project. (Integration gate — confirm the manifest compiles from the emitted markers.)
- [ ] Every acceptance scenario (S1…S13) maps to at least one non-vacuous check; no implementation-quality blockers (stubs, dead code, stale docs).

## Trade-offs and Limitations

- **Second build entry is unversioned by CI.** There's no CI here and `make` won't run `build:kit`, so the kit build can rot silently until a sync fails. `make kit` is the manual gate.
- **DockPane's contract stays imperative.** Its chrome ships, but `.dock-body` is a bare DOM node the caller reparents an xterm host into; the preview renders the empty pane. Documented, not enforced (S8).
- **MinimapOverlay isn't purely CSS-themeable.** It computes SVG fills from `theme.ts`'s `palette` in JS (SVG can't read CSS vars), so reskinning requires overriding the bundled `palette`, not just CSS custom properties (S9).
- **Hand-maintained fixtures.** Sample props are authored from each component's documented props interface and are the real bulk of the work; a props-interface change silently ages a fixture.
- **App shell stays untested per repo convention** — the machine-checkable contract is the build output + CSS closure + markers; render fidelity leans on the static previews / manual-QA.
