# 2606.0011 Doc-card screen-space sizing — real-px box + prose-only scale

**Date:** 2026-06-27
**Status:** draft
**Author:** EP Lin

## Context

Predecessor **2606.0010** moved doc cards into a screen-space layer `.doc-layer` (sibling of `.world`, no transform) and made each card a per-card wrapper that carried the *entire* visual scale through one CSS-var transform — `translate(--world-tx,--world-ty) scale(--zoom) translate(x,y)` on a base-px box. That `scale()` on the whole card box is now **disproven**: a WebKit PoC (`desktop/poc/zoom-reflow-poc.html`, Panel C) shows that scaling the box bilinearly stretches the nested clip / scroll / shadow / border layers → blurry text. **2606.0010 is superseded by this spec.**

The PoC's proven condition is **Panel H**: separate the box size from the text scale, and crisp the text by **oversample → freeze → downscale** (NOT scale-up). Scaling a layer *up* (`scale(zoom)` with zoom>1) is a GPU bitmap upscale → blurry on 1× displays; the fix lays the prose out **once at K× reference** (K = max board zoom) and the prose-scaler only ever scales **down** (`scale(zoom/K) ≤ 1`) — same oversample family as the terminal canvas `rasterScale`. The card BOX is still rendered at its zoomed size in **real pixels** with **no** transform scale (translate-only position) and chrome `calc(base × zoom)`; only the innermost PROSE is the K× oversampled, down-scaled layer. Reflow stays frozen (prose laid out at K× constants, never `--zoom`). Terminal cards are crisp by canvas oversample (`rasterScale.ts`) and stay in `.world` — untouched.

## Motivation

Deliver the crispness 2606.0010 aimed for without the box-scale blur, keeping every win from 2606.0009/2606.0010: no prose reflow across zoom, provenance edges attached at all zooms, no position snap on re-render, full doc scroll reach. This is a focused ~5-file refactor of the existing `.doc-layer`, not a revert.

## Proposed Solution

Keep the `.doc-layer` screen-space layer and per-card wrapper. Change the wrapper from a `scale()`-on-box transform to: **real-px box size + translate-only position**, with an inner **prose oversampled at K× and only ever down-scaled** (`scale(zoom/K)`) as the sole scale layer.

### Key Components

Nesting (state verbatim in implementation):

```
.doc-layer (screen-space, no transform)
  └ wrapper div  — width/height = card{w,h}×zoom (REAL px), translate-only position, NO scale
      └ .card (.doc-card) — chrome metrics ×zoom (doc-scoped)
          └ .doc-scroll — overflow-y:auto, overflow-x:hidden, fills box, NO transform, NO padding
              └ .doc-prose-scaler — BARE: transform: scale(calc(var(--zoom)/var(--oversample-k))), origin 0 0  ← the only scale, always ≤1 (downscale)
                  └ .doc-prose — frozen at K× reference: font/width/padding/max-width = base×K, depends on --oversample-k & --card-w, NEVER --zoom
```

- **`desktop/src/kit/docZoom.ts`** — pure module; the unit-test seam. Functions below.
- **`desktop/src/board/Board.tsx`** — `.doc-layer` wrapper: apply box style from kit + set per-card `--card-{x,y,w,h}` vars from `frame`.
- **`desktop/src/cards/DocCard.tsx`** — add the `.doc-prose-scaler` element (carrying the prose-scaler style) between `.doc-scroll` and `.doc-prose`.
- **`desktop/src/theme.css`** — `:root { --oversample-k: <max board zoom> }`; doc-scoped zoom-scaled chrome metrics; `.doc-scroll` (no padding, `overflow-x:hidden`); `.doc-prose-scaler` rule (bare, `transform-origin:0 0`); `.doc-prose` doc-scoped K× layout (font/width/padding/max-width `calc(base × var(--oversample-k))`, width `calc(var(--card-w) × var(--oversample-k))`, `box-sizing:border-box`).
- **`desktop/src/board/BoardEngine.ts`** — **no change**; `apply()` already writes `--zoom/--world-tx/--world-ty` once per frame (verified, lines ~258/265).

### Data Flow

- **Pan/zoom hot path:** BoardEngine writes the three global vars `--zoom/--world-tx/--world-ty` only (`--oversample-k` is a static `:root` constant). Every doc card's position, box size, chrome metrics, and prose down-scale follow via `calc()` — **no per-card JS, no React re-render.**
- **Frame change (drag/resize/restore):** React re-renders the wrapper and sets `--card-{x,y,w,h}` from the committed world frame. Drag/resize math in `CardShell` is unchanged (`inWrapper` already omits `left/top`; still reads real `frame` + `getZoom()` and commits world-coord frames). No engine per-card writes.

### Interface Contract (pure fns in `docZoom.ts`)

1. `docProseLayout()` — **unchanged.** Returns the fixed base constants `{ fontSizePx:14, maxWidthPx:720, paddingVPx:18, paddingHPx:22 }`. **No zoom parameter** (sizing structurally cannot depend on zoom → wrap points frozen).

2. `docWrapperBox()` — returns the real-px box style:
   ```
   {
     width:  "calc(var(--card-w) * var(--zoom))",
     height: "calc(var(--card-h) * var(--zoom))",
     transform: "translate(calc(var(--world-tx) + var(--card-x) * var(--zoom)),calc(var(--world-ty) + var(--card-y) * var(--zoom)))",
     transformOrigin: "0 0",
   }
   ```
   **No `scale(` anywhere; no `willChange` key.** Translate-only position (translate does not stretch raster). Box origin (0,0) lands where `worldToView` projects the card's world origin (see S6), keeping edges attached.

3. `docCardVars({x,y,w,h})` — returns the four per-card custom-property entries as a style object: `{ "--card-x": "<x>px", "--card-y": "<y>px", "--card-w": "<w>px", "--card-h": "<h>px" }`. Board spreads these onto the wrapper from `frame` on every (frame-changing) render; they are the single position/size source (React-owned, render-independent).

4. `docProseScaler()` — returns the **bare** prose-scaler style: `{ transform: "scale(calc(var(--zoom) / var(--oversample-k)))", transformOrigin: "0 0" }`. Nothing but the transform — **no width** (the width lives on the K× `.doc-prose` in CSS). It is a pure **down**-scale: with K = max board zoom, `zoom/K ≤ 1` across the whole range, so the layer never upsamples → never blurs. **No `willChange`** (a pre-hint pins raster at 1× → permanent blur — the 2606.0009/0010 finding still holds).

Plus the const **`DOC_OVERSAMPLE_K`** (exported from `docZoom.ts`) = the board's max zoom, mirrored by CSS `:root { --oversample-k }`. Load-bearing invariant: **K ≥ max board zoom** (S13).

The K× prose layout (CSS, doc-scoped, `.doc-prose`): font-size `calc(14px*var(--oversample-k))`, width `calc(var(--card-w)*var(--oversample-k))`, padding `calc(18px*var(--oversample-k)) calc(22px*var(--oversample-k))`, max-width `calc(720px*var(--oversample-k))`, `box-sizing:border-box`, line-height 1.7 — depends on `--oversample-k`/`--card-w`, **never `--zoom`**, so wrap points are frozen. Net visual at zoom z: font `14·z`, width `card-w·z` (K cancels) — correct size, reached by downscale (crisp) not upscale.

(Existing `docWrapperTransform({x,y})` is removed/renamed; its tests are superseded by S1–S6.)

### Chrome scaling (theme.css), doc-scoped

Zoom-scaled metrics live under a **doc-only** selector (`.doc-layer .card` / `.doc-card`), e.g. `border-width: calc(1px * var(--zoom))`, `border-radius: calc(10px * var(--zoom))`, `box-shadow` offsets/blur `* var(--zoom)`, header `height`/`flex-basis`/`padding`/`font-size` `* var(--zoom)`. (The prose padding is NOT in this list — it lives on the K× `.doc-prose` as `calc(base × var(--oversample-k))` and is shrunk back by the scaler's `scale(zoom/K)`, so `.doc-card .doc-scroll` has no padding.) The shared `.card` rule stays **base-px** for terminals (which are scaled by the `.world` transform). Doc cards must NOT inherit a second scale; terminal chrome must NOT gain a `var(--zoom)` factor (would double-scale). Mechanism = `calc(base * var(--zoom))` on each metric (not an `em` cascade), so each is independently auditable as doc-scoped.

## Alternatives Considered

- **Whole-box `scale(var(--zoom))` (2606.0010) — disproven.** PoC Panel C: scaling the box bilinear-stretches nested clip/scroll/shadow → blur. This spec is its fix.
- **Prose-only `scale(var(--zoom))` scale-UP (earlier draft of this spec) — disproven.** In-app QA: scaling a base-layout prose layer *up* (zoom>1) is a GPU bitmap upscale → blurry on 1× displays. Replaced by oversample→freeze→downscale (PoC Panel H): lay out at K× and only ever `scale(zoom/K) ≤ 1`.
- **`em`-cascade chrome scaling (single root font-size dial) — rejected.** Couples chrome to the prose font axis and makes "doc-only" hard to assert; explicit `calc(base*--zoom)` per metric keeps the doc/terminal scoping legible and testable.
- **Per-frame JS write of each card's px width/height — rejected for hot path.** Reintroduces per-card work on pan/zoom; `calc()` off three global vars keeps the hot path at three writes. (Settle-gated px-size is a possible *optimization*, see Open Questions — not the MVP.)
- **Unify terminals onto the same mechanism — rejected (deferred).** Terminals are `<canvas>`; CSS scale only stretches the buffer. Oversample already makes them crisp; folding them in risks the terminal path for zero gain.

## Acceptance Scenarios

- **S1 — box size + translate-only position (unit):** `docWrapperBox()` returns `width`/`height` exactly the two `calc(var(--card-w|h) * var(--zoom))` strings, the exact translate-only `transform` string, and `transformOrigin:"0 0"`.
- **S2 — box carries NO scale, no will-change (unit):** `docWrapperBox().transform` contains no `scale(`; the returned object has no `willChange` key.
- **S3 — prose scaler is the sole, down-scaling layer (unit):** `docProseScaler()` returns `transform:"scale(calc(var(--zoom) / var(--oversample-k)))"` (exactly one `scale(`, divides `var(--zoom)` by `var(--oversample-k)`), `transformOrigin:"0 0"`, **no `width`** key, and **no `willChange`** key.
- **S4 — prose layout frozen (unit):** `docProseLayout()` returns the four named constants (14 / 720 / 18 / 22). Assert the "no zoom argument" property concretely: `docProseLayout.length === 0` (zero declared parameters) → sizing structurally cannot depend on zoom. Freezing the constants is **necessary but not sufficient** — the prose's containing-block WIDTH must also be zoom-independent (S12); the two together freeze wrap points.
- **S5 — per-card vars from frame (unit):** `docCardVars({x:100,y:200,w:360,h:480})` returns exactly `{ "--card-x":"100px","--card-y":"200px","--card-w":"360px","--card-h":"480px" }`.
- **S6 — box origin projects via worldToView (unit):** With the engine's `--world-tx = vpCx − cx·zoom`, `--world-ty = vpCy − cy·zoom`, the wrapper places world card-origin `(cardX,cardY)` at screen `(world-tx + cardX·zoom, world-ty + cardY·zoom)`. The test must **derive the screen point from the actual `docWrapperBox().transform` string** — substitute numeric values for `--world-tx`/`--world-ty`/`--card-x`/`--card-y`/`--zoom` into the returned `calc(...)` translate expression (via a small calc-eval helper that resolves `var()`/`*`/`+`) and read out the resulting X/Y — **not** by re-deriving `tx + cardX·zoom` by hand (a hand-derived formula passes even if `docWrapperBox` is wrong → vacuous). Assert that derived point equals `worldToView({x:cardX,y:cardY}, zoom, center, viewportCenter)` (from `desktop/src/kit/boardTransform.ts`) at **zoom 2 and zoom 0.5**. This is the same projection the `.world` CSS transform applies to world-space cards/edges (`worldToView` expresses it), so doc boxes and provenance edges stay co-located. (Reuse `worldToView`; do not duplicate it.)
- **S7 — chrome scaling is doc-scoped; terminals untouched (structural/review):** In `theme.css`, **every** `var(--zoom)` occurrence sits inside a selector whose key contains `.doc` (e.g. `.doc-layer .card`, `.doc-card`, `.doc-card .doc-scroll`, `.doc-prose-scaler`). Observable pass condition: grepping `theme.css` for `var(--zoom)` yields only doc-scoped rules — **zero** `var(--zoom)` hits in the shared `.card` rule or in any terminal/`.term-host` selector. This is the checkable form of the "do-not-touch terminals" constraint: terminal chrome metrics keep their base-px values (terminals are scaled solely by the `.world` transform), so the doc-scoped zoom factor cannot double-scale them.
- **S8 — crisp at rest (manual-QA):** Zoom a doc to 2.5–3× and let the gesture settle. **Observable pass:** glyph stems render with hard anti-aliased edges and no soft/blurred fringe — the prose is indistinguishable in sharpness from the same text at zoom 1× (matching PoC `zoom-reflow-poc.html` **Panel H** — oversample→freeze→downscale — and visibly sharper than a scale-up approach). Crispness comes from down-sampling the K× layer, never up-sampling. Transient softness *during* an active gesture is acceptable; the assertion is on the settled frame only.
- **S9 — edges attached at any zoom (manual-QA):** **Observable pass:** the dashed provenance edge's doc-end endpoint lands on the doc card's center (within a few px, no visible gap) across zoom 0.5×–3× and while panning — the doc box in `.doc-layer` and the edge in `.world` stay co-located (the projection-equivalence S6 verifies in the pure layer).
- **S10 — no snap; drag/resize preserve world coords (structural + manual-QA):** Structural guarantee (unit, via S5): `docCardVars` is a pure function of `frame`, so a re-render driven by a recency/markdown/selection change — which does not change `frame` — produces byte-identical `--card-*` vars (no-op → no position jump). **Manual-QA observable pass:** triggering a live re-render (e.g. a file edit updating the markdown) does not visibly nudge the card; dragging/resizing then commits the same world-coord `frame` as on `main` (CardShell math unchanged under `inWrapper`).
- **S11 — terminals still crisp (manual-QA, regression):** terminal cards at 2.5–3× stay crisp via oversample; `TerminalCard`/`rasterScale`/terminal `CardShell` path unchanged; doc scroll still reaches full prose at every zoom.
- **S12 — prose layout is zoom-independent (structural/review):** In `theme.css`, the `.doc-prose` rule contains **no** `var(--zoom)` — its font/width/padding/max-width depend only on `--oversample-k` and `--card-w`; the **only** doc rule carrying `var(--zoom)` in the prose subtree is the bare `.doc-prose-scaler`. So the wrap column is computed once at the K× reference width and never moves with zoom: only a resize (which changes `--card-w`) re-wraps. *(This is the invariant that catches the in-app reflow regression — if any prose-layout metric gained `var(--zoom)`, wraps would shift on zoom.)*
- **S13 — K never upsamples (unit):** `DOC_OVERSAMPLE_K >= ` the board's max zoom (assert against the same max-zoom source the engine clamps to, e.g. 3.0). Therefore `scale(var(--zoom)/var(--oversample-k)) ≤ 1` for every in-range zoom — the prose layer is only ever down-scaled, so there is no upscale blur anywhere in the zoom range. (Zoom beyond K would re-blur but is prevented by the engine's zoom clamp.)

## Trade-offs and Limitations

- **Gesture jank risk:** per-frame `calc()` of box width/height for several doc cards *could* jank an active zoom. MVP accepts this — crisp **at rest** is the contract; transient softness during motion is allowed. Escape hatch (only if measured to jank): settle-gate via the existing `onRasterScaleSettle` — cheap during motion, px-size on settle. Not in the core contract.
- **K× oversampled prose DOM:** `.doc-prose` lays out in a box K× larger (K = max zoom, ≤4) than its visual size — a few× bigger layout/raster box, traded for crispness (same bet as the terminal canvas oversample). Fine for K≤4; a much larger K would waste backing store.
- **Zoom beyond K would re-blur** (upscale resumes), but is prevented by the engine's zoom clamp (S13). Keep `--oversample-k`/`DOC_OVERSAMPLE_K` ≥ the clamp's max whenever either changes.
- **Z-interleaving deferred:** `.doc-layer` still paints above `.world`, so doc cards stack above terminals/edges regardless of `model.z` (inherited from 2606.0010; accepted — docs and terminals rarely overlap).
- App-layer JSX/CSS stays untested per repo convention; the pure fns carry the tested contract.

## Open Questions

- **Gesture perf / settle-gating:** is always-px-size smooth enough with N doc cards on screen, or is settle-gating needed? Measure in-app before optimizing.
- **Chrome-metric mechanism:** explicit `calc(base * var(--zoom))` per metric (chosen, doc-scoped/auditable) vs an `em` font-size dial (more compact, but couples to the prose axis and weakens the doc/terminal scoping). Revisit only if the calc list proves unmaintainable.
