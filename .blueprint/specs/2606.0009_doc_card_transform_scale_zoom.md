# 2606.0009 Doc-card zoom via the ancestor world scale — layout once, promote layer for crispness

**Date:** 2026-06-27
**Status:** draft
**Author:** EP Lin

## Context

`DocCard.tsx` keeps prose crisp under board zoom by **px-relayout**: it reads the settled `rasterScale` (`deriveRasterScale(zoom)`, quantized to 0.5 steps, cap 3×) and rewrites `fontSize=14·rs`, `maxWidth=720·rs`, `padding=18/22·rs`, sets `.doc-scroll` to `rs·100%`, then counter-scales `transform:scale(1/rs)`. `theme.css` `.doc-prose`/`.doc-scroll` metrics are em-based so the `rs` font dial scales them uniformly.

This re-runs **real layout** every time `rs` crosses a 0.5 step. Sub-pixel rounding shifts wrap points, so a paragraph re-wraps mid-zoom — the prose visibly reflows.

Crucially, the `.world` ancestor **already** applies `transform: scale(zoom)` (`BoardEngine.ts:258`); doc cards are absolute children, so their content is **already** visually scaled by zoom. The fix is therefore *not* a per-card scale transform — that would compose to `scale(zoom²)`, geometrically wrong. Instead: lay the card out **once** at fixed base constants (`rs=1`), and add a **static compositing-layer hint** (`transform: translateZ(0)`) on the prose subtree so WebKit re-rasterizes that layer crisply under the ancestor scale after the gesture settles. Layout never changes → wrap points frozen; the promoted layer re-rasters → crisp.

`TerminalCard` still consumes `rasterScale` (oversample path) and the `BoardEngine` settle machinery; **only DocCard's consumption is dropped.** `deriveRasterScale`/`rasterScale.ts`/`rasterScale.test.ts` stay untouched.

## Motivation

Reflow-on-zoom is the one visible failure of the current doc card: text that was readable at the wrap you chose re-wraps as you zoom. Laying out once eliminates reflow *by construction* and is simpler — no quantization, no counter-scale, no scroll-fraction restore on `rs` change.

## Proposed Solution

### Overview

Render prose **once** at fixed metrics, inside a `.doc-prose-scaler` wrapper that carries only a static layer-promotion hint (`translateZ(0)`, `transform-origin: 0 0`, **never `will-change`**). The ancestor `.world` scale provides the zoom; the promoted layer re-rasters crisply. Push the fixed layout + the (zoom-independent) scaler style into a pure `desktop/src/kit/docZoom.ts` so the contract is unit-testable; DocCard becomes a thin consumer.

### Interface Contract (`desktop/src/kit/docZoom.ts`)

```ts
export const DOC_PROSE_FONT_SIZE_PX = 14;
export const DOC_PROSE_MAX_WIDTH_PX = 720;
export const DOC_SCROLL_PADDING_V_PX = 18;
export const DOC_SCROLL_PADDING_H_PX = 22;

/** The ONE layout the prose is rendered at — no zoom parameter, so it is
 *  structurally impossible for sizing to depend on zoom. */
export function docProseLayout(): {
  fontSizePx: number; maxWidthPx: number; paddingVPx: number; paddingHPx: number;
};

/** The prose-subtree wrapper style: a STATIC compositing-layer hint, ZOOM-INDEPENDENT
 *  (takes no zoom argument). transform = "translateZ(0)" (layer promotion, NOT a
 *  visual scale — the ancestor .world supplies the zoom). origin "0 0".
 *  MUST NOT contain `willChange` (a pre-hint pins the layer raster at 1× → blur). */
export function docScalerStyle(): { transform: string; transformOrigin: string };
```

DocCard: render `.doc-prose` (fixed metrics from `docProseLayout()`) inside a `.doc-prose-scaler` styled by `docScalerStyle()`; **delete** the `rasterScale` prop, the `fontSize/maxWidth/padding·rs` inline styles, the `rs·100%` sizing, and the `1/rs` counter-scale. `Board` stops passing `rasterScale` to `DocCard`. `theme.css` `.doc-scroll`/`.doc-prose` lose their "scaled inline by `rs`" comments and bake the fixed base metrics; add `.doc-prose-scaler { transform: translateZ(0); transform-origin: 0 0 }` (never `will-change`).

## Acceptance Scenarios

### Happy Path
- **S1 — layout sizing is zoom-independent:** Given `docProseLayout()`, when called, then it returns exactly the four named constants and takes no zoom argument.
- **S2 — scaler emits no zoom-dependent scale:** Given `docScalerStyle()`, then `.transform` does **not** match `/scale\(/` — it is a static layer hint (`translateZ(0)`). The card is never per-card-scaled (the ancestor `.world` owns the zoom).
- **S3 — scaler output is deterministic (no zoom coupling):** Given repeated calls to `docScalerStyle()`, when invoked any number of times, then every call returns a deeply-equal object — it takes no input, so nothing (including board zoom) can vary its output. This is the runtime guard that the scaler can never become zoom-dependent. *Architectural consequence (verified by code review + the S2 no-`scale()` assertion, not by a runtime unit test): with no per-card scale, net on-screen prose scale comes from the `.world` transform alone (= `zoom`), never `zoom²`.*

### Edge Cases
- **S4 — structural: `rasterScale` is not a DocCard input:** `DocCard`'s prop type does not include `rasterScale`, and `Board` does not pass it to `DocCard` (the `TerminalCard` path keeps it). *Verified by `tsc` (the prop is removed from `DocCardProps`, so any residual `rasterScale={…}` on `DocCard` becomes a compile error) plus code review — this is app-layer JSX, untested by repo convention, so it is NOT a `docZoom.test.ts` case.*
- **S5 — origin invariant:** `docScalerStyle().transformOrigin === "0 0"` (matches the world-transform origin).

### Error / Guard
- **S6 — never emits will-change (inline):** `"willChange" in docScalerStyle() === false`. *(Load-bearing regression guard — see below.) Scope: this unit test covers only the JS inline style. A `will-change` added to the `.doc-prose-scaler` rule in `theme.css` would equally pin the layer raster at 1× and is INVISIBLE to this test — that path is guarded by DoD item 4 (code review) + the manual-QA crispness gate.*
- **S7 — no `rs`-scaled box metrics:**
  - **(a) Unit (kit):** `docProseLayout()` returns exactly the four named constants with no zoom-derived arithmetic (assert each field equals its named constant).
  - **(b) App-layer (review + `tsc`, NOT a `docZoom.test.ts` case):** `DocCard`'s prose/scroll inline styles carry no `rs`-multiplied `width`/`height`/`padding`/`fontSize`/`maxWidth`; the metrics DocCard applies are read from `docProseLayout()` so the tested pure fn is the single source of truth.

## Trade-offs and Limitations

- **Crispness ceiling vs. memory.** A promoted layer re-rasters at the display-implied resolution; at extreme zoom WebKit may cap layer raster size — acceptable, the board clamps zoom to 3×. No `will-change` is the firm cap: it trades a one-frame re-raster latency on settle for correctness (crisp), invisible because the world transform shows the scaled bitmap during the gesture.
- **App-layer wiring (DocCard JSX) stays untested** per repo convention; the structural guarantee is that sizing + the scaler hint derive from the tested `docZoom` pure functions.

## Alternatives Considered

- **Keep rasterScale px-relayout (status quo) — rejected.** Re-running layout per 0.5 step is exactly what reflows wrap points; em-metrics and scroll-fraction restore are complexity that only exists to chase that moving layout. Laying out once deletes the failure mode rather than mitigating it.
- **Per-card `transform: scale(zoom)` — rejected (geometrically wrong here).** The `.world` ancestor already scales by zoom, so an additional per-card `scale(zoom)` composes to `scale(zoom²)`. The standalone PoC had no world transform, so it scaled the layer directly; in this app the ancestor supplies the scale and the card only needs layer promotion.
- **Direction D — finer rasterScale steps (e.g. 0.1) — insufficient.** Smaller steps reduce *how often* reflow happens but never eliminate it: any px-relayout at a new font size is subject to sub-pixel wrap rounding, so wrap points still drift, just more often and by less. It also multiplies re-raster churn. Laying out once makes reflow impossible; D only makes it smaller.

## Open Questions

- **#1 RISK — layer re-rasters under *ancestor* scale.** The PoC validated "a static `transform: scale` re-rasters crisply" by scaling the layer itself. This app instead relies on "a `translateZ(0)`-promoted layer re-rasters when its ANCESTOR (`.world`) scales." Same compositor mechanism in theory, but not the exact path the PoC exercised — so crispness-under-ancestor-scale is a **manual-QA gate** in the real WKWebView (above), not a unit test. If it fails, fall back options (per-card net-`zoom` scale with the world scale factored out, or keeping a coarse rasterScale only for crispness) must be reconsidered.
- **Zoom source: live vs. settled.** Moot for the pure functions (both are zoom-independent now). The layer re-raster latency is governed by WebKit's own settle, not `RASTER_SCALE_SETTLE_MS`; confirm timing feels right in QA.
