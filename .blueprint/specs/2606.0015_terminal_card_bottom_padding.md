# 2606.0015 Terminal card bottom padding

**Date:** 2026-06-30
**Status:** implemented
**Author:** EP Lin
**Issue:** [#36](https://github.com/.../issues/36) — last line of terminal output sits flush against the card's bottom border

## Context

The last line of terminal output reads as flush against the terminal card's bottom border, hurting readability and polish. `.term-host` (the div wrapping the xterm.js surface) already carries `padding: 8px 10px` (`desktop/src/theme.css:334–342`, with `.term-host .xterm { height: 100% }`), but the padding renders the same color as the terminal background, so the existing 8px reads as cramped rather than as deliberate breathing room. The user wants a distinct bottom gap: keep the top at 8px, increase the bottom to 16px.

## Motivation

A visible, asymmetric bottom gap makes the terminal card feel finished and keeps the final line legible. The concern with adding padding is clipping — but `@xterm/addon-fit`'s `proposeDimensions()` reads the parent element (`.term-host`) **content** height and subtracts padding measured off `this._terminal.element` (`.xterm`, which has zero padding). Because the padding lives on `.term-host`, it is already reflected in `.term-host`'s content height, so FitAddon already accounts for it. Adding more bottom padding there reduces the computed rows and **cannot** clip content. This was verified against the addon's sizing logic.

## Proposed Solution

### Overview

Increase `.term-host`'s bottom padding to 16px and keep the TS rasterScale logic byte-in-sync so oversampling still recomputes an identical grid.

### Key Components

| Path | Change |
|---|---|
| `desktop/src/theme.css` (`.term-host`, `:334`) | `padding: 8px 10px` → `padding: 8px 10px 16px` (top 8, sides 10, bottom 16). |
| `desktop/src/cards/TerminalCard.tsx` (`:34–36`) | Add `HOST_PADDING_BOTTOM_PX = 16` beside the existing `HOST_PADDING_V_PX` (8) / `HOST_PADDING_H_PX` (10). The comment at L34 mandates the TS constants stay byte-in-sync with the `theme.css` value. |
| `desktop/src/cards/TerminalCard.tsx` (`:329–341`) | The rasterScale `useLayoutEffect` builds `host.style.padding` from these constants × `rs`. Emit the 4-value form scaling the bottom by the same `rs` so the oversampled grid stays cols×rows-identical to `rs = 1`. |

The padding-string construction is the one genuinely pure, unit-testable slice. Optionally extract it into a helper in `desktop/src/kit/` and add a Vitest test asserting the string for `rs = 1` and `rs = 2` — recommended, but optional (see DoD). The visible-gap and no-clip facts are CSS/layout outcomes, not pure logic, so they are verified by build + eye.

## Acceptance Scenarios

- **S1 — visible bottom gap.** *Given* a terminal card filled with output at rest (`rasterScale = 1`, board zoom = 1), *when* it is viewed, *then* there is a visible ≥ 16 CSS px gap between the last text line and the card's bottom border. Manual QA via `make run`.
- **S2 — no clip.** *Given* the reduced content height, *when* FitAddon computes rows, *then* no terminal content is cut off at the bottom and the cursor on the last row is fully visible. Manual QA.
- **S3 — oversample invariant.** *Given* `rasterScale > 1` has settled, *when* host padding scales by `rs` alongside `fontSize`, *then* FitAddon recomputes the **same** cols×rows as `rs = 1` — no spurious PTY resize, no one-frame grid blow-up. Verifiable by reading the `useLayoutEffect` (+ optional unit test).
- **S4 — docked.** *Given* the terminal is docked (`rs` forced to 1), *when* the dock view renders, *then* padding resets to the `rs = 1` values and the dock view is unaffected. Manual QA.
- **S5 — sync invariant.** *Given* `theme.css` and `TerminalCard.tsx`, *when* both are read, *then* they encode the same numbers (8 top / 10 sides / 16 bottom). Verifiable by reading both.
