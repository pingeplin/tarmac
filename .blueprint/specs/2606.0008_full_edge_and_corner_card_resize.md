# 2606.0008 Full-Edge + Corner Card Resize (No Focus Gate)

**Date:** 2026-06-27
**Status:** draft
**Author:** EP Lin

## Context

In the Tauri/desktop UI, cards resize only from their **four corners**, and only
when the card is the active (focused/selected) one. Corner handles are gated by
`showsHandles(chrome) = focused || selected` (`desktop/src/kit/cardChrome.ts:39`,
`desktop/src/cards/CardShell.tsx:160` — **pre-implementation line numbers; will
drift after this change lands**). Geometry is the pure pinned-opposite-edge
`resizeFrame(start, corner, dxWorld, dyWorld)` (`desktop/src/kit/resize.ts:15-46`,
**pre-implementation**),
where `Corner = "tl"|"tr"|"bl"|"br"` always drives **both** axes.

## Motivation

Corner-only resize is fiddly — adjusting width *or* height alone forces a
diagonal drag and an eyeball correction on the other axis. Requiring focus first
adds a click before any resize. Full-edge resize with no focus gate is the
standard direct-manipulation expectation and removes both frictions, while
preserving Design Principle #2 (cards never steal keyboard focus from the prime
terminal).

## Proposed Solution

### Overview

Generalize the resize handle set from 4 corners to **8 handles** (4 edges + 4
corners) and make resize hit-areas **always live** regardless of focus. Edges
resize a single axis; corners keep today's dual-axis behavior. There are **no
visible grip chips** — the resize cursor on hover is the only affordance; the
focus ring stays focus-gated.

### Key Components

- **`resize.ts` (pure geometry)** — add `Edge = "t"|"b"|"l"|"r"` and
  `Handle = Corner | Edge`. `resizeFrame` gates its width block behind
  `activeX` (`handle !== "t" && handle !== "b"`) and its height block behind
  `activeY` (`handle !== "l" && handle !== "r"`); per-axis min clamp. Existing
  corner branch logic is unchanged.
- **`CardShell.tsx` (rendering + pointer)** — render 8 handles; widen
  `Corner → Handle` on the drag snapshot and `onHandlePointerDown`; preserve
  `e.stopPropagation()` + `setPointerCapture` on every handle. `hasClose` cards
  keep the `tr` corner suppressed and right-trim the `t` edge strip.
- **`theme.css`** — add `.t/.b/.l/.r` edge classes (**6px** strip thickness;
  insets leave corner hit-areas clear; edges below corners in z-order;
  `ns-resize`/`ew-resize` cursors). No visible grip chip is drawn; the handle
  `div` hit-area is always pointer-live and never hover-gated (this is what
  makes idle-card resize work in S9/S15).
- **`cardChrome.ts`** — decouple handle availability from `focused || selected`;
  keep `borderRole` on the focus-ring predicate so idle cards get **no** ring.

### Data Flow

1. Pointerdown on a handle snapshots `{startPx, startPy, frame, handle}`, captures
   the pointer, and stops propagation (no board select, no focus change).
2. Pointermove computes world delta `(clientX - startPx) / getZoom()` (live zoom
   read, `Board.tsx:93`) and calls `onResize(resizeFrame(...))` → `setActiveCards`.
3. Pointerup releases capture; persistence fires via the existing `boards`
   debounce (`App.tsx:391-394`, 200ms). `onResizeEnd` stays a no-op.

### Interface Contract

```ts
// desktop/src/kit/resize.ts
export type Corner = "tl" | "tr" | "bl" | "br";
export type Edge   = "t"  | "b"  | "l"  | "r";
export type Handle = Corner | Edge;

export function resizeFrame(
  start: WorldFrame, handle: Handle,
  dxWorld: number, dyWorld: number,
  minW?: number, minH?: number,
): WorldFrame;
// Pinned-opposite-edge per active axis; min clamp per active axis only.
```

Additive-only: no wire-protocol change; frame state stays client-side React.

## Acceptance Scenarios

Drags are in **world** px on a card starting `{x:0, y:0, w:200, h:200}`;
`MIN_CARD_W=160`, `MIN_CARD_H=90`.

### Happy Path — Edges (single-axis)
- **S1:** Given the card, when `resizeFrame` is called with `"r"` and `dx=+50`, then `w=250` and `x`, `y`, `h` are unchanged.
- **S2:** Given the card, when called with `"l"` and `dx=-50`, then `x=-50`, `w=250`, and `y`, `h` are unchanged.
- **S3:** Given the card, when called with `"b"` and `dy=+50`, then `h=250` and `x`, `y`, `w` are unchanged.
- **S4:** Given the card, when called with `"t"` and `dy=-50`, then `y=-50`, `h=250`, and `x`, `w` are unchanged.

### Happy Path — Corners (regression)
- **S5:** Given the card, when called with each corner (dual-axis drag), then it produces the following frames (top/left handles shift origin; opposite edge pinned; MIN_CARD_W=160, MIN_CARD_H=90):
  - `"br"` on `{x:0, y:0, w:200, h:100}` + `(dx=50, dy=30)` → `{x:0, y:0, w:250, h:130}`
  - `"tl"` on `{x:100, y:100, w:200, h:100}` + `(dx=-50, dy=-20)` → `{x:50, y:80, w:250, h:120}`
  - `"tr"` on `{x:0, y:0, w:200, h:200}` + `(dx=50, dy=-30)` → `{x:0, y:-30, w:250, h:230}`
  - `"bl"` on `{x:0, y:0, w:200, h:200}` + `(dx=-30, dy=50)` → `{x:-30, y:0, w:230, h:250}`

### Edge Cases — Per-axis min clamp
- **S6:** Given the card, when called with `"t"` and `dy=+150` (past min height), then `h=90`, `y=110` (`start.y + (start.h - 90)`), and `w`, `x` are unchanged — the inactive axis is untouched.
- **S7:** Given the card, when called with `"l"` and `dx=+100` (past min width), then `w=160`, `x=40` (`start.x + (start.w - 160)`), and `h`, `y` are unchanged.
- **S8:** Given the card, when called with `"r"` and `dx=-100` (past min width), then `w=160` and `x`, `y`, `h` are unchanged.
- **S16:** Given a corner `"tl"` on `{x:100, y:100, w:200, h:100}` with `(dx=300, dy=-20)` (width axis hits min, height axis grows freely), then width clamps to 160 (`x=140`), height grows to 120 (`y=80`): result `{x:140, y:80, w:160, h:120}` — each axis clamps independently.
- **S17:** Given any handle with `(dx=0, dy=0)`, then `resizeFrame` returns a frame byte-for-byte identical to the start frame (no mutation on zero delta).

### Behavior — No focus requirement
- **S9:** Given an idle card that was never clicked (`focused=false, selected=false`), when an edge or corner handle is dragged, then the card resizes (frame state updates on pointermove).
- **S10:** Given that same idle-card resize gesture, when it completes, then `document.activeElement` is unchanged (handler never calls `.focus()`/sets `tabIndex`) and the card shows **no** focus ring (`borderRole` stays on the `focused || selected` predicate).
- **S11:** Given a card whose hit-areas are live, no grip chip is ever displayed — there is no `.card-handle::after`. The handle `div` hit-area itself is always live and never hover-gated; on hover the resize cursor is the sole affordance.

### Edge Cases — Conflict avoidance
- **S12:** Given board zoom, when a handle is dragged, then world delta = `screenDelta / getZoom()` (live read): at `zoom=0.25` a 4-screen-px drag = 16 world-px; at `zoom=2` a 20-screen-px drag = 10 world-px.
- **S13:** Given a non-selected card, when one of its handles receives pointerdown, then the board-background handler does not run and another card's `selectedId` is not cleared (`stopPropagation` on all 8 handles).
- **S14:** Given a `hasClose` card, when the close button is clicked, then close fires (not resize) — the `tr` corner is suppressed and the `t` edge strip is right-trimmed to clear the close button.
- **S15:** Given a drag that leaves the card's bounding box, when the pointer continues moving, then resize continues (pointer captured on the active handle).

## Alternatives Considered

### Keep focus gate, add edges only
Rejected — the focus-first click is itself the friction users hit most; the
research confirms removing it does not violate Principle #2 as long as the ring
predicate stays separate.

### Visible grip chips (always-on or hover-revealed)
Rejected in favor of no chips at all — chips clutter the board (always-on) or add
visual noise on hover for an affordance the resize cursor already conveys. Hit-areas
stay live regardless; the cursor change is the sole, sufficient affordance.

## Trade-offs and Limitations

- Edge strips occupy 6px of the card's outer rim; the header drag-region starts
  just below the top strip ("edge owns the outer rim"). The top 6px of the
  header are no longer a move-grab — **accepted**.
- Left/right strips overlap the header band below the top corners; a header drag
  must start inboard of the side strips.
- The desktop component/pointer layer (S9, S12, S13, S15) is not unit-tested by
  design and relies on manual `make run` QA.

## Open Questions

- None — all decisions resolved (handle visibility, top-band ownership, 6px strip width).
