# 2606.0003 Per-Channel Daemon Socket & State Namespacing

**Date:** 2026-06-16
**Status:** approved
**Author:** EP Lin
**Design doc:** [2606.0003](../../docs/designs/2606.0003_per_channel_daemon_socket_and_state_namespacing.md)

## Context

A `make run` dev build and the installed released `Tarmac.app` collide on one socket and one state file. All three binaries resolve the same default path with the same two-line shape — explicit env override wins verbatim, else a fixed per-user default:

```rust
// core/crates/tarmacd/src/main.rs:11 (socket_path; state_path at :19 is identical with state.json)
fn socket_path() -> PathBuf {
    if let Some(p) = std::env::var_os("TARMAC_SOCKET") { return PathBuf::from(p); }
    let home = std::env::var_os("HOME").map(PathBuf::from).unwrap_or_else(|| PathBuf::from("/"));
    home.join("Library/Application Support/tarmac/tarmacd.sock")
}
```

The same logic is duplicated in `tarmac-cli/src/main.rs:27` and, in Swift, in `DaemonClient.resolveSocketPath()` (`app/Sources/TarmacKit/DaemonClient.swift:40`). There is **no channel component** anywhere, so a debug build and a release build resolve byte-identical paths and fight over one `tarmacd.sock` + one `state.json` (`claim_socket` makes the second daemon `exit(1)`; both apps share one state). The two Rust resolvers are not irreducibly separate: `tarmacd` and `tarmac-cli` already depend on the shared `tarmac-protocol` crate (which already hosts pure helpers like `repo_color_index` and constants `PROTOCOL_VERSION`/`MAX_FRAME_LEN`), so the Rust path derivation belongs there **once**; only the Swift↔Rust split is irreducible (no code crosses the language boundary).

The design doc settles the approach: make the **derived-default branch** channel-aware, where the channel is the binary's own build configuration (`cfg!(debug_assertions)` in Rust, `#if DEBUG` in Swift) routed through an explicit `Channel` enum. Release keeps today's flat path byte-for-byte (zero migration); debug resolves under a `dev/` subdir. `make run` additionally sets a per-worktree `TARMAC_SOCKET`/`TARMAC_STATE` (env-only) so concurrent worktrees don't share the `dev/` channel.

## Motivation

The **only** behavior that must be unit-tested is the **pure path derivation** — given an override value, a home directory, and a channel, what path comes out. That is where every correctness property lives: release equals the legacy path verbatim (backward compat), dev inserts exactly `dev/` (isolation), an explicit override still wins in *both* channels (the integration harness and `make run` depend on it), and the dev path stays within the `sockaddr_un` byte budget. Per project convention these decisions belong in a pure function — `TarmacKit` for Swift, a plain pure `fn` in the shared `tarmac-protocol` crate for Rust (called by both `tarmacd` and `tarmac-cli`) — table-driven unit-tested, leaving the impure shell (env read, `cfg`/`#if` mapping, `create_dir_all`, the Makefile, the startup log) integration/GUI-verified. Specifying the pure functions is what makes the change testable at all; the I/O around them is not.

This is the same seam as [2606.0002](./2606.0002_bundled_daemon_path_and_pty_path_injection.md): the override is passed *into* the pure function as a parameter (mirroring `resolveDaemonPath(env:...)`), so override-vs-default precedence is unit-testable in both build configs without compiler tricks.

## Proposed Solution

### Overview

Add one pure resolver per language that takes `(override, home, channel)` and returns the path — the Rust one **once** in the shared `tarmac-protocol` crate (both Rust binaries already depend on it), the Swift one in `TarmacKit`. Internally each resolver consults the override first and, in the default branch, delegates the per-channel directory to a single shared `channel_dir(home, channel) -> PathBuf`, then joins the filename it needs (`tarmacd.sock` / `state.json`); the override is consulted in the resolver **before** the `channel_dir` default, so override-vs-default precedence stays a pure, parameterized, unit-testable decision (S3/S4/S9). `channel_dir` itself does **not** take the override — it only produces the per-channel directory. Each binary's existing resolver becomes a thin shell: read the env override, read `HOME`/`NSHomeDirectory()`, map build-config → `Channel` (the single audited impure line), and call the shared resolver. The `dev/` segment is inserted **only** in the default branch, **after** the override is consulted, so explicit `TARMAC_SOCKET`/`TARMAC_STATE` keep winning verbatim and every existing test is untouched. The Rust daemon uses both the socket and the state resolver; the CLI and the app use the socket resolver only (the app holds no on-disk state; the CLI connects).

### Key Components

- **`ChannelPaths` (Swift, new file `app/Sources/TarmacKit/ChannelPaths.swift`)** — a `public enum` namespace of pure `static func`s, mirroring `DaemonLaunch`. Holds `Channel`, `socketPath(override:home:channel:)`, and `channelLabel(_:)`. No `ProcessInfo`, no `FileManager`. Unit-tested in `app/Tests/TarmacKitTests/ChannelPathsTests.swift`.
- **`DaemonClient.resolveSocketPath()` (Swift, impure shell — not unit-tested)** — reads `TARMAC_SOCKET` and `NSHomeDirectory()`, picks `Channel` behind `#if DEBUG`, returns `ChannelPaths.socketPath(...)`. The `connectFailed` error string gains the resolved path + channel label.
- **`tarmac-protocol` Rust pure resolvers (shared crate, `core/crates/tarmac-protocol/src/lib.rs`)** — the Rust path derivation lives **once** here (both `tarmacd` and `tarmac-cli` already depend on this crate, which already hosts pure helpers like `repo_color_index` and constants `PROTOCOL_VERSION`/`MAX_FRAME_LEN`): `pub enum Channel { Release, Dev }`, the shared `pub fn channel_dir(home, channel) -> PathBuf` (the per-channel directory), the override-aware `resolve_socket_path(over, home, channel)` / `resolve_state_path(over, home, channel)` (each early-returns a non-empty `over` verbatim, else `channel_dir(home, channel).join("tarmacd.sock" | "state.json")`), and `channel_label(channel)`. The `dev` literal therefore exists once and is shared-by-construction across both Rust binaries. Unit-tested **once** in this crate's `#[cfg(test)] mod`.
- **`tarmacd` shell (Rust, `core/crates/tarmacd/src/main.rs`)** — `socket_path()`/`state_path()` read the env + `HOME`, map `cfg!(debug_assertions)` → `Channel` (the one audited line), and call `tarmac_protocol::resolve_socket_path(...)` / `resolve_state_path(...)`; no per-binary copy of the resolver. The startup `info!` log names the resolved channel + socket path.
- **`tarmac-cli` shell (Rust, `core/crates/tarmac-cli/src/main.rs`)** — NOT a second copy of the resolver: `socket_path()` reads the env + `HOME`, maps `cfg!(debug_assertions)` → `Channel`, and calls `tarmac_protocol::resolve_socket_path(...)` (socket only); diagnostics use `tarmac_protocol::channel_label(...)`. Because the resolver and its `dev` literal live in the shared crate, they are identical to `tarmacd`'s by construction — no separate literal to pin. The `HELP` text (`:18`) and the "no daemon running (socket: …)" message (`:87`) gain the channel.
- **`Makefile` `run` target** — derives a stable per-worktree `TARMAC_SOCKET`/`TARMAC_STATE` from `$ROOT`, layered under `dev/`. Env-only; rides the verbatim-override path.
- **`core/Cargo.toml`** — an inline comment forbidding `[profile.*]` overrides of `debug-assertions` (such an override would silently flip the channel without touching path code).

### Interface Contract

```swift
public enum ChannelPaths {
    /// Build channel. `.release` is the shipped, signed bundle; `.dev` is any debug build.
    public enum Channel { case release, dev }

    /// Resolve the daemon socket path. PURE — no env, no disk.
    /// - `override`: the TARMAC_SOCKET value (nil if unset). Wins VERBATIM iff non-nil
    ///   AND !isEmpty (no trimming) — matching today's `resolveSocketPath` `!p.isEmpty`.
    /// - `home`: NSHomeDirectory().
    /// - `channel`: `.dev` for `#if DEBUG` builds, `.release` otherwise.
    /// Default (no override) = home + "/Library/Application Support/tarmac"
    ///   + (channel == .dev ? "/dev" : "") + "/tarmacd.sock".
    /// `.release` therefore returns today's flat path BYTE-FOR-BYTE (zero migration).
    public static func socketPath(override: String?, home: String, channel: Channel) -> String

    /// Human label for diagnostics: .release -> "release", .dev -> "dev".
    public static func channelLabel(_ channel: Channel) -> String
}
```

```rust
/// Build channel. Release == shipped bundle (cfg!(debug_assertions)==false);
/// Dev == any debug build (cfg!(debug_assertions)==true).
pub enum Channel { Release, Dev }

/// PURE socket resolver. `over` = TARMAC_SOCKET, where the shell passes
/// `var_os(...).filter(|v| !v.is_empty())` — so `Some` means present AND
/// non-empty, and wins VERBATIM (empty is treated as unset, unified with Swift).
/// Default = home + "Library/Application Support/tarmac"
///   + [Dev => "dev"] + "tarmacd.sock". Release == today's path byte-for-byte.
pub fn resolve_socket_path(over: Option<std::ffi::OsString>, home: &std::ffi::OsStr, channel: Channel) -> std::path::PathBuf

/// PURE state resolver (called by tarmacd only). Same shape with "state.json".
pub fn resolve_state_path(over: Option<std::ffi::OsString>, home: &std::ffi::OsStr, channel: Channel) -> std::path::PathBuf

/// PURE per-channel directory helper. Both resolvers SHARE this for the
/// default branch: each returns `over` verbatim when Some(non-empty), else
/// `channel_dir(home, channel).join("tarmacd.sock" | "state.json")`. The
/// override check stays in the resolver (not the shell) so S3/S4/S9
/// precedence is unit-testable; channel_dir carries the channel segment
/// for BOTH filenames, which guarantees S7 (state carries the same segment
/// as the socket). channel_dir takes NO override — it only produces the dir.
/// Default = home + "Library/Application Support/tarmac" + [Dev => "dev"].
pub fn channel_dir(home: &std::path::Path, channel: Channel) -> std::path::PathBuf

/// Human label for diagnostics: Release => "release", Dev => "dev".
/// Mirrors Swift `channelLabel`; unit-tested (S10). Lives once in
/// tarmac-protocol; both binaries' impure "no daemon" / startup messages format it in.
pub fn channel_label(channel: Channel) -> &'static str
```

The `dev/` literal is shared-by-construction across the two Rust binaries (it lives once in `tarmac_protocol::channel_dir`); the Swift `ChannelPaths` copy mirrors it independently across the language boundary and MUST be pinned by a test that asserts it equals the Rust literal, since the Swift↔Rust split is the only place the literal is duplicated. A drift between Swift and Rust splits the channel and the app cannot reach its daemon (`docs/architecture.md:81` per-channel invariant).

## Acceptance Scenarios

Scenarios target the pure resolvers. Unless noted **[Rust+Swift]**, a scenario applies to both the Swift `ChannelPaths.socketPath(override:home:channel:)` and the Rust `tarmac_protocol::resolve_socket_path`; **[Rust state]** scenarios apply to `tarmac_protocol::resolve_state_path` (exercised by the daemon; the resolver itself is shared-crate, unit-tested once).

### Happy Path
- **S1 (release == legacy, backward-compat guard):** Given `override` unset, `home = "/Users/eplin"`, `channel = .release`, when the socket resolver runs, then it returns exactly `"/Users/eplin/Library/Application Support/tarmac/tarmacd.sock"` — byte-for-byte today's path, so existing release users are never migrated. **[Rust+Swift]**
- **S2 (dev inserts the channel segment):** Given `override` unset, `home = "/Users/eplin"`, `channel = .dev`, then it returns exactly `"/Users/eplin/Library/Application Support/tarmac/dev/tarmacd.sock"`. **[Rust+Swift]**
- **S3 (override wins, release channel):** Given `override = "/tmp/x.sock"` (Swift) / `Some("/tmp/x.sock")` (Rust), any `home`, `channel = .release`, then it returns `"/tmp/x.sock"` verbatim — the explicit override is untouched by channel logic.  **[Rust+Swift]**

### Edge Cases
- **S4 (override wins EVEN in the dev channel — the channel never shadows an override):** Given `override = "/tmp/x.sock"`, `channel = .dev`, then it returns `"/tmp/x.sock"` verbatim. This is the load-bearing guard: the integration harness (`tests/common/mod.rs:69`, the `.env("TARMAC_SOCKET", …)` injection) and `socket_claiming` (`m0_integration.rs:161`) run **debug** builds with an injected `TARMAC_SOCKET` and must bypass the `dev/` insertion entirely. **[Rust+Swift]**
- **S5 (dev differs from release only by the inserted segment):** Given identical `override`-unset and `home`, then `socket(.dev)` equals `socket(.release)` with exactly `"/dev"` inserted immediately before `"/tarmacd.sock"` and nothing else moved — pins the token name and that release is otherwise unchanged. **[Rust+Swift]**
- **S6 (state, release — daemon):** Given `override` unset, `home = "/Users/eplin"`, `channel = .release`, then `resolve_state_path` returns `"/Users/eplin/Library/Application Support/tarmac/state.json"`. **[Rust state]**
- **S7 (state, dev — daemon):** Same inputs with `channel = .dev` → `"/Users/eplin/Library/Application Support/tarmac/dev/state.json"`. The state resolver MUST carry the identical channel segment as the socket resolver, or the dev daemon binds a dev socket but reads release state. **[Rust state]**
- **S8 (`sockaddr_un` byte boundary on the dev path):** The dev default appends a fixed **52-byte** suffix to `home`: `/Library/Application Support/tarmac/dev/tarmacd.sock` (count: `/Library`=8, `/Application Support`=20, `/tarmac`=7, `/dev`=4, `/tarmacd.sock`=13 → 52). So the byte length is `home.count + 52`. Given `override` unset, `channel = .dev`, and a `home` of length **51**, the resulting socket string is **103** bytes — accepted, because `connect` guards `len < sun_path.count` and macOS `sun_path` is **104** bytes, so the accept boundary is `len <= 103` (`DaemonClient.swift:180`, `guard len < dst.count`). Given a `home` of length **52**, it is **104** bytes — rejected as `socketPathTooLong` (`104 < 104` is false; `DaemonClient.swift:185–187`). What each language's unit test asserts: the **pure** resolver only emits a string, so both Swift (`utf8.count`) and Rust (`.as_os_str().len()`) assert the exact byte counts **103** and **104** for the two homes (pins the 52-byte suffix). The accept/reject *outcome* is the impure boundary: assert it in Swift against `connectOnce`'s `len < dst.count` check (103 accepted, 104 → `socketPathTooLong`); on the Rust side the cap is enforced at `bind`, not in the pure fn, so the resolver test asserts the byte counts only. **[Rust+Swift]** (Rust: byte-count assertion; Swift: byte-count + accept/reject.)
- **S8b (`sockaddr_un` byte boundary on the per-worktree dev path):** `make run` sets `TARMAC_SOCKET`/`TARMAC_STATE` **verbatim** to a per-worktree path that inserts a `dev/wt-XXXXXXXX/` segment (`wt-` + an 8-hex worktree id) in place of plain `dev/`, so the socket suffix after `home` grows from S8's 52 bytes to a fixed **64 bytes**: `/Library/Application Support/tarmac/dev/wt-XXXXXXXX/tarmacd.sock` (the `dev/wt-XXXXXXXX/` segment is **16** bytes vs **4** for `dev/`, i.e. +12). The byte length is `home.count + 64`, so a `home` of length **39** is **103** bytes (accepted at the same `len < 104` boundary) and length **40** is **104** bytes (rejected as `socketPathTooLong`). Because this path rides the **verbatim override**, the pure resolver returns it unchanged (override-wins, S3/S4) and does not derive it; the boundary test therefore constructs the per-worktree string for a 39- and a 40-byte `home` and asserts the byte counts **103**/**104** (Rust `.len()` / Swift `utf8.count`), pinning the 16-byte `dev/wt-XXXXXXXX/` segment so a change to the `wt-` prefix or the 8-hex width is caught; in Swift, additionally assert accept/reject at `connectOnce`'s `len < dst.count` (103 accepted, 104 → `socketPathTooLong`). The `make run` overflow itself must surface a clear error pointing at `TARMAC_SOCKET` (impure shell, not a raw `socketPathTooLong` — design success criterion 4 + the per-worktree note). **[Rust+Swift]** (Rust: byte-count assertion; Swift: byte-count + accept/reject.)

### Error Scenarios
- **S9 (empty override is treated as unset — unified across both languages):** Given an empty override (`override = ""` in Swift; `over = Some("")` in Rust) and `channel = .dev`, then the socket resolver returns the **dev default** — an empty `TARMAC_SOCKET`/`TARMAC_STATE` falls through to the channel default in *both* languages. Swift preserves today's `!p.isEmpty`; Rust changes its shell to `var_os(...).filter(|v| !v.is_empty())` so a present-but-empty value no longer binds an empty path. The pure resolver's override-wins rule is therefore "set **and** non-empty wins, verbatim" in both. **[Rust+Swift]**
- **S10 (channel label is correct — diagnostic building block):** Given `channel = .release`, `channelLabel` returns exactly `"release"`; given `channel = .dev`, it returns exactly `"dev"`. This is the only unit-testable part of the connect-failure diagnostic; assert both mappings (a swapped or constant label fails). The *surfacing* — that `DaemonClient.connectFailed`'s description and the CLI's "no daemon running" text each contain both the resolved path `P` and `channelLabel(C)` — is impure shell, **integration/GUI-verified, not unit-tested** (see Trade-offs and DoD "Diagnostics"). Rationale: a bare-shell debug `tarmac` that resolves `dev/` but finds no daemon must explain *why* instead of silently no-op'ing. **[Rust+Swift]** (`channelLabel` exists in Swift `ChannelPaths`; the Rust resolvers expose the same label for their messages.)

## For the Implementing Agent

> **Your job:** make every scenario pass with tests that would *fail if the behavior were wrong*. `/verify` hunts vacuous tests by asking, of each behavior, "what is the smallest change that breaks this, and would any test catch it?"

- **Swift:** mirror `DaemonLaunch.swift` (a `public enum` of pure `static func`s with *why* doc-comments) and its table-driven `DaemonLaunchTests.swift`. Put `ChannelPaths` in `TarmacKit` (the app layer is not unit-tested — pure logic must live in the Kit). Wire `DaemonClient.resolveSocketPath()` to read env + home, pick `Channel` behind `#if DEBUG`, and call `ChannelPaths.socketPath(...)`; that wiring is the thin impure shell and is **not** unit-tested, but it MUST route through `ChannelPaths` rather than re-deriving inline.
- **Rust:** put the pure path logic **once** in `tarmac-protocol` — `channel_dir(home, channel)` + the override-aware `resolve_socket_path`/`resolve_state_path` (which delegate their default branch to `channel_dir`) + `channel_label`, plus `enum Channel`; add their table-driven cases to that crate's existing `#[cfg(test)] mod`. `tarmacd` and `tarmac-cli` both already depend on the crate, so the `dev` literal is shared-by-construction — do **not** copy the resolver into either `main.rs`. Each binary keeps only a thin shell: read the env via `var_os(...).filter(|v| !v.is_empty())` so an empty override is treated as unset (S9), read `HOME`, map `cfg!(debug_assertions)` → `Channel` (exactly one audited line — do **not** sprinkle `cfg!(debug_assertions)` through the path code), and call the shared `resolve_*` fns. Add the inline comment to `core/Cargo.toml` forbidding `[profile.*]` `debug-assertions` overrides.
- **Make + diagnostics:** add the per-worktree `TARMAC_SOCKET`/`TARMAC_STATE` to the `run` target; the daemon logs its resolved channel + socket path at startup; the connect-failure messages name path + channel (S10).

Test-quality principles `/verify` scores against:
- **Behavioral over structural** — assert returned path strings, not internals.
- **Every test can fail** — S1 breaks if a stray `dev/` leaks into release; S4 breaks if the channel shadows the override; S5 breaks if the marker is renamed; S8 breaks if the byte math is off by one — assert the exact counts 103 and 104 (both languages), and in Swift also that 103 is *accepted* and 104 → `socketPathTooLong` at the `len < 104` boundary; S8b breaks if the per-worktree `dev/wt-XXXXXXXX/` segment length drifts — assert 103/104 for the 64-byte suffix at a 39/40-byte `home`; S10 breaks if `channelLabel`/`channel_label` returns a constant or swaps the two labels.
- **Deterministic, isolated, readable** — pure functions; no env mutation in unit tests (pass values in); AAA, table-driven, S-numbers in comments.

- [ ] Every scenario S1…S10 (including S8b) maps to at least one test. Swift `ChannelPathsTests` covers S1–S5, S8, S8b, S9 (empty override → default), S10 (the `channelLabel` mappings). The `tarmac-protocol` resolver tests cover S1–S5, S6–S7 (state), S8, S8b, S9 (empty override → default, via the filtered shell), and S10 (the `channel_label` mappings) — one set of tests exercising the helper both binaries call.
- [ ] No covered-but-vacuous scenarios — each fails under the smallest break of its behavior (release-default guard S1; override-beats-channel S4; segment-only-diff S5; state-carries-channel S7; boundary S8 and per-worktree boundary S8b).
- [ ] The override branch is preserved: the existing integration suite — `tests/common/mod.rs` injection, all `m0` tests, `socket_claiming` (`m0_integration.rs:161`), and `scripts/e2e.sh` — stays green **untouched** (verify mechanically: `git diff` against those test files and `scripts/e2e.sh` is empty). - [ ] All three shells route through a pure derivation (no inline re-derivation) — the two Rust shells through `tarmac_protocol`'s `resolve_*` resolvers, the Swift shell through `ChannelPaths`; the `cfg!(debug_assertions)` / `#if DEBUG` → `Channel` mapping is a single audited line per binary; `core/Cargo.toml` carries the no-`debug-assertions`-override comment; the Rust pure logic lives **once** in `tarmac-protocol` (so the `dev` literal is shared-by-construction across `tarmacd` and `tarmac-cli`, with no per-binary copy), and the Swift `ChannelPaths` mirror pins the same literal by test.
- [ ] Release resolves byte-for-byte the legacy flat path (S1, S6) — zero migration.
- [ ] `make run`'s `run` target exports a `TARMAC_SOCKET` and `TARMAC_STATE` derived from `$ROOT` (mechanically checkable: grep the Makefile `run` target for both vars derived from `$ROOT`, layered under `dev/`). **Manual/scripted (not unit-verified):** two `make run`s from different worktrees run simultaneously without sharing a socket or state file, and the per-worktree socket/state stay within the `sockaddr_un` budget (the boundary is pinned by S8b).
- [ ] Diagnostics, unit-checkable: `channelLabel`/`channel_label` return `"release"`/`"dev"` (S10), and `DaemonClient.connectFailed`'s description / the CLI "no daemon" message are wired to include both the resolved path and the channel label (inspect the message-construction sites). **Manual/integration (not unit-verified):** the daemon's startup `info!` line actually prints the resolved channel + socket path, and the surfaced connect-failure strings actually contain both at runtime.
- [ ] Docs updated: `docs/architecture.md:81` and `docs/protocol.md` transport note per-channel namespacing; the CLI `HELP` text reflects it.

## Trade-offs and Limitations

- **Empty-string override is unified to "treated as unset" (a small, deliberate change to Rust's override branch).** Previously Rust's `var_os` treated `TARMAC_SOCKET=""` as a present empty path (binding `""`) while Swift treated empty as unset. This spec aligns them — both fall through to the channel default on empty (S9) via `var_os(...).filter(|v| !v.is_empty())`. This is the one intentional behavior change to the override branch; it is safe because the integration harness always injects a non-empty path, and binding an empty path was never useful.
- **The impure shell is unverified by this gate.** The `cfg`/`#if` → `Channel` mapping, `make run`'s env, the startup log, and `create_dir_all` of the `dev/` (and per-worktree) dir are integration/GUI-verified, not unit-tested. The deliberate seam: pure decision here, I/O there.
- **Channel is *defined* as build configuration.** A release-config build run from a checkout (`swift build -c release && .build/release/TarmacApp`) IS the release channel and intentionally shares release state — a feature (test the shipped path locally), with `TARMAC_SOCKET` as the escape hatch.
- **Per-worktree `make run` tightens the byte budget.** The `dev/wt-XXXXXXXX/` segment needs `$HOME` ≤ ~39 bytes to stay under the cap; overflow surfaces as a clear error pointing at `TARMAC_SOCKET`, and it only affects developer machines.
- **Not isolated by C alone:** two release apps still share the flat socket (existing singleton, unchanged); finer-than-worktree isolation remains the explicit `TARMAC_SOCKET`/`TARMAC_STATE` override.
- **Bare-shell dev CLI is an explicit non-goal.** Reaching a per-worktree dev daemon from a bare-shell debug `tarmac` (no inherited `TARMAC_SOCKET`) is out of scope: a bare-shell debug CLI resolves the *plain* `dev/` path and reaches no worktree daemon. The required S10 diagnostic only makes that miss *legible* (it prints the resolved path + channel) — it is **not** a fix. Drive the CLI from the in-app terminal or export the worktree's `TARMAC_SOCKET`.
