# 2606.0005 Unified "active card" chrome — one bright ring + handles, prime stops drawing a border

**Date:** 2026-06-19
**Status:** draft
**Author:** EP Lin

## Context

A card on the Tarmac board carries several overlapping visual states, set by three independent `CardView` setters that each poke the layer border:

- **Handles** (the four resize corners + the doc ✕) show when `focused || selected` (`CardView.updateHandleVisibility`, `CardView.swift:255-264`).
- **Border color** resolves by priority (`CardView.currentBorderColor`, `CardView.swift:273-281`):
  `dead||detached → muted line` / `selected||fresh → Theme.agent` / `prime → Theme.liftBorder (#5a626a)` / `focused → Theme.focusBorder (teal)` / else `Theme.line`.
- **`prime`** is the keyboard target — the one live terminal holding the window's AppKit first responder (`AppController.setPrime` / `updatePrimacy`, `AppController.swift:1886-1915`). `setPrime` also tints the header (`#3a4046`) and deepens the resting shadow; every non-prime card goes `quiet` (alpha 0.8).

Because border and handles are computed in different places against different conditions, the "bright ring" and the "four corners" — which a user reads as one thing ("the card I'm acting on") — drift apart in two ways:

1. **Two colors for one concept.** Click-focus (`focused`) draws teal; header-grab (`selected`) draws agent-color. The same active card looks different depending on *how* you engaged it.
2. **Prime's border outlives focus.** `prime` sits above `focused` in the priority chain, so (a) a clicked live terminal shows the gray lift border instead of teal, and (b) after you click empty board (`defocus()`, which deliberately keeps prime — `AppController.swift:1318-1324`) or switch boards, the prime terminal keeps a bright border while its handles vanish. The ring stays, the corners go.

## Motivation

The user wants the bright ring and the resize handles to be a single, consistent visual event — appear together, disappear together, same color regardless of card kind or how it was engaged — so "this is the card I'm acting on" reads unambiguously. The decided direction (Option A, user-approved): **`prime` stops competing as a bright border** and is expressed purely through header tint + deeper shadow + dimming the others; the keyboard behavior is untouched. Left alone, the two drift gaps above keep producing "border without corners" and "two different ring colors," which is exactly the confusion this change removes.

## Proposed Solution

### Overview

Collapse the visual states into **one "active card" concept** = `focused || selected`. An active card (that is not dead/detached) draws a single teal ring (`Theme.focusBorder`) **and** the handles — always together, same color for docs and terminals, same color whether click-focused or header-grabbed. `prime` is removed from the border entirely; it keeps only its non-border signals (header tint, deeper shadow, others `quiet`) and its unchanged keyboard role. The border-priority decision and the handle-visibility decision are extracted into one pure TarmacKit rule (`CardChrome`) so the two derive from a single source and cannot desync again, and the coincidence is locked by an exhaustive unit test.

### Key Components

- **`CardChrome` (new, `app/Sources/TarmacKit/CardChrome.swift`)** — pure, AppKit-free rule (same shape as `EscFocusAction` / `FocusedClose`). Holds the card visual-state inputs and derives both the border role and handle visibility. The only unit-tested home for this logic.
- **`CardView` (`app/Sources/TarmacApp/CardView.swift`)** — keeps the AppKit rendering, but `currentBorderColor` and `updateHandleVisibility` both delegate to `CardChrome` (mapping `BorderRole → Theme` color). `setFocused` additionally clears the `fresh` halo (parity with `setSelected`, which already does, `CardView.swift:243`). `setPrime` no longer influences the border (it falls out of `CardChrome`); its header tint + `applyRestingShadow()` remain.
- **`AppController` (`app/Sources/TarmacApp/AppController.swift`)** — unchanged keyboard/focus wiring. `focus()`, `defocus()`, `setPrime()`, `updatePrimacy()` keep their current first-responder and prime semantics. The only observable change flows from `CardView` repainting.

### Interface Contract

New pure rule in TarmacKit (signatures illustrative; the implementing agent picks exact spelling, but the inputs, the four border roles, and the invariant are fixed):

```swift
public enum CardChrome {
    /// Visual-state inputs for a single card.
    public struct State: Equatable {
        public var dead: Bool
        public var detached: Bool
        public var fresh: Bool
        public var prime: Bool      // keyboard target — intentionally NOT a border input
        public var focused: Bool
        public var selected: Bool
    }

    public enum BorderRole: Equatable { case muted, agent, focus, plain }

    /// Priority, highest first:
    ///   dead || detached       -> .muted   (handles may still show — resize exception)
    ///   focused || selected    -> .focus   (the unified active ring; teal)
    ///   fresh                  -> .agent   (agent-opened halo)
    ///   else                   -> .plain
    /// `prime` never appears here.
    public static func borderRole(_ s: State) -> BorderRole

    /// The four resize corners + doc ✕. = focused || selected, NOT suppressed by
    /// dead/detached (a dead card stays resizable via a header grab).
    public static func showsHandles(_ s: State) -> Bool
}
```

`CardView` maps `BorderRole` to color: `.muted → Theme.line.withAlphaComponent(0.6)`, `.agent → Theme.agent`, `.focus → Theme.focusBorder`, `.plain → Theme.line`. `Theme.liftBorder` stays defined but is no longer applied.

**The invariant (locked by test):** for every `State`,
`borderRole(s) == .focus  ⟺  (showsHandles(s) && !s.dead && !s.detached)`.
Equivalently: handles show exactly for an active card; the ring is teal for active cards except dead/detached ones, which keep the muted border but still show handles for resize.

## Acceptance Scenarios

Scenarios S1–S10 are pure-rule outcomes, unit-tested in `TarmacKitTests`. (Booleans not named in a Given are false.)

### Happy Path
- **S1:** Given a card with `focused=true` (nothing else), when `borderRole`/`showsHandles` are evaluated, then `borderRole == .focus` **and** `showsHandles == true` (teal ring + handles together).
- **S2:** Given a card with `selected=true` (not `focused`), then `borderRole == .focus` (NOT `.agent`) and `showsHandles == true` — a header-grab looks identical to a click-focus.
- **S3:** Given a clicked live terminal with `prime=true, focused=true`, then `borderRole == .focus` (teal, NOT the old `.agent`/lift) and `showsHandles == true` — prime no longer overrides the active ring.

### Edge Cases
- **S4:** Given a prime-but-not-active terminal `prime=true, focused=false, selected=false`, then `borderRole == .plain` and `showsHandles == false` — after click-away/board-switch the prime terminal shows no ring and no corners (only its header tint + shadow, applied outside `CardChrome`).
- **S5:** Given `fresh=true` and not active (`focused=false, selected=false`), then `borderRole == .agent` and `showsHandles == false` — the agent-opened halo, no handles.
- **S6:** Given `fresh=true, focused=true`, then `borderRole == .focus` and `showsHandles == true` — an active card outranks the fresh halo (the app also clears `fresh` on focus, S11-manual, so this is the priority guarantee).
- **S7:** Given an idle card with everything false, then `borderRole == .plain` and `showsHandles == false`.

### Error / Exception Scenarios
- **S8:** Given a `dead=true, selected=true` card (resized via header grab), then `borderRole == .muted` and `showsHandles == true` — the one allowed divergence: muted border yet handles present.
- **S9-rule:** Given `detached=true, selected=true`, then `borderRole == .muted` and `showsHandles == true` — `detached` is an exception alongside `dead`.
- **S10 (invariant, exhaustive):** Given **all 64 combinations** of the six booleans, when each is evaluated, then `borderRole(s) == .focus` holds **iff** `showsHandles(s) && !s.dead && !s.detached`. No combination violates it. The expected right-hand side must be **computed** from the inputs (`showsHandles(s) && !s.dead && !s.detached`) — never read back from `borderRole`. Enumerating the full 64 (rather than 32) is deliberate: `prime` is one of the six, and including both its values proves it is inert (toggling `prime` alone never changes `borderRole` or `showsHandles`), which is the central claim of this change. S10 pins only the `.focus` biconditional; the non-`.focus` roles (`.muted`, `.agent`, `.plain`) are pinned individually by S4/S5/S7/S8/S9.

### Manual / compile-verified (App layer — not unit-tested by repo convention)
- **S11-manual (required change, not just regression):** `setFocused(true)` clears the `fresh` halo (mirroring `setSelected`, `CardView.swift:243`) — single-clicking a fresh card dismisses its agent halo. Without this, S6's priority is invisible in the UI.
- **S12-manual (regression):** Keyboard behavior is unchanged — typing follows the prime terminal; clicking a doc does not move first responder; `defocus()` does not touch prime; `⌥tab`/`⌘T`/board-switch keyboard focus all behave as before. Verified by code review (the first-responder / `setPrime` keyboard lines are untouched) and by the existing scroll/ESC/⌘W suites staying green.

Do **not** change keyboard wiring — first-responder / `makeFirstResponder` calls and `AppController`'s prime semantics stay exactly as they are; only how the border is painted changes.

## Trade-offs and Limitations

- **A visible color change users have seen:** a clicked live terminal now shows the teal focus ring instead of the gray `#5a626a` lift border. This is intentional (the whole point is one consistent active color) but it does change a colour shipped on `main`.
- **App-layer wiring stays untested** (`CardView`/`AppController`) per repo convention — the structural guarantee is that both visuals derive from the tested `CardChrome`, but the *delegation* itself is only compile- + manually-verified.
- **`prime` visibility now leans entirely on header tint + shadow + others' `quiet`.** If a board has a single card, "prime" has no `quiet` neighbours to contrast against; the header tint + shadow are the only cue. Acceptable for now; flagged.
