# 2606.0014 Doc card close: teardown across glass and daemon

**Date:** 2026-06-30
**Author:** EP Lin

> Implements GitHub issue #34 ‚Äî `fix(board): close ‚úï on a doc card is a no-op`.
>
> **Scope note:** the reported symptom is one bug (the ‚úï click never fires), but the
> accepted design (`docs/designs/2606.0005_doc_card_close_teardown.md`) treats "closed"
> as a two-layer fact: the card is gone from the glass **and** the OS facts behind it
> (the registry entry, the persisted row, the notify watch) are gone from the daemon.
> This spec covers both layers. All decisions below follow that design, with one
> correction grounded in `docs/protocol.md`: the design's note that "V1‚ÄìV8 must be
> regenerated" is stale (the live contract now has 9 vectors and asserts
> decode-equivalence + round-trip, not byte-exact encoder output). Adding a new message
> *type* leaves every existing variant's encoding untouched, so V1‚ÄìV9 stay unchanged and
> this work adds exactly one new vector (10).

## Context

Tarmac is a terminal-first macOS cockpit. The **board** holds **doc cards** (markdown
surfaced by `tarmac open`, registered in the daemon's `Registry.docs`/`dock` and watched
for mtime changes via `notify`). Closing a doc card has two independent defects today:

1. **Frontend no-op (the reported symptom).** `DocCard`'s close ‚úï is a descendant of
   `CardShell`'s `.card-header`, whose `onPointerDown` calls `setPointerCapture` on the
   header (the drag handle) for any left press. The pointer is captured by the header, so
   in WebKit the subsequent `click` dispatches at the header, not the ‚úï ‚Äî `onClose` never
   fires. The ‚åòW keyboard path already calls the same `removeDoc` and works; only the
   mouse button is dead.
2. **Daemon state leak.** There is no `Msg` variant that tells the daemon to forget a doc.
   `Layout` is the only app‚Üídaemon channel for dock state, and `apply_layout`
   *deliberately re-appends* any registered doc missing from the incoming dock snapshot
   (so a partial layout never drops a doc). So even a ‚åòW-closed doc survives in
   `Registry.docs`/`dock`, reappears on the next `Restore`, and keeps `watch_loop`
   emitting `FileEvent` for its parent dir (`ensure_watched` is additive-only ‚Äî there is
   no `unwatch`). App-side, `docMeta`/`docContents` cache entries are not pruned either.

The dashed SVG provenance edge is **not** a bug: the edge layer derives edge keys from
live card models, so React drops the edge automatically when the card leaves `cards`.

Two architectural facts shape the design:

- **Removal must be an explicit message, not absence in a `Layout` snapshot.**
  `apply_layout` re-appends absent docs by design; removal-by-omission conflicts with that.
- **There is no second-language codec.** The frontend consumes already-decoded data over
  Tauri IPC; only the Rust `tarmac-protocol` crate (path-dep'd by the desktop backend)
  encodes/decodes the wire, so conformance is gated in Rust alone.

## Motivation

A close ‚úï that does nothing is the most basic broken affordance on the board. And even
when the card leaves (via ‚åòW), the doc reappears on restart and silently leaks
`FileEvent`s ‚Äî "closed" doesn't mean closed. Fixing both layers makes the ‚úï land and makes
"closed" mean the card is gone *and* the watch and persisted entry are gone too.

## Proposed Solution

### Overview

Fix both layers; make `DocClose` a first-class, persisted, additive protocol message.

1. **Glass ‚Äî make the click land.** On the `.close` span, handle `onPointerDown` with
   `e.stopPropagation()` (+ `preventDefault`) before firing close. The event never reaches
   the header's `onPointerDown`, so the header never captures the pointer and the close
   fires. (Local to the control that owns the behavior ‚Äî rejected alternative: teach
   `CardShell` to sniff `e.target` for `.close`, which couples the generic shell to a
   specific child.)
2. **Protocol ‚Äî `Msg::DocClose { path }`.** Additive new app‚Üídaemon variant carrying the
   doc's absolute path (the `Registry.docs` key). New conformance vector (vector 10).
3. **Daemon ‚Äî teardown on `DocClose`.** Prune `Registry.docs`/`dock`, persist `state.json`,
   conditionally unwatch the parent dir. Idempotent: unknown path = silent no-op.
4. **Frontend cleanup.** `removeDoc` prunes `docMeta`/`docContents` and sends `DocClose`;
   both the ‚úï and ‚åòW route through `removeDoc`, so both get full teardown.

### Key Components

- **`Msg::DocClose { path: String }` ‚Äî new protocol message**
  (`core/crates/tarmac-protocol/src/lib.rs`, `Msg` enum). App‚Üídaemon, additive (new
  message type; unknown types ignored per the additive-only contract), msgpack tag
  `"t":"doc_close"`, encoded by Rust with `rmp_serde::to_vec_named`. No existing key or
  encoding changes; existing vectors V1‚ÄìV9 stay byte-for-byte unchanged.

- **Daemon `DocClose` handler** (`core/crates/tarmacd/src/conn.rs`). On receipt:
  1. Remove `path` from `Registry.docs` and `Registry.dock`.
  2. Mark the registry dirty ‚Üí persist `state.json`.
  3. Compute whether any *remaining* `docs` entry shares this path's parent dir; if **none**
     does, `unwatch()` the parent dir. This adds the first `unwatch()` caller ‚Äî keep it
     symmetric with `ensure_watched` and respect the lock discipline (the watcher `Mutex`
     must not be held across an `.await`).
  An **unknown / already-removed `path` is a silent no-op**: no error, no persist, no
  unwatch ‚Äî matching "already torn down" semantics and tolerating double-clicks, a stale
  frontend, or a `DocClose` racing `Restore`.

- **`doc_close` Tauri command + `daemon.ts` sender** (`desktop/src-tauri/src/commands.rs`,
  registered in `lib.rs`; `desktop/src/ipc/daemon.ts`), modeled on the existing
  `term_close` command. Forwards `DocClose { path }` to the daemon.

- **`removeDoc` cleanup** (`desktop/src/App.tsx`). In addition to removing the card and
  dock-order entry, prune the closed `path` from `docMeta` and `docContents`, and send the
  `doc_close` IPC. The ‚úï (`onDocClose={removeDoc}`) and the ‚åòW path both already call
  `removeDoc`, so both paths get identical teardown.

### Data Flow

**‚úï click (the fixed path):**

1. Pointer-down on the ‚úï span: `stopPropagation` + `preventDefault` keep the press from
   the header, so no `setPointerCapture`; the click lands on `.close` ‚Üí `onClose(path)` ‚Üí
   `removeDoc(path)`.
2. `removeDoc` removes the card from `cards` (React drops the provenance edge), removes the
   dock-order entry, prunes `docMeta[path]` and `docContents[path]`, and invokes
   `doc_close(path)` over IPC.
3. Daemon receives `DocClose { path }`: prunes `Registry.docs`/`dock`, persists
   `state.json`, and unwatches the parent dir iff no sibling doc shares it.
4. `watch_loop` emits no further `FileEvent` for that path (parent unwatched, or ‚Äî if a
   sibling shares the dir ‚Äî the dir stays watched but `reg.docs.contains_key(path)` is now
   false so the closed doc is never matched).

**‚åòW:** identical from step 2 onward (same `removeDoc`).

### Interface Contract

**New protocol message** (additive; conformance-gated in Rust):

```
// Rust ‚Äî core/crates/tarmac-protocol/src/lib.rs, Msg enum
DocClose { path: String }              // app -> daemon; msgpack tag "t":"doc_close"
// encoded via rmp_serde::to_vec_named (named map, string keys)
```

A new conformance vector (**vector 10** ‚Äî `docs/protocol.md`'s list currently ends at 9)
for `doc_close` MUST be added and kept in sync across `docs/protocol.md` and the Rust
`tarmac-protocol` conformance test. Per `docs/protocol.md`, the contract is
**decode-to-the-structure-shown + own-side encode‚Üídecode round-trip**; byte-exact encoder
output is NOT asserted. Existing vectors V1‚ÄìV9 stay byte-for-byte unchanged.

**Daemon `DocClose` behavior** (the testable rules):

- Known `path` ‚Üí removed from `docs` **and** `dock`; `state.json` after the handler no
  longer contains `path`.
- Parent dir unwatched **iff** no remaining `docs` entry shares the parent; a sibling doc
  in the same dir keeps the dir watched.
- Unknown `path` ‚Üí no state change, no persist, no panic (idempotent no-op).

## Acceptance Scenarios

> Legend: **[R]** = Rust daemon/protocol integration or unit test
> (`core/crates/tarmacd/tests/` or `tarmac-protocol`); **[C]** = Rust conformance-vector
> test + `docs/protocol.md` table; **[K]** = desktop `kit/` pure-logic unit test (Vitest);
> **[M]** = manual / QA of the React+Tauri shell (not unit-tested by design ‚Äî per the repo
> testing memory, only pure `kit/` logic and pure Rust are unit-tested; `/verify` checks
> [M] by build + reasoning, not an automated assertion).

### Happy path ‚Äî the click lands (the reported fix)
- **S1 [M]:** Given a doc card on the board, when its close ‚úï is clicked with the mouse,
  then the card is removed from the board (the ‚úï is no longer a no-op).
- **S2 [M]:** Given a focused doc card, when ‚åòW is pressed, then the card is removed ‚Äî the
  same outcome as S1 (both route through `removeDoc`).
- **S3 [M]:** Given a doc card with a dashed provenance edge to the terminal that opened
  it, when the card is closed (‚úï or ‚åòW), then the provenance edge disappears with the card
  (no orphaned edge remains).

### Protocol ‚Äî `DocClose`
- **S4 [C]:** Given the new `doc_close` conformance vector (vector 10), when it is decoded
  in Rust, then it decodes to `DocClose { path: "/a.md" }` and survives an encode‚Üídecode
  round-trip; and all existing vectors V1‚ÄìV9 remain byte-for-byte unchanged.
- **S5 [R]:** Given a `Msg::DocClose { path }` value, when it is round-tripped through
  `to_vec_named` ‚Üí decode, then it equals the original (the additive variant is wired into
  the `Msg` enum and survives the codec).

### Daemon teardown (observable OS facts)
- **S6 [R]:** Given a doc registered in the daemon (via `Open`/`doc_opened`, so it appears
  in `Registry.docs` and `dock`), when `DocClose { path }` is received, then `path` is
  absent from both `Registry.docs` and `Registry.dock` afterward.
- **S7 [R]:** Given the doc from S6, when `DocClose` is received, then `state.json` written
  by the handler no longer contains `path` ‚Äî so after a daemon restart + `Restore`, the doc
  does **not** reappear in the restored docs.
- **S8 [R]:** Given a doc whose parent dir is watched and that is the **only** open doc in
  that dir, when `DocClose { path }` is received, then the parent dir is unwatched and no
  further `FileEvent` is emitted for `path` when that file is subsequently touched (the
  live-before-close baseline ‚Äî a `FileEvent` observed *before* the close ‚Äî makes the
  absence a genuine transition, not a vacuous match).
- **S9 [R]:** Given **two** docs in the **same** parent dir (siblings), when `DocClose` is
  received for one, then the *other* doc's parent dir stays watched: touching the surviving
  sibling still emits a `FileEvent`, while touching the closed doc's path does not. This is
  the correctness pivot ‚Äî unwatch is conditional on no sibling sharing the dir.

### Idempotency / regression
- **S10 [R]:** Given a `path` not present in `Registry.docs`, when `DocClose { path }` is
  received, then it is a benign no-op with an observable post-condition: an unrelated open
  doc still appears in `Registry.docs`/`dock` and its dir stays watched (the test fails if
  an unknown path panics the daemon, persists a spurious state, or tears down an unrelated
  doc/watch).
- **S11 [R]:** Given a doc that was just closed via `DocClose`, when a **second**
  `DocClose` for the same `path` is received (double-click / stale frontend), then it is a
  no-op ‚Äî no error, no second persist, no effect on other docs (idempotent under repeat).
- **S12 [M]:** Given a closed doc card, when the card is removed via `removeDoc`, then the
  app-side `docMeta` and `docContents` entries for that `path` are pruned (no stale cache
  entry survives the close). *(Verified by code review + manual check: the prune lives in
  the `App.tsx` shell, which is not unit-tested; if extracted to a pure `kit/` helper it
  becomes [K].)*

## For the Implementing Agent

> **Your job:** make every acceptance scenario above pass with tests that would *fail if the
> behavior were wrong*. `/verify` will hunt for vacuous tests by asking, of each behavior,
> "what is the smallest change that breaks this, and would any test catch it?"

- **[R]/[C] scenarios are the automated contract.** Put the daemon teardown tests in a Rust
  integration suite under `core/crates/tarmacd/tests/` (e.g. `doc_close_integration.rs`),
  mirroring the existing `m*_integration.rs` style and the `FileEvent`-observation idiom.
  S8/S9 must observe a real `FileEvent` *before* the close (baseline) so the post-close
  absence is a genuine transition. S10/S11 must assert the no-op via an observable
  post-condition (an unrelated doc/watch survives), not merely "no panic". The vector-10
  round-trip (S4/S5) goes in the `tarmac-protocol` conformance test plus the
  `docs/protocol.md` vector table.
- **[M] scenarios are React+Tauri shell wiring** (`DocCard`/`CardShell` pointer handling,
  `App.tsx` `removeDoc`, `daemon.ts`/`commands.rs` IPC). The shell is not unit-tested by
  design ‚Äî verify by `make` building cleanly, by reasoning against the data flow, and by a
  manual run-through (‚úï click removes the card; edge gone; cache pruned). Keep the wiring
  thin so the daemon's testable behavior stays in Rust.

Write tests to the project's conventions and these principles:
- **Behavioral over structural** ‚Äî assert registry contents, `state.json` after restart,
  presence/absence of `FileEvent`s, and the decoded `Msg`; not internal field layout.
- **Every test can fail** ‚Äî no copy-pasted expected values, no tautologies (AP-2, AP-4).
  S8/S9 must fail if unwatch is unconditional (sibling silenced) or never happens (closed
  doc still emits). S10/S11 must fail if an unknown path is not a true no-op.
- **Deterministic, isolated, readable** ‚Äî AAA structure, inline setup, no cross-test state.

Conventions: Conventional Commits; do **not** run `cargo fmt`; Rust edition 2024; serde
stack is `=`-pinned; the additive-only + conformance-gated protocol rule applies; `make`
must pass before the work is considered done.

## Definition of Done

Done is when `/verify` passes against this spec:

- [ ] `make test` is green; `make` builds both targets cleanly.
- [ ] Every scenario (S1‚Ä¶S12) maps to at least one test or, for **[M]** scenarios, to
      wiring demonstrably present and consistent with the data flow (build + reasoning).
- [ ] No covered-but-vacuous scenarios ‚Äî each automated scenario fails under the smallest
      break of its behavior (thought-mutation). In particular: S7 fails if the entry is not
      persisted-out (doc reappears on restart); S8 fails if the dir is never unwatched; S9
      fails if unwatch is unconditional and silences the sibling; S10/S11 fail if an unknown
      or repeated path is not a true no-op.
- [ ] The additive protocol change keeps V1‚ÄìV9 byte-for-byte unchanged and adds exactly one
      new vector (10) across `docs/protocol.md` and the Rust conformance test.
- [ ] No implementation-quality blockers (stubs, dead code, stale docstrings). `unwatch()`
      respects the watcher-`Mutex`-not-held-across-`.await` lock discipline.

## Trade-offs and Limitations

- **Unwatch is conditional, not eager.** Multiple docs can share a parent dir; unwatching
  unconditionally would silence a still-open sibling. The "iff no sibling shares the parent"
  check is the correctness pivot (S9).
- **Path is the stable registry key.** `DocClose { path }` assumes the absolute path
  uniquely identifies the doc in `Registry.docs` ‚Äî consistent with how `open` registers and
  how watching is keyed by parent dir.
- **Additive bump cost.** A new `Msg` variant forces a conformance-vector regen (mandatory,
  gated). One-time and cheap; accepted because expressing close as data is the only
  contract-clean way to persist the teardown.
- **No CLI verb added.** The `socket_path()` duplication across `tarmacd` and `tarmac-cli`
  is untouched; `DocClose` is an app‚Üídaemon message, not a `tarmac` subcommand.
