# 2606.0015 Terminal-card WebGL renderer — GPU scrollback with canvas fallback

**Date:** 2026-06-30
**Status:** implemented
**Author:** EP Lin
**Issue:** #32 — perf(terminal): fix laggy scrollback in terminal cards

## Context

`TerminalCard.tsx` constructs `new Terminal({… scrollback: 5000 …})` (lines 109–118) and never loads a renderer addon. xterm.js therefore falls back to its **default CPU canvas 2D renderer**, which repaints the full viewport on every scroll step. The cost is amplified 2–4× by `rasterScale` oversampling at zoom > 1 (`deriveRasterScale` returns 1.5 at zoom 1.1, 2.0 at zoom 1.6, up to the 3.0 cap; the xterm host is laid out at `rasterScale × card size`). Result: wheel/trackpad scrollback lags the gesture — the symptom filed in #32.

`@xterm/addon-webgl` is **not** in `desktop/package.json` (only `addon-fit`, `addon-unicode11`, `xterm` at the `*-beta.287` line).

## Motivation

xterm.js's WebGL renderer moves glyph compositing to the GPU and is the project's documented remedy for exactly this scroll-repaint cost. Adding it — with a clean fallback when WebGL is absent or its context is lost — restores native-feeling scrollback without touching the load-bearing terminal input/fit/PTY paths.

## Proposed Solution

### Overview

Additively load the WebGL addon after `term.open(host)`. If WebGL is available, glyphs composite on the GPU; if not (headless, blocklisted GPU, driver crash, or a runtime `webglcontextlost`), the terminal **gracefully and permanently** falls back to the default canvas renderer for the life of that terminal. All decision logic — "attempt WebGL?" and "what is the renderer state after event X?" — lives in a pure `kit/` module so it is unit-testable; the addon load/dispose plumbing stays in the (untested-by-convention) React shell.

### Key Components

1. **`desktop/package.json`** — add `@xterm/addon-webgl` at the same beta line as the sibling addons: `"@xterm/addon-webgl": "^0.x.0-beta.287"` (use the exact published `0.x.0-beta.287` matching the installed `@xterm/xterm@6.1.0-beta.287`).
2. **`desktop/src/kit/termRenderer.ts`** (new, pure) — renderer state + decision/transition functions. No xterm imports, no DOM.
3. **`desktop/src/kit/termRenderer.test.ts`** (new, Vitest) — colocated, mirrors the `rasterScale.test.ts` pattern.
4. **`desktop/src/cards/TerminalCard.tsx`** — after `term.open(host)` (line 131) and after the addon loads at 119–123, run the WebGL attempt driven by `termRenderer`; register `onContextLoss`; dispose the addon in the teardown closure (line 282 `return () => {…}`).

### Interface Contract (`kit/termRenderer.ts`)

A minimal two-field state machine. `kind` is the active renderer; `webglDisabled` latches once WebGL is ruled out so it is **never re-attempted** on that terminal.

```ts
export type RendererKind = "webgl" | "canvas";

export interface RendererState {
  kind: RendererKind;        // currently active renderer
  webglDisabled: boolean;    // latched true once WebGL is ruled out (unsupported / lost)
}

/** Default before any WebGL attempt: canvas, WebGL still eligible. */
export const INITIAL_RENDERER_STATE: RendererState; // { kind: "canvas", webglDisabled: false }

/** Pure decision: should the shell try to load the WebGL addon now?
 *  true iff capability probe passed AND not already on webgl AND not latched-off. */
export function shouldAttemptWebgl(state: RendererState, supported: boolean): boolean;

/** Successful addon load. */
export function onWebglLoaded(state: RendererState): RendererState; // { kind:"webgl", webglDisabled:false }

/** Runtime context loss (onContextLoss): drop to canvas, latch off — idempotent,
 *  never returns kind:"webgl" and never clears webglDisabled. */
export function onContextLoss(state: RendererState): RendererState; // { kind:"canvas", webglDisabled:true }

/** Probe failed OR addon construction threw: stay on canvas, latch off (no throw). */
export function onWebglUnavailable(state: RendererState): RendererState; // { kind:"canvas", webglDisabled:true }
```

**Shell wiring (pseudocode — not part of the tested contract):**

```
let rstate = INITIAL_RENDERER_STATE;
if (shouldAttemptWebgl(rstate, probeWebglSupported())) {
  try {
    const gl = new WebglAddon();
    gl.onContextLoss(() => { gl.dispose(); rstate = onContextLoss(rstate); });
    term.loadAddon(gl);
    rstate = onWebglLoaded(rstate);
    webglRef.current = gl;            // for teardown
  } catch { rstate = onWebglUnavailable(rstate); }  // swallow → canvas
}
// teardown (line ~282): webglRef.current?.dispose(); before term.dispose();
```

The `supported` probe and the `try/catch` live in the shell; `termRenderer` only decides and transitions.

## Acceptance Scenarios

The WebGL addon load is a GPU/DOM side effect that **cannot be exercised in the headless Vitest shell**; per repo convention (app layer not unit-tested) the tested contract is the pure `termRenderer` decision/transition function. S1–S5 are kit unit tests; S6 is a manual-QA performance gate.

- **S1 — attempt when supported & fresh.** Given `INITIAL_RENDERER_STATE` (which must deep-equal `{ kind:"canvas", webglDisabled:false }`), When `shouldAttemptWebgl(state, true)`, Then it returns `true`.
- **S2 — load transitions to webgl, then never re-attempts.** Given `INITIAL_RENDERER_STATE`, When `onWebglLoaded(state)`, Then the result deep-equals `{ kind:"webgl", webglDisabled:false }`; and When `shouldAttemptWebgl(thatResult, true)`, Then it returns `false` (already `kind:"webgl"`).
- **S3 — context loss transitions webgl → canvas, latched.** Given `onWebglLoaded(INITIAL_RENDERER_STATE)`, When `onContextLoss(state)`, Then result is `{ kind:"canvas", webglDisabled:true }`; and `onContextLoss` applied again is idempotent (same value).
- **S4 — post-loss never re-attempts.** Given the S3 result, When `shouldAttemptWebgl(state, true)` (even with `supported=true`), Then it returns `false` — the lost terminal never re-attaches WebGL.
- **S5 — unsupported / unavailable → canvas, no throw.** Given `INITIAL_RENDERER_STATE`, When `shouldAttemptWebgl(state, false)`, Then `false`; and `onWebglUnavailable(state)` returns `{ kind:"canvas", webglDisabled:true }` without throwing, after which `shouldAttemptWebgl` stays `false`.
- **S6 (manual-QA, perf) — scrollback tracks the gesture.** Given a terminal card with > 1 screen of scrollback at zoom 1.5–2×, When scrolling rapidly via trackpad/wheel, Then the viewport tracks input without the stutter from #32, and the card visibly uses WebGL (no canvas-2D repaint lag). On a machine where WebGL is unavailable, the card still renders correctly via canvas with no console error.

## Alternatives Considered

- **Tune the CPU canvas renderer instead of GPU — rejected.** No addon config meaningfully removes the full-viewport repaint cost; WebGL is the documented fix. Tuning would be effort for a fraction of the win.
- **Reduce `scrollback: 5000` — deferred (out of scope).** Smaller buffers lighten repaint but change product behavior (less history) and only mask the renderer cost. Tracked separately; not done here.
- **Harden board pan/zoom vs. terminal scroll event routing — deferred (out of scope).** Issue #32 also notes possible wheel-event contention between board pan/zoom and the terminal's own scroll. That is a distinct change to event capture/propagation with its own regression surface; this spec is renderer-only. Revisit if lag persists after WebGL lands.
- **`@xterm/addon-canvas` (explicit canvas addon) — rejected as primary.** It is the same CPU path we are leaving; the default renderer already covers the fallback, so no extra dependency is warranted.

## Trade-offs and Limitations

- **App-shell wiring stays untested** per repo convention; the tested contract is the pure `termRenderer` state machine. The actual addon load, GPU compositing, and `onContextLoss` firing are validated by manual-QA only (they cannot run headless).
- **One-way fallback:** after a context loss a terminal stays on canvas for its lifetime (no re-attach). Acceptable — context loss is rare (GPU reset / sleep) and re-attaching to a lost context is the failure mode we are explicitly avoiding.
- **Oversample interaction:** WebGL reduces but does not eliminate per-frame cost at high `rasterScale`; if lag persists at the 3× cap, the deferred event-routing/scrollback items above are the next levers.
- **WebGL + the BCR selection-coord override / IME paths** are unchanged here; manual-QA should confirm selection and CJK input still behave under the GPU renderer.
