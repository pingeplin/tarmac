# 2606.0006 Fresh card drops its border — agent-opened docs signal via halo + header only

**Date:** 2026-06-19
**Status:** draft
**Author:** EP Lin

## Context

Spec [2606.0005](2606.0005_unified_active_card_chrome.md) unified the card "active" visual onto one teal ring (`CardChrome` `.focus`) and moved `prime` off the border (header tint + shadow instead). One state still draws a bright edge without being active: **`fresh`**.

A doc opened via `tarmac open` is marked `fresh` (agent just dropped it, unread). The path `handleDocOpened` → `landFreshCard` → `landDocCard(fresh: true)` → `CardView.setFresh(true)` gives it a **solid cyan border** (`CardChrome.borderRole` returns `.agent` → `Theme.agent` `#1abc9c`), plus an `agentDim` halo and a `✚ now` header meta. Crucially, opening never sets `focusedCardID`, so the card has **no focus effect** (scroll still pans the board over it; it is not the copy/ESC/⌘W target).

The problem: `Theme.agent` (`#1abc9c`) and the focus ring `Theme.focusBorder` (`#1abc9c @0.5`) are the **same hue**, and the fresh edge is actually *brighter* (solid + halo). So a non-active fresh doc reads as "the active card" — exactly the "bright border ≠ actual focus state" inconsistency 2606.0005 set out to remove, now showing up on the `fresh` axis.

## Motivation

After 2606.0005, "teal cyan edge = the card you're acting on" is the board's consistent language. `fresh` borrowing that hue for "agent-opened, unread" re-introduces the ambiguity on a card that has zero focus effect. The decided fix (Option A, user-approved) applies the same move we made for `prime`: **`fresh` stops competing as a border** and is signalled by the channels it already owns — the `agentDim` halo glow and the `✚ now` header meta. The teal edge then belongs to active cards alone, and `CardChrome.borderRole` collapses to the dead/active/plain axis with both `prime` and `fresh` proven border-inert.

## Proposed Solution

### Overview

Remove `fresh` from the border rule. `CardChrome.borderRole` no longer returns `.agent`; the `.agent` case leaves the `BorderRole` enum. A fresh-but-not-active doc draws the plain line border and keeps its halo + `✚ now`. Active (focus/selection) still wins via the unchanged `.focus` branch. `CardChrome.State` keeps the `fresh` and `prime` fields as **inert** inputs so the exhaustive test can lock "neither draws a border."

### Key Components

- **`CardChrome` (`app/Sources/TarmacKit/CardChrome.swift`)** — `BorderRole` becomes `{ .muted, .focus, .plain }` (drop `.agent`). `borderRole`: `dead||detached → .muted` / `focused||selected → .focus` / else `.plain`. `State` and `showsHandles` unchanged (`fresh`/`prime` retained, both inert).
- **`CardView` (`app/Sources/TarmacApp/CardView.swift`)** — `currentBorderColor`'s switch drops the `.agent` case (→ muted/focus/plain). `setFresh` keeps the `agentDim` halo ring + `header.setFreshMeta` but drops its `layer?.borderColor = currentBorderColor` repaint (fresh no longer drives the border — same move as `setPrime` in 2606.0005). Update the fresh-state doc comment.

### Interface Contract

```swift
public enum CardChrome {
    // State unchanged from 2606.0005: { dead, detached, fresh, prime, focused, selected }.
    // fresh and prime are both BORDER-INERT inputs (retained for the inertness invariant).

    public enum BorderRole: Equatable { case muted, focus, plain }   // .agent removed

    /// dead || detached -> .muted ; focused || selected -> .focus ; else -> .plain
    /// Neither `prime` nor `fresh` appears.
    public static func borderRole(_ s: State) -> BorderRole

    /// Unchanged: focused || selected (not suppressed by dead/detached).
    public static func showsHandles(_ s: State) -> Bool
}
```

`CardView` maps `.muted → Theme.line.withAlphaComponent(0.6)`, `.focus → Theme.focusBorder`, `.plain → Theme.line`. `Theme.agent` / `agentDim` stay defined and in use (provenance edges, owner chips, the fresh halo) — only the `BorderRole.agent` case is removed.

**Invariant (carried from 2606.0005, still exactly true):** `borderRole(s) == .focus ⟺ (showsHandles(s) && !s.dead && !s.detached)`. The formula never referenced `fresh`, so dropping the `.agent` branch does not change it.

## Acceptance Scenarios

S1–S5 are pure-rule outcomes, tested in the existing `CardChromeTests`. (Booleans not named are false.)

### Happy Path
- **S1:** Given a fresh-but-not-active card (`fresh=true`, `focused=false`, `selected=false`, not dead/detached), when evaluated, then `borderRole == .plain` (NOT `.agent`) and `showsHandles == false`. *(This is the behavior change.)*
- **S2:** Given an active card (`focused=true`) with no fresh, then `borderRole == .focus` and `showsHandles == true` *(2606.0005 regression guard, must still hold)*.

### Edge Cases
- **S3:** Given `fresh=true, focused=true`, then `borderRole == .focus` and `showsHandles == true` — an active card outranks fresh; fresh is inert on the border.
- **S4:** Given `dead=true, selected=true`, then `borderRole == .muted` and `showsHandles == true` — the dead/detached resize exception is unchanged.

### Invariant
- **S5 (exhaustive over all 64 states):** For every combination of the six booleans, compute the expected role *from the inputs* — `(dead||detached) ? .muted : (focused||selected) ? .focus : .plain` — and assert `borderRole(s)` equals it. The expected side is computed independently, never read back from `borderRole`, so the test can't pass by mirroring the implementation. This one positive equality locks three things at once:
  - **(a)** `borderRole(s) == .focus` **iff** `(focused||selected) && !dead && !detached` — the 2606.0005 biconditional, unchanged.
  - **(b)** every state lands in `.muted` / `.focus` / `.plain`, so no state takes the old fresh-driven role. It must be this positive equality, not an `!= .agent` check — after the change `.agent` no longer exists on `BorderRole`, so `!= .agent` would not compile.
  - **(c)** toggling **only** `prime`, or **only** `fresh`, changes neither `borderRole` nor `showsHandles` — both are border-inert.

### Manual / compile-verified (App layer — not unit-tested by repo convention)
- **S6-manual:** `tarmac open` a doc → the landed card shows the `agentDim` halo + `✚ now` header but **no solid cyan border edge** (plain line). Single-clicking it shows the teal active ring and clears the halo (the `setFocused`-clears-`fresh` path from 2606.0005). Keyboard/scroll/focus semantics unchanged.

## For the Implementing Agent

> **Your job:** make every scenario pass with tests that fail if the behavior were wrong. `/verify` will hunt for vacuous tests. S5's expected side must be computed from the inputs; the role-coverage and `fresh`-inert sub-assertions must be real, load-bearing checks. The regression they guard: if a future change re-added the `.agent` case and routed `fresh → .agent` in `borderRole`, S1 would fail (fresh-not-active expects `.plain`) and S5(b)/(c) would fail (the computed role would be `.plain` while `borderRole` returned `.agent`, and flipping `fresh` would flip the role).

## Trade-offs and Limitations

- **Fresh becomes quieter.** Losing the solid cyan edge, a fresh doc is signalled only by the `agentDim` halo (`#1abc9c @0.16`) + `✚ now`. Halo brightness is intentionally left as-is this spec; making it punchier (if fresh under-signals in practice) is a separate one-line tuning knob, not part of this change.
- **App-layer wiring stays untested** (`CardView.setFresh` / `currentBorderColor`) per repo convention — the structural guarantee is that the border derives from the tested `CardChrome`; the delegation is compile- + manually-verified.
