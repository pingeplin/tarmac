# 2606.0014 Doc Card Provenance Survives Move

**Date:** 2026-06-30
**Status:** implemented
**Author:** EP Lin

## Context

A `tarmac open <path>` call places a doc card next to its originating terminal, tied by a dashed provenance edge. Dragging that card currently dims it (`opacity: 0.5`) and drops the edge. Issue #33.

Root cause: `onCardMoveEnd` (`desktop/src/App.tsx` ~1533) sets `attached: false` on any doc-card move. `attached` conflates two unrelated ideas — **gravity** (does the doc co-move when its terminal moves?) and **provenance** (the edge + un-dimmed chrome). Two consumers read `attached` and regress:

- `desktop/src/board/Board.tsx:209` — `detached={c.ownerTermId != null && !c.attached}` → adds `.detached` → `opacity: 0.5` (`theme.css:156`).
- `desktop/src/board/EdgeLayer.tsx` gates on `!c.attached` in **two** places that must stay in lockstep: `edgeKeys()` (line ~23, decides which edge `<path>` elements render) and `updateEdges()` (line ~42, decides which paths get their geometry `d` computed). Both emit/draw an edge only when `attached === true`; fixing only one leaves the edge emitted-but-undrawn (or drawn-but-unkeyed).

The edge already reads **live** world frames, so it would follow the card if it were not gated off.

## Motivation

The card+edge are the only on-screen record of which terminal produced a doc. Losing them on a routine drag destroys provenance for an OS-fact-backed mark — the core promise of the board. Users reorganize cards constantly.

## Proposed Solution

### Overview

Decouple provenance from gravity. Provenance (edge visibility + dim) depends **only** on whether the doc has an owner terminal present on the board — never on `attached`. A manual drag may still set `attached = false` to stop gravity co-move; it must no longer affect the edge or brightness. Owner-linked doc cards never dim.

### Key Components

- **`desktop/src/kit/provenance.ts`** — add two pure, unit-tested predicates (the app/React layer is not unit-testable per repo convention, so the decision logic lives here):
  - `docDimmed(ownerTermId): boolean` — an owner-linked doc (`ownerTermId != null`) is never dimmed; an owner-less doc was never dimmed either. Effectively `false` for the doc-card path.
  - `provenanceEdgeShown(ownerTermId, ownerCardPresent): boolean` — `ownerTermId != null && ownerCardPresent`. No `attached` term.
- **`desktop/src/kit/ownerChip.ts`** — drop the `attached` parameter from `ownerChipName`. The "← terminalname" chip is provenance, so it shows whenever the owner term exists with a non-empty label, regardless of `attached`. Update its breaking signature, `Board.tsx:206` caller, and `ownerChip.test.ts` (the "detached → null" case flips to "loose doc still shows label").
- **`Board.tsx`** — derive `detached` from `docDimmed(c.ownerTermId)` (line ~209); call `ownerChipName(c.ownerTermId, termLabel)` (line ~206, no `attached` arg).
- **`EdgeLayer.tsx`** — gate each edge on `provenanceEdgeShown(...)` instead of `!c.attached` in **both** `edgeKeys()` (which `<path>` elements render) and `updateEdges()` (which paths get their `d` geometry written). Both gates must use the same predicate or the edge will be keyed-but-undrawn / drawn-but-unkeyed.

`onCardMoveEnd` keeps setting `attached: false` (gravity only) — unchanged. Gravity co-move on terminal drag is untouched.

### Data Flow

1. Doc opened → `ownerTermId` set, `attached: true` → bright, edge shown.
2. User drags doc → `onCardMoveEnd` sets `attached: false`.
3. `Board.tsx` calls `docDimmed(ownerTermId)` → `false` → no `.detached` → stays bright.
4. `EdgeLayer.edgeKeys()` calls `provenanceEdgeShown(ownerTermId, ownerPresent)` → `true` → edge re-emitted; `updateEdges()` redraws to the card's live frame.

### Interface Contract

```ts
// desktop/src/kit/provenance.ts
export function docDimmed(ownerTermId: string | null | undefined): boolean;
export function provenanceEdgeShown(
  ownerTermId: string | null | undefined,
  ownerCardPresent: boolean,
): boolean;

// desktop/src/kit/ownerChip.ts — breaking: `attached` removed
export function ownerChipName(
  ownerTermId: string | undefined,
  labelOf: (termId: string) => string | undefined,
): string | null;
```

## Acceptance Scenarios

### Happy Path
- **S1:** Given an owner-linked doc card (`ownerTermId` set) that has been dragged (`attached === false`), when `docDimmed(ownerTermId)` is evaluated, then it returns `false` (card stays bright).
- **S2:** Given an owner-linked doc whose owner card is present, when `provenanceEdgeShown(ownerTermId, true)` is evaluated, then it returns `true` (edge shown) regardless of `attached`.
- **S3 (no-regression, verified by inspection):** Given an attached owner-linked doc, when its owner terminal moves, then the doc still co-moves. Gravity lives in `App.tsx` `onCardMove` and is App-layer (not unit-tested per repo convention); this scenario is satisfied by leaving the gravity co-move path and `onCardMoveEnd`'s `attached: false` write untouched, not by a `provenance.test.ts` case.

### Edge Cases
- **S4:** Given a doc with no owner (`ownerTermId == null`), when `docDimmed(null)` and `provenanceEdgeShown(null, false)` are evaluated, then both return `false` (never dimmed, no edge).
- **S5:** Given an owner-linked doc whose owner card is absent (terminal closed), when `provenanceEdgeShown(ownerTermId, false)` is evaluated, then it returns `false` (no dangling edge).
- **S6:** Given a dragged (loose) doc whose owner term exists with a non-empty label, when `ownerChipName(ownerTermId, labelOf)` is evaluated, then it returns that label (chip survives the move); a missing owner or empty label still yields `null`.

## Trade-offs and Limitations

- The `.detached` CSS rule (`theme.css:156`) becomes effectively unreachable for doc cards. Leaving it is harmless; removing it is out of scope.
- Per locked decision, there is no "owner exited" dim signal — owner-linked docs are simply never dimmed.
