# 2606.0004 Cmd+W closes the focused card; ESC defocuses instead of shelving

**Date:** 2026-06-19
**Status:** approved
**Author:** EP Lin

> Implements GitHub issue #15 — `feat(board): Cmd+W to close cards; ESC clears card focus instead of closing`.
>
> **Scope note (supersedes the issue's literal wording):** the issue title says "close **all** cards," but the resolved intent is **`Cmd+W` closes the single *focused* card** (the keyboard twin of the header ✕), not a whole-board sweep. This matches the issue's own motivation — *"leaving removal to the ✕ / `Cmd+W`"* — and the ESC change (ESC defocuses; `Cmd+W` removes the focused card). All decisions below were resolved with the issue author before this spec.

## Context

Tarmac is a terminal-first macOS cockpit. The **board** is an infinite canvas holding two kinds of cards: **terminal cards** (live PTYs owned by the Rust daemon `tarmacd`) and **doc cards** (markdown surfaced by `tarmac open`). All keyboard input is intercepted by a single global `NSEvent` keyDown monitor in `app/Sources/TarmacApp/AppController.swift` (`start()`, ~lines 422–547), which runs *before* the AppKit responder chain and returns `nil` to swallow an event or the event to pass it through.

Two card-management gaps exist today:

1. **No keyboard close for the card you're looking at.** Doc cards close to the shelf via the header ✕ or ESC; terminal cards have **no manual close affordance at all** (they only vanish when their shell exits, via `handleExit` → `TermExit.decide`).
2. **ESC is overloaded.** When a doc card is focused, ESC acts as the "keyboard twin of the ✕" and sends the card to the shelf (the last branch of the ESC cascade, ~lines 539–543, calls `moveToShelf(path)`). This conflates "I'm done looking at this" (defocus) with "remove this card" (close).

The fix splits the two verbs cleanly: **ESC = defocus** ("done looking"), **`Cmd+W` = close the focused card** ("remove it"). The card the verbs act on is `focusedCardID` — *the card the user last clicked into*.

Two architectural facts established during reconnaissance shape the design:

- **There is no per-terminal kill in the wire protocol today.** `board_delete` is the only path that kills PTYs, and it kills *every* PTY on a board at once. `Cmd+W` closing a single live terminal therefore requires a new additive protocol message.
- **`focus` and `prime` are orthogonal.** `focusedCardID` gates *scroll routing* (which card the scroll wheel drives); the *prime* terminal (window first responder) gates *keyboard typing*. `defocus()` clears focus but never touches the prime terminal — so typing (and ESC) still follow the prime terminal. This is why ESC currently passes through to a focused, non-docked terminal and reaches the running program (vim, `less`, an agent's ESC-to-interrupt) — behavior this spec preserves.

## Motivation

`Cmd+W` is universal macOS muscle memory for "close the thing in front of me." Pointing it at the focused card gives terminal cards their first keyboard close and gives doc cards a non-destructive-surprise close, while ESC stops silently shelving a doc you only wanted to stop scrolling. Without this, terminal cards can't be closed by hand at all, and ESC keeps conflating "done looking" with "remove."

## Proposed Solution

### Overview

Two independent keyboard behaviors, wired into the existing global keyDown monitor:

1. **`Cmd+W` → close the focused card.**
   - **Focused doc** → `moveToShelf(path)` (recoverable on the shelf). This is the removal that moves *off* ESC.
   - **Focused terminal** → terminate it: send the new `TermClose` (daemon SIGHUPs the process group) and tear down app-side. Reuse the existing clean-exit partition (`TermExit`): if it was the board's **last** live terminal, spawn one fresh replacement shell (the existing ≥1-live-terminal invariant); otherwise remove it and show the existing single-terminal undo toast.
   - **Nothing focused** → no-op.
   - `Cmd+W` is **always swallowed** by the monitor (whether it closed a card or no-oped) so it never reaches the macOS window-close menu — "close the focused card, never the window."
2. **`ESC` on a focused *doc* card → `defocus()` instead of `moveToShelf()`.** A focused *terminal* is intentionally left unmatched by this branch so ESC keeps passing through to the program. All higher-priority ESC behaviors are preserved in their current order.

The two decision points are extracted into small, pure, unit-tested TarmacKit helpers (mirroring the `TermExit.decide` pattern); the AppKit wiring and the daemon kill are verified by build + Rust integration test + cross-language conformance + manual/e2e.

### Key Components

- **`TermClose { term_id: String }` — new protocol message** (`core/crates/tarmac-protocol/src/lib.rs`). App→daemon, additive (new message type; unknown types ignored per the additive-only contract), msgpack tag `"t": "term_close"`, encoded by Rust with `rmp_serde::to_vec_named`. Mirrored in Swift `app/Sources/TarmacKit/Messages.swift` (encode + decode) and sent via a new `DaemonClient.termClose(termID:)` (`app/Sources/TarmacKit/DaemonClient.swift`, modeled on `boardDelete` ~line 189).

- **Daemon `TermClose` handler** (`core/crates/tarmacd/src/conn.rs`, modeled on the `BoardDelete` arm ~349–405). Looks up the term's `TermHandle`, **clones the handle and drops the lock before calling `kill()`** (the established lock discipline), then `handle.kill()` (`core/crates/tarmacd/src/term.rs` ~54–96 — `libc::kill(-pid, SIGHUP)` to the process group; no-op when pid unknown; ESRCH ignored). The existing pump/wait thread then emits the usual `Exit` and performs normal map cleanup; the `TermClose` arm itself touches no maps. Unknown `term_id` is a benign no-op.

- **`FocusedClose` — pure routing helper** (new, `app/Sources/TarmacKit/FocusedClose.swift`). Given the focused card's kind (doc / terminal / none) and the count of *other* live terminals on the board, returns what `Cmd+W` should do. The terminal "replace vs. remove+undo" branch is derived from the existing tested rule (`TermExit.decide(code: 0, otherLiveTerminals:)`), so the "a deliberate close is treated like a clean exit; replace only when it was the last terminal" rule lives in one place. No AppKit, no side effects — unit-tested in `app/Tests/TarmacKitTests/FocusedCloseTests.swift`.

- **`EscFocusAction` — pure ESC card-focus rule** (new, `app/Sources/TarmacKit/EscFocusAction.swift`). Encodes "focused doc → defocus; focused terminal or nothing focused → not handled (pass through)." The helper input is a single `isDoc: Bool`, so at the pure-helper level "focused terminal" and "nothing focused" are the **same** call (`forFocusedDoc(false) → nil`); the terminal-vs-nothing distinction is made by the *caller* (`if case .doc(let path)? = focusedCardID`) and is therefore an [M] wiring concern, not a separable [K] assertion. The anti-regression that a focused *terminal* keeps ESC pass-through is enforced at the wiring level (the branch only matches `.doc`); the [K] test pins that `forFocusedDoc(false)` is `nil` (it must not return `.defocus`).

- **`Cmd+W` wiring in the keyDown monitor** (`AppController.swift` `start()`). New `isCmdW = event.keyCode == 13 && mods == .command` branch (W = ANSI virtual keycode 13, matching the existing `isCmdT`/`isCmdK`/`isCmdC` constants' inline-key-name convention) that maps `focusedCardID` + the board's other-live-terminal count through `FocusedClose.decide(...)`, executes the result, and **always returns `true` (swallow)**.

- **ESC branch change** (`AppController.swift` ~539–543). The focused-doc branch calls `defocus()` instead of `moveToShelf(path)`; the higher-priority branches above it are untouched.

### Data Flow

**`Cmd+W`:**

1. Monitor matches `isCmdW`. Map `focusedCardID` to a kind (`.doc` / `.term` / `.none`); when it is `.term(id)`, compute `otherLive` = count of live terminals on the active board excluding `id` (mirrors `handleExit`'s `otherLive`).
2. `action = FocusedClose.decide(kind:, otherLiveTerminals: otherLive)`.
3. Execute:
   - `.noop` → do nothing.
   - `.shelfDoc` → `moveToShelf(path)` for the focused doc path (removeCard + append to `shelfPaths` + persist + refresh; `moveToShelf` already clears the doc's focus).
   - `.closeTerminal(replace:)` → snapshot the card's world frame (`card.worldFrame`); send `client.termClose(termID: id)` (daemon SIGHUPs the group); tear down app-side **proactively** — `removeTerminalCard(termID:on:)` (also clears focus when it was focused) plus `termIndex.remove(termID: id)`. Then, per `replace`: `true` → `spawnTerminal(on: activeBoard, at: frame)` (one fresh shell, no toast); `false` → `showUndoToast(on: activeBoard, at: frame)` (existing single-terminal undo). Do **not** wait for the natural `Exit` — a SIGHUP exit reports as a signal (`code == nil`), which `TermExit.decide` would map to `.holdOpen`; the proactive teardown avoids that.
4. `persistLayout()` (when a card was actually closed); always `return true` (swallow — the window is never closed by `Cmd+W`).
5. Later, the daemon's `Exit` for the killed terminal arrives; `handleExit`'s guard (`board.sessions[termID]` is already `nil`) makes it a no-op.

**`ESC` (card-focused branch only):**

1. Monitor matches ESC; the higher-priority branches (cancelDrag → undock → fly-back → sendFreshCardToShelf → hide-peek → clear-toasts) run first, unchanged.
2. If none matched and a **doc** card is focused (`if case .doc(let path)? = focusedCardID`, card exists): `defocus()`, return `true`. The doc card **stays on the board**.
3. If a **terminal** is focused (or nothing is focused): the branch does not match → monitor returns `false` → ESC passes through to the first responder (the prime terminal program).

### Interface Contract

**New protocol message** (additive; conformance-gated in both languages):

```
// Rust — core/crates/tarmac-protocol/src/lib.rs, Msg enum
TermClose { term_id: String }          // app -> daemon; msgpack tag "t":"term_close"
// encoded via rmp_serde::to_vec_named (named map, string keys)

// Swift — app/Sources/TarmacKit/Messages.swift
case termClose(termID: String)         // encode "t":"term_close","term_id":<string>; decode the same
// app/Sources/TarmacKit/DaemonClient.swift
public func termClose(termID: String)  // send(.termClose(termID:))
```

A new conformance vector (**vector 9** — `docs/protocol.md`'s numbered list currently ends at 8) for `term_close` MUST be added and kept in sync across: `docs/protocol.md`, the Rust `tarmac-protocol` conformance test, and the Swift `TarmacKit` `ConformanceTests`. Per `docs/protocol.md`, the contract is **decode-to-the-same-structure + own-side round-trip of the shared pinned hex vector** — byte-exact *encoder* output is explicitly NOT required (Rust `to_vec_named` and Swift `MsgPack.encode` may legally order map keys differently). Do **not** assert `rustBytes == swiftBytes`. No existing key or encoding changes; existing vectors V1–V8 stay byte-for-byte unchanged.

**`FocusedClose` pure helper** (TarmacKit):

```
public enum FocusedClose {
    public enum Kind: Equatable { case doc, term, none }   // kind of the focused card
    public enum Action: Equatable {
        case noop                              // nothing focused → do nothing (still swallow the key)
        case shelfDoc                          // focused doc → moveToShelf(path)
        case closeTerminal(replace: Bool)      // focused terminal → kill+remove; replace iff it was the last live terminal
    }
    public static func decide(kind: Kind, otherLiveTerminals: Int) -> Action
}
```

Rules: `.none → .noop`; `.doc → .shelfDoc`; `.term → .closeTerminal(replace: r)` where `r == (TermExit.decide(code: 0, otherLiveTerminals: otherLiveTerminals) == .removeAndReplace)`, i.e. `r == (otherLiveTerminals == 0)`. `otherLiveTerminals` is ignored for `.none`/`.doc`.

**`EscFocusAction` pure rule** (TarmacKit):

```
public enum EscFocusAction {
    public enum Outcome: Equatable { case defocus }     // handled → swallow
    /// nil ⇒ ESC is NOT handled by the card-focus branch (pass through to the
    ///       prime terminal). Returned for a focused terminal AND for no focus.
    public static func forFocusedDoc(_ isDoc: Bool) -> Outcome?   // .defocus iff isDoc, else nil
}
```

## Alternatives Considered

### Cmd+W closes ALL cards on the board (issue's literal wording)
A whole-board sweep (shelf every doc, kill every terminal, respawn one shell, bulk undo). Rejected by the issue author in favor of closing only the focused card — simpler, matches the ✕/Cmd+W "remove this one" framing, and reuses the existing single-terminal undo/replace machinery instead of needing a bespoke multi-term undo.

### Reuse `board_delete` to kill the terminal
`board_delete` kills every PTY on the board and removes the board itself — wrong scope. A per-terminal message is required.

### Make the shell exit itself via `Input` (Ctrl-D / `exit`)
Reuse the existing `Input` message instead of adding a protocol message. Rejected: fragile — it won't terminate a busy foreground process or a non-shell program, and isn't a reliable PTY teardown.

### Let the natural `Exit` → `handleExit` path remove the terminal after `TermClose`
Send only `TermClose` and let the daemon's `Exit` drive removal. Rejected: a SIGHUP exit reports as a signal (code `nil`) → `TermExit.decide` returns `.holdOpen` (keeps a read-only placeholder). `Cmd+W` removes the card app-side proactively and treats the later `Exit` as a no-op.

### ESC also defocuses a focused terminal (issue's literal ESC AC)
Repoint the branch to defocus *any* focused card. Rejected by explicit decision: it swallows ESC for a focused terminal, so the program needs a second ESC to receive one — breaking agent-interrupt / vim in the agent cockpit. ESC defocuses **doc cards only**; terminal focus is unaffected and ESC passes through.

### Cmd+W with nothing focused falls back to closing the window
macOS-style "close tab, else close window." Rejected by decision: in board-navigation mode `focusedCardID == nil` is common, so a window-close fallback risks accidental window loss. `Cmd+W` with no focus is a swallowed no-op.

## Acceptance Scenarios

> Legend: **[K]** = pure-helper unit test (TarmacKit, XCTest); **[R]** = Rust daemon/protocol test; **[C]** = cross-language conformance; **[M]** = manual/e2e (AppKit wiring, not unit-testable per repo convention — `/verify` checks these by reasoning + build, not an automated assertion).

### Happy Path — Cmd+W (pure decision)
- **S1 [K]:** Given a focused **doc** card, when `FocusedClose.decide(kind: .doc, otherLiveTerminals: n)` is computed (any `n`), then it returns `.shelfDoc`.
- **S2 [K]:** Given a focused **terminal** with **other** live terminals remaining, when `FocusedClose.decide(kind: .term, otherLiveTerminals: n)` is computed with `n ≥ 1`, then it returns `.closeTerminal(replace: false)`.
- **S3 [K]:** Given a focused **terminal** that is the board's **last** live terminal, when `FocusedClose.decide(kind: .term, otherLiveTerminals: 0)` is computed, then it returns `.closeTerminal(replace: true)`.
- **S4 [K]:** Given **nothing** focused, when `FocusedClose.decide(kind: .none, otherLiveTerminals: n)` is computed (any `n`), then it returns `.noop`.
- **S5 [K]:** Boundary — the replace flag flips exactly at the last terminal: `decide(kind: .term, otherLiveTerminals: 0)` is `.closeTerminal(replace: true)` and `decide(kind: .term, otherLiveTerminals: 1)` is `.closeTerminal(replace: false)` (anti-mutation pin that `replace == (otherLiveTerminals == 0)`).

### Happy Path — Cmd+W (wiring) and the kill path
- **S6 [M]:** Given a focused doc card, when `Cmd+W` is pressed, then the doc card is removed from the board and lands on the shelf, can be restored from the shelf (not permanently destroyed), and no card remains focused.
- **S7 [M]:** Given a focused terminal card with at least one other live terminal on the board, when `Cmd+W` is pressed, then that terminal card is removed, its PTY is terminated, a single undo toast is shown, and **no** replacement terminal is spawned.
- **S8 [M]:** Given a focused terminal card that is the board's only live terminal, when `Cmd+W` is pressed, then that terminal card is removed, its PTY is terminated, exactly **one** fresh replacement terminal is spawned (≥1-live-terminal invariant holds), and **no** undo toast is shown.
- **S9 [R]:** Given a terminal spawned in the daemon and shown live *before the close* (it echoed an `Input`, establishing a baseline — the `board_delete` suite's idiom), when a `TermClose { term_id }` is received, then the daemon SIGHUPs that term's process group and its pump emits an `Exit` for that `term_id`. The emitted `Exit` is the **primary** teardown proof (it is what the existing `board_delete_kills_terms_fixes_active_and_refuses_last` asserts — a live pty does not emit `Exit` until its child is reaped); the test MUST fail if no `Exit` arrives within the deadline. The live-before baseline makes the `Exit` a genuine transition, not a vacuous match. A pid-introspection / orphan check MAY be added where the harness allows it, but is not required.

### Happy Path — ESC
- **S10 [K]:** Given a focused **doc** card, when the ESC card-focus rule is evaluated (`EscFocusAction.forFocusedDoc(true)`), then it returns `.defocus`.
- **S11 [K]:** Given **not** a focused doc (`EscFocusAction.forFocusedDoc(false)` — covers both a focused terminal and nothing focused), then it returns `nil` (not handled → pass through). *Do not add a second `forFocusedDoc(false)` test; the terminal-vs-nothing distinction is [M] wiring, see S13.*
- **S12 [M]:** Given a focused doc card, when `ESC` is pressed (and no higher-priority ESC behavior is active), then the card's focus is cleared (teal border gone) and the card **remains on the board** — it is **not** sent to the shelf.

### Edge / Regression Scenarios
- **S13 [M]:** Given a focused **terminal** card, when `ESC` is pressed (and no higher-priority ESC behavior is active), then the terminal is **not** defocused and the ESC reaches the terminal's program (agent-interrupt / vim pass-through preserved). Anti-regression for the doc-only ESC rule.
- **S14 [M]:** Given **no** card focused, when `Cmd+W` is pressed, then nothing is removed, no crash/error occurs, and the app window is **not** closed (the event is swallowed; it does not fall through to the macOS close-window action).
- **S15 [M]:** Given **no** card focused, when `ESC` is pressed (and no higher-priority ESC behavior is active), then nothing is removed or thrown and the ESC passes through (no card-focus handling fires).
- **S16 [M]:** Given any higher-priority ESC behavior is active (an in-progress board drag/resize, a docked terminal, a pending fly-back viewport, a visible peek panel, or on-screen toasts), when `ESC` is pressed, then that higher-priority behavior fires and takes precedence over the doc-defocus branch — the cascade priority order is unchanged.
- **S17 [R/C]:** Given the new `term_close` conformance vector (vector 9), when it is decoded in both Rust and Swift, then both decode to the same structured value and each side round-trips it (encode→decode) successfully; no `rustBytes == swiftBytes` assertion is made; and all existing vectors V1–V8 remain byte-for-byte unchanged.
- **S18 [R]:** Given a `TermClose` for an **unknown** `term_id`, when the daemon handles it, then it is a benign no-op with an observable post-condition: an unrelated live terminal still echoes its input and still appears in `board_list` (the test fails if an unknown id panics the daemon or tears down a real terminal).

## Trade-offs and Limitations

- **`Cmd+W` no longer closes the app window via keyboard.** By decision, `Cmd+W` is always swallowed; the window is closed via the red traffic-light button or `Cmd+Q`. The Window-menu "Close" item (key `w`, `main.swift`) still closes the window when clicked, but its `⌘W` shortcut is intercepted — consider relabeling/removing the menu key-equivalent to avoid a misleading shortcut hint (out of scope to require; noted for the implementer).
- **Closing the last terminal immediately respawns a fresh shell** (the existing `removeAndReplace` invariant), so `Cmd+W` on the last terminal "resets" rather than "empties." Intentional — boards keep ≥1 live terminal.
- **Terminal undo is lossy** (fresh shell; live process and scrollback are not recovered) — identical to the existing single-terminal undo toast. Doc close is recovered from the shelf, not via a toast.
- **ESC does not defocus a focused terminal** (by decision), so there is no keyboard-only way to drop scroll-focus on a terminal; clicking the empty board still does. This is the deliberate cost of preserving ESC pass-through to terminal programs.
- **`TermClose` is fire-and-forget** — the app does not wait for the daemon's `Exit`; it removes the card proactively and ignores the late `Exit`. A `term_id` the daemon doesn't know is a silent no-op.

## Security Considerations

`TermClose` only signals (SIGHUP) a process group the daemon already owns and that is addressable solely by an opaque `term_id` over the local Unix socket; it adds no new capability beyond what `board_delete` already exposes. No new external surface, auth, or data exposure.
