# 2606.0015 Dev Runtime Dir & Socket-Path Length Guard

**Date:** 2026-06-30
**Status:** draft
**Author:** EP Lin
**Issues:** [#25](../../docs/) (dev runtime dir), [#20](../../docs/) (socket-path length guard)

## Context

Two small, related papercuts in the dev socket/state plumbing.

**#25 — dev runtime dir.** `make run` and `make kill-daemon` currently derive the dev socket under `$HOME` from an 8-hex `shasum` of `$(ROOT)`:

```make
# Makefile run target (lines 40-47) and kill-daemon (51-59)
wt="$$HOME/Library/Application Support/tarmac/dev/wt-$$(echo -n "$(ROOT)" | shasum | head -c8)"
TARMAC_SOCKET="$$wt/tarmacd.sock" TARMAC_STATE="$$wt/state.json"
```

That `wt-<8hex>` scheme is indirection for a property each worktree already has for free — its own directory. Routing the dev socket back under `$HOME` also burns bytes against the 104-byte `sockaddr_un.sun_path` cap (a long `$HOME` overflows) and scatters dev state far from the checkout it belongs to.

**#20 — socket-path length guard.** There is **no explicit length check anywhere in Rust today**. The macOS `sun_path` cap (104 bytes) is enforced only by the OS at `UnixListener::bind` (`tarmacd/src/main.rs:~71`) and `UnixStream::connect` (`tarmac-cli/src/main.rs:~95`, desktop `bridge.rs:~331`), surfacing as a generic `io::Error` / `"connect failed: {e}"`. A developer who sets an over-long `TARMAC_SOCKET` gets an opaque OS error with no hint that the path length is the cause or that a shorter path is the fix.

The shared pure resolvers `resolve_socket_path` / `resolve_state_path` (`tarmac-protocol/src/lib.rs:~296-333`) stay as-is — non-empty `TARMAC_SOCKET`/`TARMAC_STATE` win verbatim. This spec touches the Makefile, `.gitignore`, and adds **one** pure validator alongside the resolvers; the wire protocol is untouched.

## Motivation

The only new behavior worth unit-testing is the **pure length validator**: given a path, is it within the 104-byte budget? The accept/reject boundary is the single correctness property (103 = OK, 104 = rejected), and it must carry an *actionable* message (names the byte length vs the 104-byte cap, and the remedy: set `TARMAC_SOCKET` to a shorter path). Per project convention that decision is a pure function in the shared `tarmac-protocol` crate, table-driven unit-tested; the impure shell (the `bind`/`connect` call sites that invoke it, the Makefile env, the `.gitignore` line) is integration-verified.

Keeping the validator **separate** from `resolve_socket_path` is deliberate SRP: resolution (env override vs default) and validation (byte budget) are distinct concerns with distinct tests. The Makefile change is pure plumbing — switching the dev runtime dir from a hashed `$HOME` subdir to `$(ROOT)/.dev/` preserves per-worktree isolation by construction (each worktree owns its `$(ROOT)`) while shedding both the `shasum` indirection and the byte pressure.

## Proposed Solution

### Overview

**A (#25):** Point `make run` and `make kill-daemon` at `$(ROOT)/.dev/tarmacd.sock` and `$(ROOT)/.dev/state.json` — absolute, cwd-safe (`$(ROOT)` is `$(abspath ...)`), and per-worktree by construction. Delete the `wt-<8hex>` / `shasum` logic entirely. Add `.dev/` to the root `.gitignore`.

**B (#20):** Add a pure `check_socket_path_len(path: &Path) -> Result<(), String>` to `tarmac-protocol`. Reject when the path's byte length `>= 104`. Call it just before `bind` in `tarmacd` and before `connect` in `tarmac-cli` and the Tauri `bridge`, surfacing the actionable message in place of the generic OS error.

### Key Components

- **`Makefile` `run` target (lines 40-47)** — replace the `wt=...shasum...` line and the `TARMAC_SOCKET`/`TARMAC_STATE` exports so both point under `$(ROOT)/.dev/`. No `wt-`, no `shasum`.
- **`Makefile` `kill-daemon` target (lines 51-59)** — derive the same `$(ROOT)/.dev/tarmacd.sock` for the `lsof`/`kill`. No `wt-`, no `shasum`.
- **`.gitignore` (root)** — add `.dev/`.
- **`check_socket_path_len` (`tarmac-protocol/src/lib.rs`, new pure fn)** — beside `resolve_socket_path`; `Ok(())` for `< 104` bytes, `Err(msg)` for `>= 104`. The `msg` names the derived path's byte length, the 104-byte `sun_path` cap, and the remedy (set `TARMAC_SOCKET` shorter, e.g. under `/tmp`). Unit-tested in the crate's existing `#[cfg(test)] mod`.
- **`tarmacd` shell (`main.rs:~71`)** — call `check_socket_path_len(&path)` immediately before `bind`; on `Err`, surface the message instead of letting `bind` fail with a bare OS error.
- **`tarmac-cli` shell (`main.rs:~95`) + Tauri `bridge` (`bridge.rs:~331`)** — call the validator before `connect`, surfacing the actionable message in place of `"connect failed: {e}"`.

### Interface Contract

```rust
/// PURE length guard for a Unix-domain socket path. macOS caps
/// sockaddr_un.sun_path at 104 bytes; bind/connect fail opaquely past it.
/// Ok(())  iff path.as_os_str().len() < 104  (103 = OK).
/// Err(msg) iff len >= 104 (104 = rejected); msg names the byte length,
/// the 104-byte cap, AND the remedy (set TARMAC_SOCKET shorter, e.g. /tmp).
/// SEPARATE from resolve_socket_path: resolution vs validation (SRP).
pub fn check_socket_path_len(path: &std::path::Path) -> Result<(), String>
```

The boundary is exactly the existing S8 convention: `len < 104` accepts, so 103 bytes is OK and 104 bytes is rejected. The resolvers are unchanged — no override-precedence change, no new key, no encoding change.

## Acceptance Scenarios

### Happy Path
- **S1 (dev runtime dir is the worktree's `.dev/`):** The Makefile `run` target sets `TARMAC_SOCKET` to `$(ROOT)/.dev/tarmacd.sock` and `TARMAC_STATE` to `$(ROOT)/.dev/state.json`, and `kill-daemon` derives `$(ROOT)/.dev/tarmacd.sock` — mechanically: grep the `run` and `kill-daemon` targets and confirm both reference `$(ROOT)/.dev/` and contain **no** `wt-` and **no** `shasum`.
- **S2 (`.dev/` is gitignored):** The root `.gitignore` contains a `.dev/` entry, so a dev socket/state file written into a worktree's `.dev/` never shows up in `git status`.
- **S3 (validator accepts within budget):** Given a path of **103** bytes, `check_socket_path_len` returns `Ok(())`.

### Edge Cases
- **S4 (validator boundary — 104 rejected):** Given a path of **104** bytes, `check_socket_path_len` returns `Err(_)`. This pins the exact `len < 104` boundary shared with the existing S8 byte-boundary tests (103 accepted, 104 rejected — off-by-one in either direction fails this scenario together with S3).
- **S5 (per-worktree isolation preserved):** Two worktrees with distinct `$(ROOT)` resolve distinct `$(ROOT)/.dev/tarmacd.sock` paths — isolation now comes from the worktree dir, not a `shasum`. (Manual/scripted: two `make run`s from different worktrees don't share a socket or state file.)

### Error Scenarios
- **S6 (rejection message is actionable):** The `Err` string from S4 contains both `"104"` (the cap / byte budget it names) **and** `"TARMAC_SOCKET"` (the remedy it points at) — a bare or constant message fails this. The point is that an over-long path explains *why* and *how to fix*, not a raw OS error.
- **S7 (validator wired before bind/connect):** With an over-long `TARMAC_SOCKET`, `tarmacd` (before `bind`), `tarmac-cli` and the Tauri `bridge` (before `connect`) surface the actionable S6 message rather than the generic `io::Error` / `"connect failed: {e}"`. (Inspect the call sites; integration-verified, not unit-tested.)

### Regression
- **S8 (`dev_socket_byte_boundary` still holds):** The existing S8 unit test `dev_socket_byte_boundary` (`tarmac-protocol/src/lib.rs:~1380`) pinning the default dev path byte boundary stays green, untouched.
- **S9 (stale per-worktree boundary test removed):** The S8b test `per_worktree_dev_socket_byte_boundary` (`~1402`) hardcodes `dev/wt-0123abcd/tarmacd.sock` byte counts pinning the **removed** Makefile `wt-` scheme — it must be removed (or replaced by the S3/S4 validator boundary test), since it would otherwise pin a scheme that no longer exists.

- **Behavioral over structural** — assert the returned `Result` and message substrings, not internals.

## Trade-offs and Limitations

- **`$(ROOT)/.dev/` lives inside the checkout.** A clutter `.dev/` dir now sits in each worktree (gitignored). That is the trade for shedding the `shasum` indirection and the `$HOME` byte pressure — and it puts dev state next to the code it belongs to.
- **The validator only makes overflow *legible*; it is not isolation.** `check_socket_path_len` rejects an over-long path with a clear message — it does not shorten it. Routing the dev socket under `$(ROOT)/.dev/` is what reclaims the bytes; the guard is the backstop for an explicit over-long `TARMAC_SOCKET`.
- **The wiring shell is unverified by this gate.** The three `bind`/`connect` call sites, the Makefile env, and the `.gitignore` line are integration-verified, not unit-tested. Pure decision here, I/O there.
- **Rollback is trivial.** Revert the diff. Stale `~/Library/Application Support/tarmac/dev/wt-*` dirs from the old scheme are orphaned harmlessly (no process references them once `make run` points at `.dev/`); delete at leisure. *(No Security or Migration section — not applicable.)*
