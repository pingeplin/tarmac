# 2606.0010 Doc-card self-scale crispness — screen-space layer, one var-driven wrapper transform

**Date:** 2026-06-27
**Status:** superseded by 2606.0011
**Author:** EP Lin

## Context

Predecessor **2606.0009** (`d331ee3`) stopped doc-card reflow by laying prose out **once** at fixed metrics and promoting the subtree with `translateZ(0)`. Reflow is gone, but text is **slightly blurry at high zoom**: the prose sits under the ancestor `.world` `transform: scale(zoom)` (`BoardEngine.ts:258`), and WebKit rasters a layer that is **not itself scaling** at 1× density, then bilinear-stretches it. A nested-canceller experiment only partially fixed it (WebKit settles raster against the intermediate layer → raster ceiling).

The crisp PoC condition (Panel A) is a **single self-scaling layer with no scaled ancestor**. Terminal cards are already crisp via canvas oversample (`rasterScale.ts`) — verified — and must not move.

## Motivation

The 2606.0009 reflow fix shipped a known crispness regression. This delivers the missing density without reintroducing reflow, and without the engine and React fighting over card position.

## Proposed Solution

**A-scoped:** render **only** doc cards in a screen-space layer `.doc-layer` (sibling of `.world`, **no transform**). Each doc card is a **wrapper** that carries the *entire* visual scale via **one CSS-var transform** composed from global engine vars + per-card static vars. No per-frame per-card JS writes; no `left/top` for React and the engine to fight over.

### Interface Contract

**(a) Per-card wrapper transform — the single scale (whole card box: chrome + border + anchors + prose):**
```
transform: translate(var(--world-tx), var(--world-ty)) scale(var(--zoom)) translate(var(--card-x), var(--card-y));
transform-origin: 0 0;   /* load-bearing for the matrix math */
```
Emitted by a **pure kit fn `docWrapperTransform({x,y})`** returning the exact string above (with `${x}px`/`${y}px` baked into the inner translate) so it is unit-testable. Exactly **one** `scale(var(--zoom))` (the crispness lever); **never `will-change`** anywhere on the doc subtree (pins raster at 1× → permanent blur — 2606.0009 S6 finding still holds).

**(b) Engine global vars (set ONCE per frame, alongside `--zoom` at `BoardEngine.ts:265`):** `--world-tx`/`--world-ty` = the **same** `tx,ty` translate already computed for the `.world` transform at line 258, exposed as CSS vars. One extra `setProperty` pair per frame, **global — not per card**.

**(c) DocCard:** renders into `.doc-layer` as the per-card wrapper carrying the transform above (built by calling `docWrapperTransform({x:frame.x, y:frame.y})` — production MUST call the kit fn, not inline a copy, so S1/S2 actually guard the rendered transform), sized **world-px** `width:${frame.w}px; height:${frame.h}px`. The wrapper's world-px position is baked into the transform's inner `translate(${frame.x}px,${frame.y}px)` (no separate `--card-x/--card-y` vars); only `--world-tx/--world-ty/--zoom` are CSS vars (global, engine-set). `CardShell` nests **inside the wrapper**.

**(c′) CardShell gains ONE optional, backward-compatible prop (e.g. `inWrapper`/`positionless`):**
- When set (doc path): CardShell renders at **static 0,0** and **omits its own `left/top`** (line 162) — the wrapper owns position; but its drag/resize math **still reads the real `frame` + `getZoom()`** and commits world-coord frames exactly as today (header: `frame.x + (dxScreen)/zoom`; resize: `resizeFrame(frame, handle, dxScreen/zoom, …)`, lines 86/97–98/127/133–136). A committed frame → React updates `--card-x/--card-y` → the wrapper repositions. **No engine per-card writes.**
- When **unset** (terminal path): byte-for-byte unchanged — terminals don't pass the prop, so their `left:frame.x/top:frame.y` path is untouched.

This is **not** "CardShell untouched" — it is a single additive prop; only the doc path's position-source changes.

**(d) Prose layer-promotion REMOVED:** under Variant A the per-card wrapper is the **sole** self-scaling layer. Do **not** separately promote the prose — delete the predecessor's `translateZ(0)` / `docScalerStyle` promotion so the prose paints **into the wrapper's high-density backing** (the exact PoC condition: text painted into the single scaled element). The prose has **no additional scale** → no zoom². 2606.0009's `docScalerStyle()` no-scale/no-willChange tests are **superseded** by `docWrapperTransform`.

**(e) `.doc-layer` CSS:** `position:absolute; inset:0; pointer-events:none`, with `.doc-layer .card { pointer-events:auto }`.

**Supersedes the imperative `setDocShadows` mechanism** from the prior draft — no engine per-card position writes.

### Why this fixes both blockers (terse proofs)

- **Box vs prose / crispness:** the scale is on the wrapper = the whole card scales uniformly by zoom, exactly as under `.world scale(zoom)` before. Single self-scaling layer + un-transformed `.doc-layer` ancestor → WebKit re-rasters text at high density = crisp. No inner second scale → no zoom².
- **Edges connect:** with origin `0 0`, a local offset `(lx,ly)` maps to `world-tx + (card-x+lx)·zoom`. Since `tx = w/2 − cx·zoom` (the same `tx` `worldToView`/`boardTransform.ts` uses), that equals the projected world point. Card center (`lx = w/2`) → the projected `center` — exactly where `EdgeLayer` (in `.world`, scaled by zoom) draws the endpoint. Edges stay attached at every zoom. `EdgeLayer.tsx` **unchanged**.
- **No React/engine fight:** engine never writes per-card `left/top`; position is the var-driven transform. React owns only `--card-x/--card-y` (true world coords, render-independent); re-setting them on a recency-tick / markdown / selection re-render is a no-op → no snap.

**Inverts 2606.0009 S2/S3/S7a:** the scaler now *intentionally* carries `scale(var(--zoom))`; the old "no per-card scale → avoid zoom²" reasoning is superseded (docs no longer have a scaled ancestor, so the wrapper's own scale **is** the only scale). Update those expectations.

### Known limitation (z-stacking)

`.doc-layer` paints **above** `.world`, so doc cards stack above all terminals/edges regardless of `model.z`. **ACCEPTED for now** (Open Question below). "Hit-testing unchanged" holds for non-overlapping cards.

## Acceptance Scenarios

- **S1 — wrapper transform string:** Given `docWrapperTransform({x,y})`, then it returns exactly `translate(var(--world-tx),var(--world-ty)) scale(var(--zoom)) translate(<x>px,<y>px)` with `transformOrigin:"0 0"`.
- **S2 — exactly one scale, never will-change:** the transform contains exactly one `scale(var(--zoom))`; `"willChange"` is absent from the style object (and from the prose subtree — no `docScalerStyle`/`translateZ(0)` promotion remains).
- **S3 — layout zoom-independent (no reflow):** `docProseLayout()` returns the fixed constants and takes no zoom argument; the world-px box derives from `frame`, not zoom — wrap points cannot move.
- **S4 — world→screen projection:** Given a pure `worldToLocal`-style helper (`cx,cy,zoom,viewport w/h`) and a world point, it returns the screen-space coord where the wrapper places that point — i.e. the same screen point the `.world` transform projects EdgeLayer's *world-coordinate* endpoint to (EdgeLayer keeps drawing in world space; the proof is wrapper-transform ≡ `.world`-transform, sharing `--world-tx/--world-ty/--zoom`). This pure projection already exists as `worldToView` (`kit/boardTransform.ts`, `view = (p − center)·zoom + viewportCenter`) — reuse it rather than adding a duplicate in `docZoom.ts`. Unit-tested for representative pan/zoom.
- **S5 — per-card wrapper scales whole box; prose not separately promoted:** the single `scale(var(--zoom))` sits on the per-card wrapper, so box + chrome + prose scale together; no inner/second scale anywhere in DocCard, and the prose carries **no** `translateZ(0)`/`docScalerStyle` promotion (it paints into the wrapper's scaled backing — the PoC condition). *(Structural: review + the S2 single-scale assertion.)*
- **S6 (edge) — edges connect at any zoom:** consistent with S5/S4 — card center maps to the same screen point `worldToView(center)` the EdgeLayer endpoint uses, at zoom≠1. *(Manual-QA.)*
- **S7 (edge) — position render-independent (no snap) AND drag/resize preserve world coords:** (i) card position comes only from the var-driven transform with React-owned `--card-x/--card-y`; a recency/markdown/selection re-render re-sets the same vars (no-op) → no position snap. (ii) Dragging the header or a resize handle on a doc card commits the **same** world-coord frame as today — CardShell's `frame.x + dxScreen/zoom` drag math and `resizeFrame(...)` run unchanged under the `inWrapper` prop — and the wrapper follows via the updated vars. *(Structural + manual-QA; replaces the old imperative-teleport scenario.)*
- **S8 (regression) — terminals still crisp:** terminal cards at 2.5–3× stay crisp via oversample; `TerminalCard`/`rasterScale` and the terminal `CardShell` path (prop unset → `left:frame.x/top:frame.y` untouched) unchanged. *(Manual-QA.)*

## Trade-offs and Limitations

- **Z-interleaving across layers deferred:** docs always paint above terminals/edges (see Known limitation + Open Question). Acceptable today (docs and terminals rarely overlap by design).
- App-layer JSX (`.doc-layer`, DocCard wiring) stays untested per repo convention; the pure transform + projection fns carry the tested contract.

## Alternatives Considered

- **A-full / unifying terminals onto the doc self-scale model — rejected (deferred).** Terminal cards render to a `<canvas>`; a CSS `scale()` only bilinear-stretches the buffer, so self-scale does nothing for them — canvas crispness comes from oversampling the buffer (`rasterScale`), which they already have. Forcing terminals onto the doc mechanism would blur them (drop oversample) or add complexity (keep both) for zero crispness gain, while risking the subtle terminal path (BCR selection, IME, fit, dock). A unified `.card-layer` (all cards as CSS-var self-scaling wrappers in one screen-space layer) is cheap per-frame (one global var write drives every card via CSS, no per-card JS) and would *additionally* fix the z-stacking limitation — but is deferred until doc self-scale crispness is validated in-app. `.doc-layer` is the stepping stone toward it, not a dead-end.
- **Variant B (one shared `scale` on the whole `.doc-layer`, cards positioned within) — rejected.** Simpler (single scale layer, cards just need `--card-x/--card-y`), but a card-SIZED scaled element is exactly what re-rastered crisply in the PoC; one large shared scale layer is a weaker crispness bet and risks backing-store size limits at high zoom. Variant A (per-card wrapper) is the PoC-faithful choice.
- **Imperative `setDocShadows` (engine writes per-card `left/top` each frame) — rejected.** Two position writers (React + engine) fight over `left/top` → stale-frame teleport and re-render snap. The var-driven wrapper transform has no per-frame per-card writes and a single position owner (React's `--card-x/--card-y`).
- **Nested canceller (keep docs in `.world`, counter-scale inner layer) — rejected.** WebKit settles raster against the intermediate layer → partial-raster ceiling, never fully crisp.
- **Status quo 2606.0009 (`translateZ(0)` under ancestor scale) — rejected.** Reflow-free but blurry at high zoom; this is its crispness follow-up.

## Open Questions

- **Z-interleaving across the two layers:** docs always paint above terminals/edges. Deferred; revisit if overlapping doc/terminal stacks become common (would need a unified paint order or projecting both layers).
- **Settle timing:** the self-scale layer re-rasters on WebKit's own settle, independent of `RASTER_SCALE_SETTLE_MS` — confirm in QA that crispness lands fast enough to feel right.
