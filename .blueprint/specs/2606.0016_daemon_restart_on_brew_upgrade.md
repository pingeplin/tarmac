# 2606.0016 — Restart daemon on brew upgrade

- **Date:** 2026-06-30
- **Status:** draft
- **Author:** EP Lin
- **Issue:** #44 "restart daemon on brew upgrade"

## Context

Tarmac runs as two binaries that ship together from one Cargo workspace:
`tarmacd` (the long-lived daemon) and `TarmacApp` (the Tauri cockpit, which
auto-spawns the daemon). `brew upgrade tarmac` replaces both binaries on disk,
but the *already-running* `tarmacd` keeps running. On relaunch the app loads the
new frontend yet reconnects to the stale daemon over the existing socket, leaving
a mismatched app+daemon pair. New daemon fixes silently go undelivered until the
user manually kills `tarmacd`. There is no version signal on the wire today —
`tarmacd` ignores `argv` entirely and has no version constant — so the app cannot
even detect the mismatch.

## Motivation

A user who upgrades expects the upgrade to take effect. Today it does not for the
daemon half, with no visible cue. We want: after upgrade + relaunch, the running
`tarmacd` is the freshly installed binary, the app detects a mismatch on connect
and restarts the daemon gracefully, and no doc cards or layout are silently lost
during the handoff.

## Proposed Solution

### Overview

Stamp a build version into the daemon, surface it during the handshake, and have
the app compare the daemon's reported version against its own expected version on
every connect. On mismatch the app SIGTERMs the stale daemon, spawns the
newly-installed binary, and reconnects — emitting a visible daemon-status event so
the restart is not silent. Restored state (doc cards, layout, viewport) is replayed
from `state.json` on the new connection.

### Key Components

1. **Daemon version, CLI + wire.** `tarmacd` gains a `--version` flag that prints
   `env!("CARGO_PKG_VERSION")` and exits 0 (main currently ignores argv). The same
   value is surfaced on the wire (below). Version is inherited from
   `core/Cargo.toml` `[workspace.package] version` via `version.workspace = true`.

2. **Protocol field.** Add an additive optional field to the existing handshake
   reply in `tarmac-protocol`:
   `HelloOk { v: u32, daemon_version: Option<String> }` with
   `#[serde(default, skip_serializing_if = "Option::is_none")]`. `HelloOk` is **not**
   a byte-pinned conformance vector (V1–V10), so no conformance vectors change. The
   daemon populates `daemon_version` in `conn.rs::handshake()` with
   `env!("CARGO_PKG_VERSION")`.

3. **App expected version.** `desktop/src-tauri/src/bridge.rs` exposes its own
   `env!("CARGO_PKG_VERSION")` (same workspace) as the expected daemon version.

4. **Mismatch detection + restart.** In `run_connection()`, after the app sends
   `Hello{role:"app"}`, the daemon's `HelloOk` arrives as the first inbound frame.
   A **pure** decision function gates the restart:

   ```rust
   /// true  => stale daemon, must restart-and-reconnect
   /// false => versions compatible OR already restarted once (proceed / surface)
   ///
   /// `already_restarted` is the restart-once latch: once the app has restarted
   /// the daemon for this mismatch, a still-mismatched reconnect returns `false`
   /// (so the app surfaces the persistent-mismatch state instead of looping).
   fn should_restart(expected: &str, reported: Option<&str>, already_restarted: bool) -> bool;
   ```

   Returns `true` only when the reported version differs from `expected`
   (or is `None`) **and** `already_restarted` is `false`. On `true`, the app SIGTERMs the tracked daemon child, **waits for the
   socket to disappear** (the old daemon removes it on `exit(0)`; poll with a
   bounded timeout), then spawns the freshly-installed binary, and lets the
   existing `connection_loop` backoff reconnect. The wait is load-bearing:
   `claim_socket()` in the daemon `exit(1)`s if it can still connect to a live
   daemon on the socket, so spawning before the old daemon releases the socket
   leaves the stale daemon in place. `spawn_daemon()` must retain the `Child`/pid
   (e.g. `Mutex<Option<Child>>` on `Bridge`) instead of discarding it, and the
   `spawned` latch must reset so `connect()` re-spawns.

5. **Handoff safety.** On SIGTERM the daemon removes its socket and `exit(0)`. Doc
   cards, layout, and viewport are persisted continuously to `state.json`
   (`persist.rs` save_loop, 150ms debounce, atomic write) and replayed via
   `Msg::Restore` on the next connect. Live PTY shells already die on any daemon
   restart (documented, deliberate); a version mismatch only occurs right after
   upgrade+relaunch — i.e. at app launch when terminals are cold-restored anyway —
   so no *live* terminal is lost beyond existing behavior.

### Data Flow

upgrade replaces binaries → app relaunches → `connect()` → `Hello{app}` →
daemon replies `HelloOk{daemon_version}` →
`should_restart(expected, reported, already_restarted)` →
**false:** normal session; **true:** emit daemon-status `{connected:false,
reason:"version mismatch / restarting"}` → SIGTERM old child → wait for socket
removal (bounded) → spawn new binary → reset `spawned` → backoff reconnect →
`HelloOk` now matches → `Msg::Restore` replays docs+layout.

### Interface Contract

- `tarmacd --version` → prints version, exits 0.
- `HelloOk { v: u32, daemon_version: Option<String> }`; serialized **without** the
  key when `None`; decoding a key-less `HelloOk` yields `daemon_version = None`.
- `fn should_restart(expected: &str, reported: Option<&str>, already_restarted: bool) -> bool` —
  `true` iff (`reported != Some(expected)`) **and** `!already_restarted`.
- daemon-status event carries `{connected: bool, reason: Option<String>}`.

## Acceptance Scenarios

### Happy Path

- **S1** — Given a built daemon, when `tarmacd --version` runs, then it prints
  `CARGO_PKG_VERSION` and exits 0.
- **S2** — Given the daemon, when an app completes the handshake, then `HelloOk`
  includes `daemon_version` equal to the daemon's build version.
- **S3** — Given app and daemon at the same version, when the app connects, then
  `should_restart` is `false`, no restart occurs, and a normal session proceeds.
- **S4** — Given a version mismatch, when the app connects, then it emits a
  daemon-status "restarting" event, SIGTERMs the stale daemon, spawns the new
  binary, reconnects, and ends bound to the new version (verifiable via `ps` /
  `tarmacd --version`).
- **S5** — Given a completed restart, when the new daemon sends `HelloOk`, then the
  app receives `Msg::Restore` and doc cards + layout reappear (not lost).

### Edge Cases

- **S6** — Given `daemon_version = None` (old, pre-version daemon that omits the
  key), when the app connects, then `should_restart` returns `true` — an absent
  version is definitionally stale — and the restart path runs.
- **S7** — Given encode/decode of `HelloOk`, when `daemon_version` is `None`, then
  the wire bytes contain no `daemon_version` key and a round-trip preserves `None`
  (backward compatibility / additive-key rule).

### Error Scenarios

- **S8** — Given the freshly-spawned daemon still reports a mismatch (e.g. PATH
  resolves a stale binary), when the app reconnects with `already_restarted =
  true`, then `should_restart` returns `false` (the restart-once latch) so no
  second restart fires; the app surfaces the persistent-mismatch state via
  daemon-status rather than thrashing.
- **S9** — Given a restart in progress, when the app spawns the new binary, then it
  does so only after the old daemon's socket has been removed (or a bounded
  timeout elapses), so the new daemon's `claim_socket()` does not see a live
  daemon and `exit(1)` — the running daemon ends up being the new binary.

## For the Implementing Agent

- Keep the version-compare + restart decision in the **pure** `should_restart`
  function so it is unit-testable without spawning processes — per repo memory,
  only pure kit/Rust logic is unit-tested; the Tauri shell is not.
- Add `should_restart` unit tests covering, with `already_restarted = false`:
  equal versions → false; differing versions → true; `None` → true. Plus the
  restart-once guard: differing versions (or `None`) with `already_restarted =
  true` → false.
- Add a `tarmac-protocol` round-trip test for the additive `daemon_version` field
  (present/absent), asserting no conformance vector (V1–V10) changes.
- The full SIGTERM/spawn/reconnect path is integration/manual: verify with
  `ps`/`tarmacd --version` after a simulated upgrade. Reset the `spawned` latch and
  retain the `Child` handle; do not hold the daemon Mutex across `.await`.
- Update `docs/protocol.md` handshake section and the `docs/architecture.md`
  "what survives a daemon restart" table.

## Definition of Done

- `tarmacd --version` works; `daemon_version` flows over the handshake.
- `should_restart` exists, is pure, and is unit-tested (incl. `None` case).
- Protocol round-trip test passes; conformance vectors unchanged.
- App restarts the daemon on mismatch, emits a visible status event, reconnects,
  and replays state via `Msg::Restore`; restart-once guard prevents looping.
- `make` passes; docs updated.

## Alternatives Considered

- **Homebrew cask `postinstall`/`uninstall` stanza that kills the daemon.** Moves
  the contract into packaging (no stanzas exist today) and fires even when the app
  is mid-session; the in-app check is OS-portable and keeps the truth in the
  app↔daemon handshake.
- **Daemon self-detects its own binary mtime and self-exits.** Couples the daemon
  to its install path and races the app; the app already owns daemon lifecycle.
- **Bump `PROTOCOL_VERSION` per release.** Conflates wire compatibility with build
  identity; the protocol is intentionally additive and rarely changes.

## Trade-offs and Limitations

- Live PTY shells are lost on the handoff — unchanged from today's documented
  behavior, and mismatch only arises at cold launch. "No silent loss" means: layout
  + doc cards restored from `state.json`, plus a visible status signal.
- Requires app and daemon to ship from the same workspace version (already true).
- Restart-once guard means a genuinely broken install surfaces an error rather than
  auto-healing — acceptable, and visible.
- **Version-bump dependency.** The restart logic compares `env!("CARGO_PKG_VERSION")`
  values. `scripts/release.sh` stamps `$VERSION` into `core/Cargo.toml`
  `[workspace.package]` and `desktop/src-tauri/Cargo.toml` `[package]` at build
  time (before `bundle.sh` runs), so `CARGO_PKG_VERSION` reflects the shipped
  version. If the release flow is bypassed (e.g. building ad-hoc without
  `release.sh`), both Cargo versions remain at their committed dev value and the
  version check will never fire — the feature silently becomes a no-op for that
  build. Always cut releases through `scripts/release.sh`.
