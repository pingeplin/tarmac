# 2606.0013 Dock pane stacking context over cards

**Date:** 2026-06-29
**Status:** draft
**Author:** EP Lin
**Issue:** [#35](https://github.com/.../issues/35) ‚Äî whiteboard dock pane is covered by board cards

## Context

The whiteboard dock pane (`.dock-pane`, `desktop/src/theme.css:954‚Äì967`, `z-index:65`) is a sibling of `.board` inside `.board-stack`. `.board` (`theme.css:93`, `position:relative; overflow:hidden`) does **not** establish its own stacking context. Card wrappers under `.board > .card-layer` carry `zIndex:c.z`, and `c.z` grows without bound: each grab sets the card to `topZ(cards)+1` (`onCardGrab`, `App.tsx:1550‚Äì1555`; `topZ` = `reduce(Math.max, 0)`, `board/model.ts:100`).

## Motivation

Because `.board` is not a stacking context, those per-card z-indices compete in the **same** stacking context as the sibling `.dock-pane`. After enough grabs a card's `z` exceeds 65 and paints over the dock ‚Äî the dock becomes partially or fully hidden. The model is correct (z must grow to keep focus-on-top working); the layout boundary is the bug.

## Proposed Solution

### Overview

Give `.board` its own stacking context so every descendant card z-index is contained *beneath* the sibling dock overlay, regardless of how high `c.z` climbs. Use `isolation: isolate` ‚Äî it triggers a stacking context with **no** z-index side effects (no `z-index` value to pick, no reflow, no paint change to existing cards).

### Key Components

| Path | Change |
|---|---|
| `desktop/src/theme.css` (`.board`, `:93`) | Add `isolation: isolate;` Single line. All `.card-layer` card z-indices now resolve within `.board`'s context; `.dock-pane` (sibling at `z-index:65`) always paints above the whole board. |
| `desktop/src/kit/zOrder.test.ts` | No change required. The unbounded-growth behavior of `topZ`/`onCardGrab` (the *reason* the stacking context is needed) is already covered here; this spec adds no vacuous test for the CSS fact. |

The real invariant ‚Äî "dock stays above any card regardless of z" ‚Äî is a CSS stacking-context fact, not pure logic, so it is verified by S1 (rule exists) and S2 (observable paint order), not by a unit test asserting a constant.

## Acceptance Scenarios

- **S1 ‚Äî stacking context declared.** *Given* `desktop/src/theme.css`, *when* the `.board` rule is read, *then* it carries `isolation: isolate` (specifically ‚Äî not `z-index` or another trigger, per the Definition of Done). Verifiable by grep.
- **S2 ‚Äî dock survives unbounded z.** *Given* a card grabbed enough times that its `z` far exceeds 65 (e.g. open the dock, then repeatedly grab one card), *when* the dock is shown, *then* the dock pane renders fully above that card with no card bleeding through. Manual QA via `make run`.
- **S3 ‚Äî z-order behavior unchanged.** *Given* the existing `zOrder.test.ts` suite, *when* `npm test` runs, *then* `topZ` cross-type folding and `onCardGrab`'s `topZ+1` bump (S9/S10) still pass ‚Äî the CSS change does not touch the model.

## For the Implementing Agent

> **Your job:** add `isolation: isolate` to `.board`; confirm S1 by grep, S3 by `cd desktop && npm test`, S2 by `make run` manual QA. Do **not** add a `z-index` to `.board` (would reintroduce a magic number competing with the dock) and do **not** invent a unit test for the paint-order fact ‚Äî it is not pure logic.

## Definition of Done

- [ ] `.board` carries `isolation: isolate` in `theme.css` (S1).
- [ ] `cd desktop && npm test` green; `npm run build` (tsc) clean (S3).
- [ ] Manual QA: grab a card many times with the dock open ‚Üí dock stays fully above it (S2).
- [ ] No `z-index` added to `.board`; no new test asserting the CSS constant.

## Trade-offs and Limitations

- `isolation: isolate` is preferred over `z-index: 0` purely to avoid introducing a competing stack-level; both establish a context, only one is side-effect-free.
- App-layer CSS is untested per repo convention; S2 is necessarily manual. The pure z-order model stays covered by `zOrder.test.ts`.
