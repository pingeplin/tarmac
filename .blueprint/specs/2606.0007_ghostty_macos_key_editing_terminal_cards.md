# 2606.0007 Ghostty macOS Key Editing in Terminal Cards

**Date:** 2026-06-21
**Status:** draft
**Author:** EP Lin

## Context

Tarmac terminal cards host a SwiftTerm `TerminalView`. Several standard macOS
editing keys that Ghostty supports don't behave like Ghostty in a card: **⌘⌫ /
⌘← / ⌘→** are swallowed (no-op), and **⌥↑ / ⌥↓** emit a meta-prefixed private-use
sequence instead of Ghostty's `ESC[1;3A` / `ESC[1;3B`. SwiftTerm's legacy keyDown
encoder has no mapping for these keys; Tarmac's only keyboard interceptor
(`escMonitor`) already passes them through untouched.

Tracking issue: [#21](https://github.com/pingeplin/tarmac/issues/21).

## Motivation

Muscle memory from Ghostty / Terminal.app / standard macOS text fields doesn't
carry into a terminal card today — friction for the primary use case (running a
CLI agent in a real terminal). Two robustness constraints are first-class, not
nice-to-have: many CJK users toggle **CapsLock** to switch IME ASCII/中文 mode, so
CapsLock-on is routine; and the shortcuts must never corrupt the line while a
**CJK IME** is composing.

## Proposed Solution

SwiftTerm's `TerminalView.keyDown(with:)` is `public`, not `open`, so it cannot be
overridden from a subclass in another module. Instead the five shortcuts are
injected in `AppController`'s existing global key monitor (`escMonitor`,
`AppController.swift:422`) — the one place every keyDown already passes, and which
already swallows the board shortcuts (⌘K/⌘T/⌘W/⌘C). For each keyDown, if a live
terminal holds focus (a small `focusedTerminalView()` first-responder walk, shared
with `reconcilePrimeToFocus`) and the pure decider maps the event to bytes, the
monitor `send()`s them to that terminal and swallows the event; otherwise it falls
through, leaving every existing path (kitty encoding, IME marked text, ⌥←/→ word
move, ⌥⌫, plain keys, and the board shortcuts) untouched.

All decision logic — physical-key match, modifier masking, exact-modifier match,
IME/kitty gates — lives in **`TermKeyBinding`** (new, `app/Sources/TarmacKit/`), a
pure AppKit-free `enum` taking the raw modifier bitfield as a `UInt`. It is the
only unit-tested unit; the escMonitor branch is a thin wire (read the focused
terminal's `hasMarkedText()` / kitty flags, call the decider, `send` or defer) and,
like all of `TarmacApp`, is not unit-tested (CLAUDE.md).

### Interface Contract

```swift
public enum TermKeyBinding {
    /// Bytes to emit to the PTY for a recognized Ghostty-parity shortcut, or nil
    /// to defer to SwiftTerm's own keyDown. `modifierFlags` is the raw
    /// NSEvent.ModifierFlags bitfield; the function masks it to the four intent
    /// modifiers internally (so CapsLock / function / numericPad never matter).
    static func bytes(keyCode: UInt16,
                      modifierFlags: UInt,
                      composing: Bool,
                      kittyActive: Bool) -> [UInt8]?
}
```

Recognized mappings (physical Carbon keyCodes; exact masked modifier shown):

| Keys | keyCode | exact modifier | bytes |
|---|---|---|---|
| ⌘⌫ | 51 | ⌘ only | `[0x15]` (Ctrl-U) |
| ⌘← | 123 | ⌘ only | `[0x01]` (Ctrl-A) |
| ⌘→ | 124 | ⌘ only | `[0x05]` (Ctrl-E) |
| ⌥↑ | 126 | ⌥ only | `[0x1b,0x5b,0x31,0x3b,0x33,0x41]` (`ESC[1;3A`) |
| ⌥↓ | 125 | ⌥ only | `[0x1b,0x5b,0x31,0x3b,0x33,0x42]` (`ESC[1;3B`) |

Returns `nil` for anything else, and **unconditionally** `nil` when
`composing == true` or `kittyActive == true`. "Exact modifier" means: after
masking to `[control, command, option, shift]` (which drops CapsLock / function /
numericPad), the set equals exactly the one shown — so any combo adding a fifth
intent modifier (e.g. ⌃⌘←, ⌥⌘←, ⇧⌘←, ⇧⌥↑) returns `nil`.

## Alternatives Considered

- **Subclass `TerminalView` and override `keyDown`** — the natural place (direct
  access to the view's `send()` / marked-text / kitty state), but SwiftTerm declares
  `keyDown` `public`, not `open`, so an out-of-module override won't compile.
  Infeasible without patching the dependency.
- **Patch vendored SwiftTerm** (mark `keyDown` `open`, or add the keys upstream) —
  pinned to an untagged commit for CJK/IME support (CLAUDE.md); edits would be lost
  on any bump. Rejected.

## Acceptance Scenarios

All scenarios are assertions on `TermKeyBinding.bytes(...)`. Modifier raw-bit
constants used below: command `0x100000`, control `0x40000`, option `0x80000`,
shift `0x20000`, capsLock `0x10000`, function `0x800000`, numericPad `0x200000`
(stable `NSEvent.ModifierFlags` raw values). **macOS always sets
function+numericPad on the four arrow keyCodes (123/124/125/126), never on ⌫
(51).** Every arrow scenario below — happy-path, gating, and no-regression —
therefore passes those two bits in the raw bitfield, so the tests exercise the
real masking the decider must perform. A test that omits function+numericPad on
an arrow would pass against a decider that forgot to mask them, i.e. be vacuous
for the exact behavior it guards; do not write such a test.

### Happy Path
- **S1:** Given keyCode 51 and modifiers = command (a real ⌘⌫ event; ⌫ is not an arrow, so no function/numericPad bits), when decided, then bytes == `[0x15]`.
- **S2:** Given keyCode 123 and modifiers = command+function+numericPad (a real ⌘← event — ← is an arrow, so the OS always sets function+numericPad), when decided, then bytes == `[0x01]`.
- **S3:** Given keyCode 124 and modifiers = command+function+numericPad (a real ⌘→ event — → is an arrow), when decided, then bytes == `[0x05]`.
- **S4:** Given keyCode 126 and modifiers = option+function+numericPad (a real ⌥↑ event), when decided, then bytes == `[0x1b,0x5b,0x31,0x3b,0x33,0x41]`.
- **S5:** Given keyCode 125 and modifiers = option+function+numericPad (a real ⌥↓ event), when decided, then bytes == `[0x1b,0x5b,0x31,0x3b,0x33,0x42]`.

### Edge Cases (CapsLock & exact-modifier)
- **S6:** Given **all five** recognized combos (S1–S5) re-run with the **capsLock** bit additionally OR'd into the raw modifiers (e.g. command+capsLock on keyCode 51; option+function+numericPad+capsLock on keyCode 126), when decided, then each returns bytes identical to its CapsLock-off result. CapsLock never changes the outcome for any of the five rows.
- **S7:** Given keyCode 123 (plus its always-present function+numericPad bits) with command **plus another intent modifier** (command+control, command+option, or command+shift), when decided, then result is `nil` — the ⌘ shortcuts fire only when ⌘ is the sole intent modifier.
- **S8:** Given keyCode 126 (plus function+numericPad) with option **plus** shift (⇧⌥↑), when decided, then result is `nil`.

### IME & Kitty Gates
- **S9:** Given a combo that returns bytes with the gates off (e.g. ⌘⌫ → `[0x15]`), when the *same* input is re-decided with `composing == true`, then result is `nil` — a composing CJK IME owns the keystroke; nothing is injected. (The paired bytes-when-off / nil-when-on assertion is what proves the gate is load-bearing, not vacuous.)
- **S10:** Given that same recognized combo, when re-decided with `kittyActive == true`, then result is `nil` — kitty-protocol apps (Claude Code, neovim) get their normal encoding via `super`. (Assert bytes-when-off and nil-when-on on identical inputs.)

### No-Regression (keys that must stay SwiftTerm's)
- **S11:** Given keyCode 123 or 124 with **option** as the sole intent modifier (a real ⌥←/⌥→ word-move event — raw bits option+function+numericPad), when decided, then result is `nil`.
- **S12:** Given keyCode 51 with **option** as the sole intent modifier (a real ⌥⌫ word-delete event — ⌫ is not an arrow, so raw bits = option only), when decided, then result is `nil`.
- **S13:** Given keyCode 51 with no intent modifiers (raw bits empty) and keyCodes 123, 124, 125, 126 with no intent modifiers (raw bits = function+numericPad, a real plain arrow event), when decided, then result is `nil` for each.
- **S14:** Given an unrecognized keyCode that is none of {51,123,124,125,126} — e.g. keyCode 0 ('a') with command, or keyCode 36 (Return) with command — when decided, then result is `nil`. (Guards the "anything else ⇒ nil" default so the decider can't accidentally match on modifiers alone.)

## For the Implementing Agent

> **Your job:** make every acceptance scenario above pass with tests that would
> *fail if the behavior were wrong*. A green suite that passes for the wrong
> reason does not satisfy this contract — `/verify` will hunt for vacuous tests.

- Put all decision logic in `TermKeyBinding` (TarmacKit) and unit-test it in
  `app/Tests/TarmacKitTests/TermKeyBindingTests.swift`. Mirror the style of
  `EscFocusAction` / `FocusedClose` (pure `enum`, single static func, focused
  tests with explanatory docstrings).
- Keep the escMonitor branch a thin wire (focused-terminal lookup → extract
  primitives → call decider → `send` or defer). It is not unit-tested by design;
  correctness is proven by `TermKeyBinding` tests + the fact that it acts only on a
  non-nil result.
- Recognition must be by **physical keyCode**, not by `event.characters`, so it is
  layout- and CapsLock-independent.
- Mirror the four `NSEvent.ModifierFlags` raw bits (command/control/option/shift)
  as named constants in `TermKeyBinding`, commented as stable Cocoa ABI values, so
  the masking stays AppKit-free.
- `make` (both `cargo test` and `swift test`) must pass before done.

Tests must be behavioral (assert returned bytes / `nil`), each able to fail, and
deterministic — see `references/test-desiderata.md` and `references/anti-patterns.md`.

## Definition of Done

- [ ] Test suite is green (`make test`).
- [ ] Every acceptance scenario (S1…S14) maps to at least one test.
- [ ] No covered-but-vacuous scenarios — each scenario's test fails under the smallest break of its behavior (e.g. wrong byte, wrong keyCode, dropped gate).
- [ ] Tests meet the Desiderata bar (Behavioral first); no AP-1…AP-8 violations.
- [ ] `make app` compiles the escMonitor branch + `focusedTerminalView()`; no implementation-quality blockers (stubs, dead code, stale docstrings).

## Trade-offs and Limitations

- The escMonitor wiring (focused-terminal lookup + `send()`) is not unit-tested
  (AppKit), consistent with the repo's testing boundary; correctness rests on the
  `TermKeyBinding` tests.
- ⌃←/→ word move and Shift+Enter are explicitly out of scope (see issue #21):
  ⌃←/→ is claimed by macOS Mission Control system-wide; Shift+Enter is deferred.
