# 2606.0001 Terminal-card lifecycle on shell exit

**Date:** 2026-06-15
**Status:** accepted
**Author:** EP Lin

## Context

Tarmac terminals are PTYs owned by the `tarmacd` daemon; the app renders one card per session. The original design goal was tmux-like: closing/reopening the app **detaches** and restores live sessions. Restore already works correctly — `Restore.liveTerms` lists the term_ids the daemon still owns a live PTY for, and `TermRestore.plan()` re-binds those and cold-spawns the rest (`app/Sources/TarmacKit/TermRestore.swift`).

The conflict is at the **other** end of the lifecycle: when a shell exits, `handleExit(termID:code:)` calls `card.setDead(code)` (`app/Sources/TarmacApp/AppController.swift:1431`, `CardView.swift:344`), which turns the card into a dimmed, un-closeable tombstone labeled `exit N · ↵ respawn`. The `↵ respawn` affordance was a Phase-5b placeholder that was never wired (Return is bound to dock toggle, not respawn). Worse, `persistLayout(for:)` snapshots every terminal card in `sessionOrder` including dead ones (`AppController.swift:2220`), so on relaunch the dead card's tile isn't in `liveTerms` and **cold-spawns a fresh shell** — the terminal resurrects as a fake new session.

So a deliberate `exit` today both lingers (un-closeable tombstone) *and* comes back (phantom respawn) — the opposite of the intended model on both axes.

## Motivation

A card should exist exactly as long as its session is meant to exist. **Detach** (app quit while the daemon keeps the PTY alive) preserves; an intentional **exit** ends it and the card goes away. Fixing this removes a confusing dead-but-immortal state, makes `exit` behave the way every terminal user expects, and makes "what restores on relaunch" mean exactly "what the daemon still holds a live PTY for."

## Proposed Solution

### Overview

Replace the single "dead tombstone" outcome with a decision driven by the exit code, and stop persisting exited terminals. Concentrate the branching in one **pure, table-testable** function in TarmacKit (mirroring `TermRestore.plan()`), so the AppKit layer (`AppController`/`CardView`, which has no unit tests) only does the wiring.

Two outcomes, by exit code:
- **Clean exit** (`code == 0`, e.g. `exit` / Ctrl-D): the card is **removed** from the board. If it was the last live terminal, a fresh boot terminal is spawned in its place; otherwise a brief **"shell closed · undo"** toast lets the user re-spawn a fresh shell in the same slot.
- **Error exit** (`code != 0`) or **killed by signal** (`code == nil`): the card is **held open** as a read-only placeholder that shows the failure (`exit N` / `killed`), so the error stays visible. It has **no close affordance this iteration** (see Open Questions) — it is session-local (not persisted) and clears on next launch.

An exited terminal — whether removed or held-open — is **never persisted**, so it does not reappear on relaunch.

### Key Components

- **`TermExit` (new, `app/Sources/TarmacKit/TermExit.swift`)** — pure decision logic, no AppKit. Maps `(code, otherLiveTerminals)` to an action, exposes the persistence-exclusion predicate, and provides the persisted-tile partition (`persistedTermIDs`) that `persistLayout` routes through. Unit-tested in `app/Tests/TarmacKitTests/TermExitTests.swift`.
- **`AppController.handleExit(termID:code:)`** (`:1427`) — integration site. Replaces the unconditional `setDead` with: compute the action via `TermExit.decide`, then remove / hold-open / remove-and-replace accordingly, wiring the undo toast and the last-terminal auto-spawn. `advancePrime` already moves prime off a gone terminal; `removeCard(id:.term(...))` already exists (`:1949`); the boot-spawn path already exists (`makeSession` + `spawn`, e.g. `:1092-1098`, `spawnNewTerminal` `:1386`).
- **`AppController.persistLayout(for:)`** (`:2220`) — exclude **exited** terminal cards from the tile snapshot by routing the terminal tiles through `TermExit.persistedTermIDs` (sourcing `exited` from each card's `dead` state). Must key off "exited", **not** `session.live` (see Interface Contract — detached sessions are `live == false`).
- **`CardView`** (`:344`) — a hold-open placeholder state: read-only, dimmed, header shows the exit status with the non-functional `↵ respawn` text **removed**. No close affordance this iteration (the card is session-local and clears on relaunch).
- **`ToastStackView.show(…, chips:)`** (`Toasts.swift:192`) — already supports a clickable action chip with 7 s auto-dismiss; the undo toast uses one `("undo", …)` chip. No toast-system change needed.

### Data Flow

1. Daemon child exits → daemon removes the term_id from its registry (`core/crates/tarmacd/src/term.rs:290`) and pushes `Exit { term_id, code }` (`:296`). **No daemon change.**
2. App `handleExit(termID:code:)` marks the session not-live, then calls `TermExit.decide(code:, otherLiveTerminals:)`.
3. `.remove` → remove the card; show the undo toast. `.removeAndReplace` → remove the card; immediately spawn a fresh boot terminal at the vanished card's world frame (no toast). `.holdOpen` → keep the card, switch it to the closeable placeholder state.
4. Undo → cold-spawn a fresh shell with a **new** term_id (`BootTerminal.mint()`, `BootTerminal.swift:10`) at the vanished card's world frame.
5. `persistLayout` runs on the active board; its terminal tiles pass through `TermExit.persistedTermIDs` (exited cards dropped), so the persisted tiles contain only live/detached terminals.

### Interface Contract

New pure module in TarmacKit (names are the contract; signatures may carry labels):

```swift
public enum TermExit {
    public enum Action: Equatable {
        case remove           // clean exit, other live terminals remain → vanish + offer undo
        case removeAndReplace // clean exit, this was the last live terminal → vanish + spawn a fresh boot terminal in its place
        case holdOpen         // error (non-zero) or signal (nil) exit → keep a read-only, closeable placeholder
    }

    /// `code`: nil = killed by signal, 0 = clean exit, non-zero = error.
    /// `otherLiveTerminals`: count of OTHER terminals on the same board still
    /// backed by a live pty (i.e. excluding the one that just exited).
    public static func decide(code: Int?, otherLiveTerminals: Int) -> Action

    /// Whether a terminal tile is written to the persisted layout. An `exited`
    /// shell (clean-removed, or held-open after an error) is excluded so it never
    /// reappears on relaunch. A DETACHED shell (connection dropped, pty may still
    /// be live) is NOT exited and IS persisted, so it re-binds on reconnect.
    public static func persistsTile(exited: Bool) -> Bool

    /// The persisted-tile partition: given each terminal tile's id paired with
    /// whether its shell has exited, returns the ids that survive into the
    /// persisted layout (in input order), i.e. `tiles.filter { persistsTile(exited:) }`.
    /// `persistLayout` MUST route through this rather than re-deriving the rule
    /// inline, so the `exited`-vs-`live` guard (S6/S9) is unit-tested here. `exited`
    /// is sourced from the card's `dead` state — NOT from `live` (a detached
    /// survivor is `live == false` but `exited == false`).
    public static func persistedTermIDs(_ tiles: [(termID: String, exited: Bool)]) -> [String]
}
```

Decision table `decide` must satisfy:

| `code` | `otherLiveTerminals` | result |
|---|---|---|
| `0` | `> 0` | `.remove` |
| `0` | `0` | `.removeAndReplace` |
| non-zero (e.g. `1`, `130`) | any (incl. `0`) | `.holdOpen` |
| `nil` (signal) | any (incl. `0`) | `.holdOpen` |

**Precedence (do not get this wrong):** the failure outcome wins over the last-terminal guarantee. When an exit is *both* an error/signal *and* the last live terminal (`code != 0 || code == nil`, `otherLiveTerminals == 0`), the result is `.holdOpen` — the card is held open and **no** boot terminal is auto-spawned at exit time. The last-terminal guarantee is re-applied only later, when the user *closes* the placeholder (see S5), not at the moment of the failing exit. An implementation that checks "last terminal?" before "error?" and emits `.removeAndReplace` here is wrong.

`persistsTile`: `exited == false → true`, `exited == true → false`.

**Load-bearing assumption (do not get this wrong):** in this codebase `session.live == false` does **not** mean "exited." On a daemon disconnect the app sets `s.live = false` *and* `s.detached = true` for still-alive shells (`AppController.swift:653-654`), to be re-bound on reconnect. The persistence exclusion and the `exited` input must therefore distinguish **exited** (shell is gone: clean-removed or held-open — corresponds to the card's `dead` state) from **detached** (`detached == true`, not dead). Keying exclusion off `!live` would silently drop reconnect-survivors from the persisted layout and break restore.

## Acceptance Scenarios

### Happy Path
- **S1:** Given a board with ≥2 live terminals, when one shell exits cleanly (code 0), then its card is removed from the board, the board's other terminals are untouched, and a "shell closed · undo" toast is shown.
- **S2:** Given the undo toast from S1, when the user chooses **undo**, then a fresh live terminal appears at the **same world frame** as the vanished card, carrying a **new** term_id (not the exited one).
- **S3:** Given a board whose **only** live terminal exits cleanly (code 0), when the exit is handled, then the card is removed and a fresh boot terminal is spawned in its place (board never ends up with zero live terminals); no undo toast is shown (the replacement already restores a working shell).

### Edge Cases
- **S4 — (Deferred):** Closing a hold-open placeholder is out of scope this iteration — terminal cards have no close affordance for now (see Open Questions). A placeholder is cleared only by app relaunch (it is `exited`, so it is not persisted).
- **S5 — (Deferred):** The close-driven branch of the last-terminal guarantee is deferred with S4. The guarantee still fires on clean-exit removal (S3); it does **not** fire for an error/signal exit (S8b).
- **S6 (regression — detach safety):** Given a set of terminals containing a **detached** one (`detached == true`, `live == false`, shell still alive — `exited == false`) and an **exited** one (`dead == true` — `exited == true`), when the persisted-tile partition is computed, then the detached terminal **is** retained and the exited one is dropped. The partition is driven by `TermExit.persistsTile(exited:)`, never by `live` — so a detached survivor (whose `live == false`) is NOT dropped. The pure partition is exercised by a unit test over this mixed set (a detached `!live` tile that survives and an exited `dead` tile that does not), so the `live`-vs-`exited` confusion is caught at the unit level, not only on manual relaunch.

### Error Scenarios
- **S7:** Given a shell that exits with a non-zero code (e.g. `exit 1`), when the exit is handled, then the card is **not** removed; it becomes a read-only placeholder whose header shows the failing code (e.g. "exit 1") with the old `↵ respawn` text gone. (No close affordance this iteration — see S4 / Open Questions.)
- **S8:** Given a shell killed by a signal (Exit `code == nil`), when the exit is handled, then the card becomes a read-only placeholder labeled to indicate it was killed (e.g. "killed"), not removed.
- **S8b (failure wins over last-terminal):** Given a board whose **only** live terminal exits with a non-zero code (or is killed by a signal — `otherLiveTerminals == 0` and `code != 0`/`code == nil`), when the exit is handled, then the card is held open as a closeable placeholder and **no** boot terminal is auto-spawned at this point (the live-terminal count after handling is 0); the last-terminal guarantee fires only when the user later closes the placeholder (S5). This pins that `.holdOpen` takes precedence over `.removeAndReplace` when an exit is simultaneously the last terminal and a failure.
- **S9 (persistence / no resurrection):** Given a clean-exited terminal (removed) and a held-open placeholder (both `exited`) alongside at least one live terminal, when the persisted-tile partition is computed, then neither exited term_id appears in the persisted tiles while the live one does — so after an app restart neither exited terminal reappears. Like S6, this is asserted on the output of the pure partition (the retained term_id set), not on the daemon round-trip.
- **S10 (pure decision table):** `TermExit.decide` returns exactly the Interface-Contract table across `code ∈ {0, 1, 130, nil}` × `otherLiveTerminals ∈ {0, 2}` (8 cells). The expected results are: `(0,2) → .remove`; `(0,0) → .removeAndReplace`; `(1,2)`, `(1,0)`, `(130,2)`, `(130,0)`, `(nil,2)`, `(nil,0)` → `.holdOpen`. `130` (a signal-induced non-zero code, SIGINT = 128+2) is included so the "any non-zero → holdOpen" class is non-vacuous against a mutation that special-cases high or negative codes. `TermExit.persistsTile` returns `!exited` for both `exited ∈ {true, false}`.
- **S11 (no regression in restore):** `TermRestore.plan()` still re-binds tiles whose id is in `liveTerms` and cold-spawns the rest — existing `TermRestoreTests` remain green and unchanged.

Write tests to the project's conventions and to these principles:

## Alternatives Considered

### Keep the tombstone, add a working respawn button
The original Phase-5b intent (revive-in-place). Rejected in the prior design discussion: it preserves the un-closeable dead state the user explicitly wants gone, and "revive" can't restore the dead session anyway (the daemon already dropped the PTY) — it can only cold-spawn, which is what undo/replacement already do with a clean frame.

### Vanish on every exit (no hold-open)
Simplest. Rejected (locked product decision): a shell dying with a non-zero code or a signal would disappear before the user can read why it failed. Error exits hold open so the failure stays visible (mirrors iTerm2/kitty "hold on failure").

### Exclude from persistence by `!session.live`
Rejected as incorrect: detached-but-alive sessions are `live == false` (`AppController.swift:653`), so this would drop reconnect-survivors and break restore. Exclusion must key off "exited" (the card's `dead`/terminal state).

## Trade-offs and Limitations

- A clean exit is irreversible once the undo toast auto-dismisses (~7 s); the dead session's scrollback is not recoverable (only a fresh shell is). This is intended.
- The last-terminal guarantee means a board can never be emptied of terminals by a clean exit — consistent with the chosen "auto-spawn a fresh one" product decision, but it does prevent a deliberately terminal-free (docs-only) board via this path.
- Hold-open placeholders are session-local (not persisted); a held-open error card seen before an app quit is simply gone on relaunch (the shell is dead and nothing to restore).
- This iteration ships **no close affordance** for terminal cards. An error/signal exit of the *last* terminal therefore leaves a read-only placeholder with zero live terminals on the board until the user spawns a new terminal (⌘T) or relaunches; the placeholder cannot be manually dismissed yet (deferred — see Open Questions).
