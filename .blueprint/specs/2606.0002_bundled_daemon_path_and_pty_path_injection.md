# 2606.0002 Bundled Daemon Path Resolution & PTY PATH Injection

**Date:** 2026-06-16
**Status:** draft
**Author:** EP Lin
**Design doc:** [2606.0002](../../docs/designs/2606.0002_macos_distribution_homebrew_tap_and_notarized_dmg.md)

## Context

To ship Tarmac as a double-clickable `.app` (see the design doc), the app must stop depending on the two environment facts that `make run` injects today:

```make
TARMAC_DAEMON="$(ROOT)/core/target/debug/tarmacd" \
PATH="$(ROOT)/core/target/debug:$$PATH" \
"$(ROOT)/app/.build/debug/TarmacApp"
```

Two code paths assume those facts and break without them:

- `DaemonClient.connect()` (`app/Sources/TarmacKit/DaemonClient.swift:56`) **throws** `connectFailed` when `TARMAC_DAEMON` is unset or empty. A Finder-launched bundle, which has no such env var, can never start its daemon.
- `DaemonClient.spawnDaemon()` (`:200`) runs `Process()` with **no** `environment` override, so the daemon — and every PTY it spawns — inherits the app process's `PATH`. A Finder-launched `.app` gets the minimal launchd `PATH` (`/usr/bin:/bin:/usr/sbin:/sbin`), which contains no `tarmac`. So `tarmac open` fails inside Tarmac's own terminals even after a Homebrew cask has symlinked `tarmac` into `/opt/homebrew/bin`.

## Motivation

These are the **only** two behavior changes the whole distribution effort needs from the app; everything else (bundling, signing, notarization, the cask) is packaging scripts verified by installing a real build, explicitly out of this spec gate per the design. Both changes are pure decisions that the project convention says belong in `TarmacKit` and get unit-tested there, leaving `DaemonClient` as a thin impure shell (filesystem check + `proc.environment` assignment) that is GUI/integration-verified. Specifying them as pure functions is what makes them testable at all — the IO around them is not.

## Proposed Solution

### Overview

Add one new `TarmacKit` namespace, `DaemonLaunch`, holding two pure, side-effect-free functions. `DaemonClient` calls them and performs the I/O (the filesystem existence check, and setting `proc.environment`). No behavior of the pure functions depends on disk, the clock, or process state, so the existing `make run` path (explicit `TARMAC_DAEMON` + `PATH`) is preserved by construction.

### Key Components

- **`DaemonLaunch.resolveDaemonPath(env:bundleURL:bundledBinaryExists:)`** — decides which daemon binary to spawn. Lives in new file `app/Sources/TarmacKit/DaemonLaunch.swift`. The caller (`DaemonClient.connect`) computes `bundledBinaryExists` via `FileManager` and passes it in; the function itself touches no disk.
- **`DaemonLaunch.injectCLIPath(base:cliDir:)`** — computes the `PATH` string handed to the spawned daemon so it and its PTYs can resolve `tarmac`. Called from `DaemonClient.spawnDaemon` to build `proc.environment["PATH"]`.
- **`DaemonClient` wiring (impure shell, not unit-tested)** — `connect()` replaces the hard `guard env[TARMAC_DAEMON] else throw` with a call to `resolveDaemonPath`, falling back to today's `connectFailed` only when the result is `nil`; `spawnDaemon()` sets `proc.environment` from the current environment with `PATH` replaced by `injectCLIPath(...)`.

### Interface Contract

```swift
public enum DaemonLaunch {
    /// Which daemon binary to spawn, or nil if none is resolvable.
    /// Pure: the FILESYSTEM existence check is the caller's job and is passed in
    /// as `bundledBinaryExists`, so this is unit-testable away from disk.
    ///
    /// - If env["TARMAC_DAEMON"] is non-empty, return it VERBATIM (the explicit
    ///   override wins unconditionally — `bundledBinaryExists` is ignored). This
    ///   preserves `make run`, which points it at a debug build.
    /// - Else, derive `<bundleURL>/Contents/MacOS/tarmacd`; return it iff
    ///   `bundledBinaryExists`, else nil.
    /// "Non-empty" means `!value.isEmpty` (no trimming) — matching today's
    ///   `!daemonBin.isEmpty` check; a whitespace-only value counts as set.
    static func resolveDaemonPath(
        env: [String: String],
        bundleURL: URL,
        bundledBinaryExists: Bool
    ) -> String?

    /// PATH to hand the spawned daemon so it (and its PTYs) can resolve `tarmac`.
    /// - Prepend `cliDir` as the FIRST segment of `base`.
    /// - If `cliDir` already appears as an EXACT colon-delimited segment anywhere
    ///   in `base`, return `base` UNCHANGED (no duplicate, no reorder) → idempotent.
    /// - If `base` is nil or empty, return `cliDir`.
    /// - Substrings are not segments: a `base` segment of "/x/binfoo" does NOT
    ///   count as containing cliDir "/x/bin".
    /// - Existing empty segments (leading/trailing colons) are preserved verbatim.
    /// - Defensive: if `cliDir` is empty, return `base` unchanged (nil → "").
    static func injectCLIPath(base: String?, cliDir: String) -> String
}
```

## Acceptance Scenarios

### Happy Path
- **S1:** Given `env["TARMAC_DAEMON"] = "/dbg/tarmacd"` (non-empty) and `bundledBinaryExists = false`, when `resolveDaemonPath` runs, then it returns `"/dbg/tarmacd"` verbatim — the `make run` override is preserved even with no bundled binary.
- **S2:** Given no `TARMAC_DAEMON` key, `bundleURL = file:///A/Tarmac.app`, and `bundledBinaryExists = true`, when `resolveDaemonPath` runs, then it returns exactly `"/A/Tarmac.app/Contents/MacOS/tarmacd"`.
- **S3:** Given `base = "/usr/bin:/bin"` and `cliDir = "/opt/homebrew/bin"`, when `injectCLIPath` runs, then it returns `"/opt/homebrew/bin:/usr/bin:/bin"`.

### Edge Cases
- **S4:** Given `env["TARMAC_DAEMON"] = ""` (present but empty) and `bundledBinaryExists = true`, when `resolveDaemonPath` runs, then it returns the bundled path (empty is treated as unset, not as a verbatim override).
- **S5:** Given `env["TARMAC_DAEMON"] = "/dbg/tarmacd"` **and** `bundledBinaryExists = true`, when `resolveDaemonPath` runs, then it returns `"/dbg/tarmacd"` (the explicit override wins over the bundled path; precedence is pinned).
- **S13:** Given `env["TARMAC_DAEMON"] = "   "` (whitespace only, non-empty) and `bundledBinaryExists = true`, when `resolveDaemonPath` runs, then it returns `"   "` verbatim — non-empty means `!isEmpty`, with no trimming, so a whitespace-only value counts as set and wins over the bundled path (matches today's `!daemonBin.isEmpty`).
- **S6:** Given `base = "/usr/bin:/opt/homebrew/bin"` and `cliDir = "/opt/homebrew/bin"` (already present, not first), when `injectCLIPath` runs, then it returns `base` unchanged — no duplicate, no reorder.
- **S7:** Given `base = nil` (and separately `base = ""`) with `cliDir = "/opt/homebrew/bin"`, when `injectCLIPath` runs, then it returns `"/opt/homebrew/bin"`.
- **S8:** Given `base = "/usr/bin:"` (trailing colon → empty last segment) and `cliDir = "/x"`, when `injectCLIPath` runs, then it returns `"/x:/usr/bin:"` — the existing empty segment is preserved, only a prefix is added.
- **S9:** Given `base = "/opt/homebrew/binfoo"` and `cliDir = "/opt/homebrew/bin"` (substring, not a full segment), when `injectCLIPath` runs, then it returns `"/opt/homebrew/bin:/opt/homebrew/binfoo"` — substring is not a match.
- **S12:** (Empty `cliDir`, the defensive clause) Given `cliDir = ""`, when `injectCLIPath` runs, then it returns `base` unchanged for `base = "/usr/bin:/bin"` (→ `"/usr/bin:/bin"`) and returns `""` for `base = nil` — an empty `cliDir` never gets prepended (no leading colon is introduced).
- **S10:** (Idempotency property) Given any `base` and `cliDir`, when `injectCLIPath` is applied to its own output, then the result equals the single-application output — `injectCLIPath(base: injectCLIPath(base: base, cliDir: cliDir), cliDir: cliDir) == injectCLIPath(base: base, cliDir: cliDir)`. Cover this over a table that includes the prepend case (S3), the already-present case (S6), the empty-base case (S7), and the substring case (S9), so idempotency is asserted across every branch, not just the trivial one.

### Error Scenarios
- **S11:** Given no `TARMAC_DAEMON` key (or an empty one) and `bundledBinaryExists = false`, when `resolveDaemonPath` runs, then it returns `nil` — and the design contract is that the caller (`DaemonClient.connect`) then throws today's unchanged helpful `connectFailed` ("is tarmacd running? … set TARMAC_DAEMON to auto-spawn it").

## For the Implementing Agent

> **Your job:** make every acceptance scenario above pass with tests that would *fail if the behavior were wrong*. `/verify` will hunt for vacuous tests by asking, of each behavior, "what is the smallest change that breaks this, and would any test catch it?"

Build to the existing `TarmacKit` conventions — mirror `TermExit` / `TermRestore`: a `public enum` namespace of `public static func`s with doc comments that explain *why*, and `XCTest` table-driven tests in `app/Tests/TarmacKitTests/DaemonLaunchTests.swift` whose comments cite the S-numbers (see `TermExitTests.swift` for the house style). Keep both functions pure — no `FileManager`, no `ProcessInfo`, no `Process`. Then wire them into `DaemonClient.connect()` and `DaemonClient.spawnDaemon()`; that wiring is the thin impure shell and is **not** unit-tested (the app/GUI layer is GUI-verified by project convention), but it MUST route through these functions rather than re-deriving the logic inline, and it MUST preserve the existing `connectFailed` message when `resolveDaemonPath` returns `nil`.

Test-quality principles `/verify` scores against:
- **Behavioral over structural** — assert returned strings/paths, not internals.
- **Every test can fail** — no tautologies; S5 and S9 in particular must distinguish the right answer from the plausible wrong one (override-vs-bundled precedence; segment-vs-substring).
- **Deterministic, isolated, readable** — these are pure functions; no clock, no disk, AAA with inline cases.

## Definition of Done

Done is when `/verify` passes against this spec:

- [ ] `swift test` is green — every pre-existing TarmacKit test still passes (no regressions) and the new `DaemonLaunchTests` are added to the suite.
- [ ] Every scenario S1…S13 maps to at least one test.
- [ ] No covered-but-vacuous scenarios — each fails under the smallest break of its behavior (e.g. flipping override/bundled precedence breaks S5; `contains` instead of segment-match breaks S9; dropping the dedup breaks S6/S10).
- [ ] Tests meet the Desiderata bar; no AP-1…AP-8 violations.
- [ ] `DaemonClient` routes through `DaemonLaunch` (no inline re-derivation) and preserves the existing `connectFailed` message on the `nil` path; `make run` still launches (explicit `TARMAC_DAEMON` honored verbatim).

## Trade-offs and Limitations

- **The impure shell is unverified by this gate.** Whether `DaemonClient` actually sets `proc.environment` and computes `bundledBinaryExists` correctly is checked by running the bundled app on a real machine, not by `swift test`. This is the deliberate seam: pure decision here, I/O there.
- **`resolveDaemonPath` does not existence-check the explicit override.** An explicit `TARMAC_DAEMON` pointing at a missing file is returned verbatim and fails later at `proc.run()` (today's behavior, preserved) rather than falling through to the bundled path. Honoring the override unconditionally is intentional — a developer who sets it means it.
- **Dedup leaves an already-present `cliDir` in place rather than promoting it to the front.** If some earlier `PATH` segment also contains a `tarmac`, it would shadow the bundled one. Accepted: not reordering a user/admin-configured `PATH` is safer than guaranteeing our `tarmac` wins, and the bundled-launch case starts from the minimal launchd `PATH` where the conflict can't arise.
