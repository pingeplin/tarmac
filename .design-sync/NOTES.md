# design-sync notes

## Known accepted substitutions

- **SF Pro Text → system sans-serif fallback.** `desktop/src/theme/card.css` has a body-text font stack `-apple-system, "SF Pro Text", system-ui, sans-serif` (prose/body text, not the chrome font — chrome is IBM Plex Mono and ships as a real vendored `.woff2`). Apple's SF Pro can't legally ship as a redistributable web font, so the DS pane renders this text in a system sans-serif substitute instead. User-approved 2026-07-01. `[FONT_MISSING]` for `"SF Pro Text"` is expected and should not be chased on future syncs.

## Preview authoring (calibration pass: DocCard, ZoomControl, BoardSwitcher)

- **Run preview-rebuild.mjs / package-capture.mjs from `desktop/`, with `desktop/.design-sync` a symlink to `../.design-sync`.** Both scripts hardcode `resolve('.design-sync', 'previews')` / `resolve('.design-sync', '.cache', 'review')` relative to **cwd**, with no config-home indirection — and `cwd` must be `desktop/` anyway, because `componentSrcMap` paths (e.g. `src/cards/DocCard.tsx`) are resolved against `PKG_DIR`, which for this repo's `cfg.entry: "./src/kit-entry.ts"` resolves to `desktop/` itself. But the durable, committed previews/config live at the **repo root** `.design-sync/` per this project's convention. The fix: `desktop/.design-sync` is a **symlink** to `../.design-sync` (added to `desktop/.gitignore` as `.design-sync`, since it's a per-clone convenience, not tracked content). **Recreate it on a fresh clone**: `cd desktop && ln -s ../.design-sync .design-sync`. Without it, `preview-rebuild.mjs` reports `[ZERO_MATCH] nothing to rebuild` even though the previews exist at the repo-root path the skill docs point at.
- **Import shape confirmed**: `import { DocCard } from "tarmac-app";` (named imports, matching `cfg.pkg`) — resolves to `window.TarmacKit.DocCard` via the story-imports shim. Each named export in a preview `.tsx` must be a **zero-arg function component** (`export function Selected() { return <DocCard .../> }`), not a bare JSX element or a component invoked with props at module scope — the harness calls `h(window.__dsPreview[key])` with no arguments.
- **Every one of these 10 kit components is board-chrome, not a self-contained widget** — CardShell/DocCard render `position: absolute; inset: 0` (CardShell's `inWrapper` mode), ZoomControl is `position: absolute; left/bottom: 12px`, BoardSwitcher's veil+panel are both `position: absolute` (`inset: 0` and `top/left: 50% + translate(-50%,-50%)`). In the real app these all live inside `.board`/`.app`, which app-only.css sizes via flex/`position:relative` — but **app-only.css is deliberately excluded from the kit CSS bundle** (`kit.css`'s own header comment), so `.board`/`.app` carry ZERO styling in a kit preview. Worse, design-sync's single-story capture wraps every story in a bare `.ds-single` div (`transform: translateZ(0)`, no explicit size) — a real containing block for position:absolute/fixed, but with **zero intrinsic height** since these components have no normal-flow content of their own. The result: `inset:0` collapses to 0 height (DocCard rendered as a 1px sliver), `bottom:12px` resolves against a 0-height box and renders almost entirely above the viewport (ZoomControl), and `top:50%+translate(-50%,-50%)` centers against a 0-height box so ~half the panel clips above the fold (BoardSwitcher). **Fix**: wrap every story's returned JSX in a plain `<div style={{ position: "relative", width: <plausible-px>, height: <plausible-px> }}>` (add `className="board"` too for DOM-shape parity with production, even though the class itself does nothing) — this gives the component a real, sized containing block. Sizes used here: DocCard `360×520` (matches its `frame.w/h`), ZoomControl `220×90` (just enough room around the pinned corner), BoardSwitcher `820×600` (stand-in for `.board-stack`). **Expect the same fix to be needed for all 7 remaining components** (CardShell, CycleHud, DockPane, MinimapOverlay, OffscreenHints, StatusBar, ToastOverlay) — check each one's root CSS `position` before assuming a bare `<Component .../>` will render correctly; if the sheet shows a collapsed sliver or content clipped near a viewport edge, this is almost certainly why.
- **`kit-fixtures.mjs`'s `wrapInBoard: true` signal is necessary but NOT sufficient** — it gets you DOM-shape parity (`.closest(".board")` lookups, e.g. DocCard's zoom-change MutationObserver) but not a working containing block; you still need the explicit inline size above. Only `DocCard` had `wrapInBoard: true` in the fixture; treat `wrapInBoard` as "also give this one the `.board` class" and treat the sizing fix as required for every component regardless of that flag.
- **`kit-fixtures.mjs` props matched current prop shapes exactly** for all three calibration components (DocCard, ZoomControl, BoardSwitcher) — no drift found. The fixture's `BoardSwitcher` rows and `DocCard` markdown were reused/extended as-is.
- **Emitted `.d.ts` prop bodies are all `{ [key: string]: unknown }`** (no real prop extraction) — this repo has no `.d.ts` tree (package shape reads `src/` directly via `componentSrcMap`), and ts-morph's extraction from raw `.tsx` source apparently didn't populate per-component prop bodies. Harmless for authoring (previews compile fine against the real component signatures via plain esbuild/JSX, no type-checking gate in this pipeline), but worth fixing later with `cfg.dtsPropsFor` if the generated `.prompt.md`/`.d.ts` quality matters for the design agent.

## Preview authoring (fan-out waves: CardShell, DockPane, CycleHud, StatusBar, ToastOverlay, MinimapOverlay, OffscreenHints)

All 7 graded `good` on first or second pass. Two additions to the calibration-pass guidance above:

- **Capture viewport ceiling: ~900×700px, fixed.** The `.board`-style sized wrapper (learning above) must ALSO fit inside the capture tool's fixed raw-screenshot viewport (confirmed via `sips` on `_screenshots/review/raw/*.png`) — the sheet HTML's cosmetic `max-width/max-height` `<img>` cap is not the real limit. A wrapper larger than ~900×700 silently clips corner/edge-pinned chrome (discovered on MinimapOverlay/OffscreenHints at 960×640/1100×640). Keep wrapper dimensions ≤ ~860×600, and for components with literal pixel-coordinate children (e.g. OffscreenHints' pill `left`/`top`, unlike percentage/inset-based layouts) re-derive coordinates against the new wrapper's actual edges rather than reusing `kit-fixtures.mjs`'s raw numbers (which assume a full app-window-sized board).
- **Not every board-chrome component needs the sizing wrapper fix.** Components that are normal flow children with an explicit height (e.g. `StatusBar`, `flex: 0 0 27px`) rather than `position: absolute/fixed` don't hit the zero-height-containing-block gotcha — check the root rule's `position` before assuming the fix is needed (applying the `.board` wrapper anyway for DOM-shape convention is harmless).
- `kit-fixtures.mjs` prop shapes matched live `.tsx` sources exactly across all 10 components now authored — zero drift found in this sync.

**Latent app CSS gap found while authoring (not a design-sync issue, flagging for a follow-up):** `.dock-header .hint` in `desktop/src/theme/chrome.css` lacks `white-space: nowrap` (unlike its sibling `.dock-header .label`) — an unrealistically long single-token label can starve its flex-shrink share and wrap "esc ↩" onto a second line, overflowing the fixed 34px header height. Only reachable with unrealistic label lengths (not shipped as a preview story), but real. Consider adding `white-space: nowrap` (or `flex-shrink: 0`) to `.dock-header .hint` in a follow-up PR.

## Re-sync risks

- The kit's CSS entry (`cfg.cssEntry: "dist-kit/tarmac-kit.css"`) is our OWN pre-built, esbuild-flattened stylesheet from the issue-#47 build-kit pipeline, not raw source — `cfg.buildCmd: "npm run build:kit"` must run before the converter on every sync or the CSS is stale. The driver's re-sync flow does this automatically when told to rebuild; a manual `package-build.mjs` invocation without first running `build:kit` will silently reuse a stale `dist-kit/tarmac-kit.css`.
- `componentSrcMap` pins are load-bearing, not just a nicety: `desktop/` has no `.d.ts` tree, so the converter's component list comes ENTIRELY from `componentSrcMap`'s 10 explicit entries. Adding a component to `kit-entry.ts` without adding it to `componentSrcMap` means it silently never syncs.
- `tokensGlob` (`src/theme/tokens.css`) is currently a no-op (`copyTokens` only fires for a sibling `tokensPkg`, which doesn't exist here) — tokens ship only via the flattened `cssEntry`. Harmless today; if a future `tokens/` browsing feature depends on `copyTokens` output, this will need revisiting.
