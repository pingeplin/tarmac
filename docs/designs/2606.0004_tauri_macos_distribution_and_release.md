# 2606.0004 macOS Distribution: Tauri Build/Release Pipeline

**Date:** 2026-06-29
**Status:** accepted
**Author:** EP Lin
**Doc type:** RFC
**Supersedes:** 2606.0002 (Swift/AppKit era — retained as history)

## TL;DR

The GUI layer migrated from Swift/AppKit (`app/`) to Tauri 2 + React + xterm.js (`desktop/`). The three-programs-one-product invariant — `Tarmac.app` GUI + `tarmacd` daemon + `tarmac` CLI travel and install together — is unchanged, as are the notarized `.dmg` artifact, the self-hosted Homebrew tap, and the bespoke inside-out signing pipeline. What changed is how the `.app` bundle is assembled: `tauri build` now generates `Info.plist` and embeds icons from `tauri.conf.json`, making the hand-crafted `packaging/Info.plist` and `packaging/Tarmac.icns` obsolete (deleted). The two Rust binaries ride as Tauri sidecars. The `packaging/` tree now holds only `Tarmac.entitlements` and `Casks/tarmac.rb`.

## What changed since 2606.0002

### Bundle assembly

2606.0002 assumed a hand-assembled `Tarmac.app` skeleton (manual `Info.plist`, manual icon copy, SwiftPM resource bundle). That is gone.

`scripts/bundle.sh` now:
1. `cargo build --release` the Rust workspace.
2. Stage `tarmacd` and `tarmac` into `desktop/src-tauri/binaries/` with the Tauri-required target-triple suffix (`tarmacd-aarch64-apple-darwin`, `tarmac-aarch64-apple-darwin`).
3. `npx tauri build --bundles app --target aarch64-apple-darwin` — Tauri compiles the frontend, links the Rust binary (`tarmac-app` crate), generates `Contents/Info.plist` from `tauri.conf.json`, copies icons from `desktop/src-tauri/icons/`, and places the two sidecars into `Contents/MacOS/`.
4. Copy the resulting bundle from `desktop/src-tauri/target/aarch64-apple-darwin/release/bundle/macos/Tarmac.app` → `dist/Tarmac.app`.
5. Assert both `tarmacd` and `tarmac` are present in `Contents/MacOS/`.

The bundle's `externalBin` in `tauri.conf.json` declares the two sidecars:
```json
"externalBin": ["binaries/tarmacd", "binaries/tarmac"]
```
Tauri strips the target-triple suffix when embedding; they land in `Contents/MacOS/` as `tarmacd` and `tarmac`.

### Daemon resolution

The Swift-era approach injected `TARMAC_DAEMON` at launch (Makefile env wiring). The Tauri backend resolves `tarmacd` as a sibling of `current_exe()` in `Contents/MacOS/` (`bridge.rs: resolve_daemon_path_pure`). In a packaged bundle, `current_exe()` → `Contents/MacOS/tarmac-app` (the GUI exe is named after the `tarmac-app` crate; it is deliberately **not** named `Tarmac`, which would collide case-insensitively with the `tarmac` CLI sidecar in the same dir); the parent dir is `Contents/MacOS/`; resolution joins `tarmacd` there — it keys off the parent dir, not the exe name. A non-empty `TARMAC_DAEMON` env override still wins (so `make run` dev wiring is unchanged). The sibling-of-exe strategy is pure and unit-tested in `bridge.rs`.

### Versioning

2606.0002 assumed `PlistBuddy -c "Set :CFBundleShortVersionString …"` to stamp the Info.plist at bundle time. Tauri now generates Info.plist from `tauri.conf.json:version`. The authoritative embedded bundle version is the `version` field in `desktop/src-tauri/tauri.conf.json`. `scripts/release.sh` reads it and emits a warning if the `VERSION` env disagrees:
```sh
TAURI_VERSION=$(grep '"version"' "$ROOT/desktop/src-tauri/tauri.conf.json" | head -1 | grep -oE '[0-9]+\.[0-9]+\.[0-9]+')
[ "$VERSION" = "$TAURI_VERSION" ] || echo "WARNING: …" >&2
```
A release bump touches `desktop/src-tauri/tauri.conf.json:version` and `packaging/Casks/tarmac.rb:version`/`sha256`. The Rust crate version in `core/Cargo.toml` is left at its own cadence.

### Entitlements (new dependency)

Swift/SwiftTerm required no special entitlements beyond hardened runtime. The Tauri WebView (`WKWebView`-backed) requires two:
- `com.apple.security.cs.allow-jit` — for V8/WebKit JIT.
- `com.apple.security.cs.allow-unsigned-executable-memory` — Tauri runtime requirement.

`packaging/Tarmac.entitlements` carries exactly these two. The `.app` is still signed with `-o runtime --timestamp` and this entitlements file; no `--deep`.

### Deleted packaging artifacts

`packaging/Info.plist` and `packaging/Tarmac.icns` are deleted — Tauri generates both from `tauri.conf.json` and `desktop/src-tauri/icons/`. `packaging/` now contains only `Tarmac.entitlements`, `Casks/tarmac.rb`, and `icon/` (source artwork).

## Current pipeline

### Bundle (`scripts/bundle.sh`)

```
cargo build --release (arm64)
↓
stage sidecars → desktop/src-tauri/binaries/tarmacd-aarch64-apple-darwin
                                              tarmac-aarch64-apple-darwin
↓
npx tauri build --bundles app --target aarch64-apple-darwin
  (emits to desktop/src-tauri/target/aarch64-apple-darwin/release/bundle/macos/)
↓
cp -R → dist/Tarmac.app
assert Contents/MacOS/{tarmacd,tarmac} present
```

### Sign → DMG → Notarize (`scripts/release.sh`)

```
bundle.sh (unsigned)
↓
codesign inside-out, no --deep:
  Contents/MacOS/tarmacd   (runtime + timestamp, no entitlements)
  Contents/MacOS/tarmac    (runtime + timestamp, no entitlements)
  dist/Tarmac.app          (runtime + timestamp + Tarmac.entitlements)
codesign --verify --strict
↓
stage DMG: Tarmac.app + standalone tarmac at root + /Applications symlink
codesign standalone tarmac copy (runtime + timestamp)
hdiutil create → dist/Tarmac-$VERSION.dmg
↓
xcrun notarytool submit --wait
xcrun stapler staple dist/Tarmac-$VERSION.dmg
print sha256
↓
bump packaging/Casks/tarmac.rb version + sha256
gh release create → upload .dmg
PR → pingeplin/homebrew-tarmac
```

### Tap and cask (unchanged structure)

Self-hosted tap `pingeplin/homebrew-tarmac` (`github.com/pingeplin/homebrew-tarmac`):

```
brew install pingeplin/tarmac/tarmac
```

Cask delivers both:
- `app "Tarmac.app"` → `/Applications/`
- `binary "tarmac"` → symlinked onto PATH (standalone copy at `.dmg` root, not an in-bundle path)

## Constraints and trade-offs

**Why keep bespoke signing instead of Tauri's bundler signing?** Tauri's native signing/notarization support (`tauri.conf.json bundle.macOS`) would handle the `.app` but not the standalone `tarmac` CLI at the DMG root that the cask's `binary` stanza links. The bespoke pipeline signs all three Mach-Os explicitly and controls the DMG layout that the cask contract depends on. Switching to Tauri-native would require re-validating the cask layout and the stapling behavior; the proven pipeline has cleared notarization successfully, so the cost of migration exceeds the benefit.

**Why `--bundles app` not `--bundles dmg`?** Tauri's own DMG output has a different layout than the cask-shaped DMG (`scripts/release.sh` stage: app + standalone CLI + `/Applications` symlink). Producing our own DMG from the bare `.app` is one `hdiutil create` call and keeps full layout control.

**arm64-only.** No Intel/universal ship yet (no x86_64 toolchain in the release env). `packaging/Casks/tarmac.rb` guards with `depends_on arch: :arm64`; Intel users get a clear error rather than a broken install.

## Carried forward unchanged from 2606.0002

- Three-programs-one-product invariant (`Tarmac.app` + `tarmacd` + `tarmac` travel together).
- Self-hosted tap `pingeplin/homebrew-tarmac`; cask's `app` + `binary` dual stanza.
- Notarized `.dmg` on GitHub Releases as the direct-download artifact.
- Inside-out signing (embedded Mach-Os first, `.app` last, no `--deep`).
- Notarize via `notarytool submit --wait`, staple `.dmg` (not `.zip`).
- `DEVID_IDENTITY` and `NOTARY_PROFILE` env vars hard-asserted in `release.sh`.
- Stapled `.dmg` as Gatekeeper translocation mitigation.
- No Sparkle auto-update (deferred).
- No `homebrew/homebrew-cask` submission (notability bar).
- `brew upgrade` via version + sha256 bump as the update mechanism.

Operational click-by-click lives in the release runbook (project memory: `macos-release-runbook.md`), not here.
