# 2606.0003 Per-Channel Daemon Socket and State Namespacing

**Date:** 2026-06-16
**Status:** draft
**Author:** EP Lin
**Doc type:** RFC

## TL;DR

A `make run` dev build and the installed, released `Tarmac.app` currently **collide**: all three binaries — the Swift app, the Rust `tarmacd`, the Rust `tarmac` CLI — independently default to the *same* Unix socket `~/Library/Application Support/tarmac/tarmacd.sock` and the *same* `state.json`, with no channel component anywhere. Run both at once and they fight over one socket and one state file: the dev app silently attaches to the released daemon, or the released daemon's `claim_socket` makes the second `tarmacd` `exit(1)`, and both share one `state.json`.

We fix this by making the **derived-default** socket/state path **channel-aware**, where the channel is the binary's own **build configuration**: a *debug* build resolves under a `dev/` subdir (`~/Library/Application Support/tarmac/dev/{tarmacd.sock,state.json}`); a *release* build keeps the **byte-for-byte current flat path** (zero migration, no orphaned boards). The channel is read once per binary through an explicit `Channel` enum — `cfg!(debug_assertions)` in Rust, `#if DEBUG` in Swift — and that token is the **only** thing that differs.

This works with **no env injection, no runtime detection, and no cross-process coordination**, because `make run` builds all three binaries *debug* and `scripts/bundle.sh` builds all three *release*. Each binary self-determines the same channel, so the architecture invariant "one socket, resolved identically by daemon, CLI, and app" (`docs/architecture.md:81`) holds *per channel* by construction. Explicit `TARMAC_SOCKET` / `TARMAC_STATE` overrides keep winning verbatim, so the integration harness and every existing test are untouched.

## Problem

The socket + state path is resolved in three places, with identical logic and **no shared source of truth**:

| Binary | Resolver | Source |
|---|---|---|
| Swift app | `DaemonClient.resolveSocketPath()` | `app/Sources/TarmacKit/DaemonClient.swift:40` |
| Rust daemon | `socket_path()` + `state_path()` | `core/crates/tarmacd/src/main.rs:11`, `:19` |
| Rust CLI | `socket_path()` | `core/crates/tarmac-cli/src/main.rs:27` |

Each resolver is the same two lines: if the env override (`TARMAC_SOCKET` / `TARMAC_STATE`) is set non-empty, return it verbatim; otherwise default to `$HOME/Library/Application Support/tarmac/tarmacd.sock` (and `.../state.json`). There is no channel, bundle-id, or build component — the default is one fixed per-user path.

`make run` sets only `TARMAC_DAEMON` (which debug daemon to spawn) and `PATH`; it does **not** set `TARMAC_SOCKET` or `TARMAC_STATE` (`Makefile:21`). The app's `spawnDaemon` overrides only `PATH` when spawning the daemon (`DaemonClient.swift:207`). So a dev run and a released run resolve to the **identical** socket and `state.json`. Concretely:

- **Dev attaches to release (or vice-versa).** Whichever daemon is up owns the one socket; the other app connects to it. A dev build drives the released daemon's boards and writes the released `state.json`.
- **Singleton lockout.** If both daemons try to start, `claim_socket` sees the live socket and the second `tarmacd` logs *"another tarmacd is already listening"* and `exit(1)` (`tarmacd/src/main.rs:36-37`). You cannot run both daemons.
- **Shared state corruption surface.** Both write the same debounced, atomic `state.json` — two writers, one file.

The goal: **a dev build and the released app run simultaneously, each with its own daemon, fully isolated** — without breaking the existing override mechanism, the tests, or existing release users' data.

## Constraints

These are load-bearing; the chosen approach is shaped by them.

1. **Cross-language resolution.** The path is computed by Swift and by Rust. The Swift↔Rust split is irreducible (no code is shared across the language boundary), but the two Rust binaries (daemon, CLI) **already share the `tarmac-protocol` crate**, so the Rust derivation belongs there **once** rather than copied per-binary — that crate already hosts pure shared helpers (`repo_color_index`) and contract constants (`PROTOCOL_VERSION`, `MAX_FRAME_LEN`). Any scheme must produce the **same** path in every binary, or the app/CLI can't reach the daemon (`docs/architecture.md:81`: "resolved identically by daemon, CLI, and app").
2. **`sockaddr_un` budget.** The bound/connected socket path must be ≤ 103 bytes (Darwin `sun_path` is 104 incl. NUL; the Swift `connect` throws `socketPathTooLong` when `strlen(path) >= 104`, `DaemonClient.swift:180,187`). The current default is ~48–57 bytes for typical homes. Any channel marker must stay within the remaining budget.
3. **Backward compatibility.** Existing release users already have boards in `~/Library/Application Support/tarmac/state.json`. Moving the release path orphans them unless migrated. The cheapest correct answer is to **not move the release path at all**.
4. **Override precedence is sacred.** Explicit `TARMAC_SOCKET` / `TARMAC_STATE` must keep winning **verbatim**. The integration harness injects them per-test (`core/crates/tarmacd/tests/common/mod.rs:69`), and every `m0` test, the `socket_claiming` test (`m0_integration.rs:160`), and `scripts/e2e.sh` depend on it. A channel may alter **only** the derived-default branch.
5. **Standalone CLI has no inherited env.** The released `tarmac` is cask-symlinked onto `PATH` and run from any shell with no tarmac env (`packaging/Casks/tarmac.rb`). It must resolve the correct channel **without** relying on an injected variable.
6. **House conventions.** Path derivation is a **pure** function (table-driven unit tests in `TarmacKit` and the Rust crates) with a thin **impure** shell (env read, `create_dir_all`). The app layer itself is not unit-tested, so the pure Swift logic must live in `TarmacKit`, not `TarmacApp`.

## The key insight: build configuration already *is* the channel

`make run` and `scripts/bundle.sh` already partition the binaries cleanly by build configuration — verified in the scripts:

```make
# Makefile:21 — make run: all DEBUG
run: core app                                  # `cargo build` + `swift build`, both debug
	TARMAC_DAEMON="$(ROOT)/core/target/debug/tarmacd" \
	PATH="$(ROOT)/core/target/debug:$$PATH" \
	"$(ROOT)/app/.build/debug/TarmacApp"
```
```sh
# scripts/bundle.sh:28,31 — the bundle: all RELEASE
cargo build --release --locked --manifest-path "$ROOT/core/Cargo.toml"
( cd "$ROOT/app" && swift build -c release )
```

`scripts/release.sh:50` copies that *same* release `tarmac` to the `.dmg` root for the cask. So **every shipped binary is a release build, and every `make run` binary is a debug build** — across both languages. `core/Cargo.toml` has **no `[profile.*]` override**, so `cfg!(debug_assertions)` is reliably `true` in debug and `false` in `--release`; Swift's `#if DEBUG` tracks the SwiftPM config the same way.

Therefore each binary can determine its channel **from its own immutable build configuration**, with nothing passed between processes. The cross-language resolution constraint (1) — a liability for any env-passing scheme — becomes a non-issue: the three binaries agree because they were built together at the same configuration.

## Proposed approach (Decision)

**Insert a channel subdir into the derived-default branch only, keyed on build configuration, routed through an explicit `Channel` enum.**

### Path scheme

| Channel | Build | Socket | State |
|---|---|---|---|
| **release** | `--release` / non-DEBUG | `~/Library/Application Support/tarmac/tarmacd.sock` *(unchanged)* | `~/Library/Application Support/tarmac/state.json` *(unchanged)* |
| **dev** | debug | `~/Library/Application Support/tarmac/dev/tarmacd.sock` | `~/Library/Application Support/tarmac/dev/state.json` |

- Release is **byte-for-byte identical to today** → zero migration, no orphaned boards, satisfies constraint (3).
- The `dev/` segment adds 4 bytes; a dev socket is ~52–64 bytes for normal homes, well under the 103-byte cap (constraint 2). A boundary unit test pins this.
- Socket and state stay **colocated** under one per-channel dir, so the dir is removable as a unit and `state.json`'s data root (docs/blobs) follows automatically — `state::Daemon::new(state_path())` makes the parent the data root (`tarmacd/src/main.rs:63`).

### Where the logic goes — pure, single audited mapping

The channel branch lives **strictly inside the `else` arm, after the env early-return**, in each resolver. The pure derivation is a **single `channel_dir` helper** that takes the channel as an *argument* (so it's testable in both build configs without compiler tricks) and returns the per-channel directory; callers join the filename they need (`tarmacd.sock`, `state.json`). It lives **once in `tarmac-protocol`** for the two Rust binaries and is mirrored in `TarmacKit` for Swift (the language boundary is the only thing forcing a second copy). Each binary contributes exactly **one thin impure line** mapping build-config → `Channel`:

```rust
// tarmac-protocol (shared crate; tarmacd & tarmac-cli already depend on it) — pure, unit-tested ONCE
pub enum Channel { Release, Dev }
pub fn channel_dir(home: &Path, ch: Channel) -> PathBuf {
    let base = home.join("Library/Application Support/tarmac");
    match ch { Channel::Release => base, Channel::Dev => base.join("dev") }
}

// in each binary: the ONE impure mapping line, audited at the call site, then join the filename:
let ch = if cfg!(debug_assertions) { Channel::Dev } else { Channel::Release };
let socket = channel_dir(&home, ch).join("tarmacd.sock"); // tarmacd + tarmac-cli
let state  = channel_dir(&home, ch).join("state.json");   // tarmacd only
```
```swift
// TarmacKit (Swift) — pure, unit-tested, mirrors the Rust helper + DaemonLaunch.swift's enum-of-static-funcs shape
public enum ChannelPaths {
    public enum Channel { case release, dev }
    public static func channelDir(home: String, channel: Channel) -> String { /* …/tarmac[/dev] */ }
}
// the ONE impure mapping in DaemonClient (TarmacApp side); caller joins the filename:
#if DEBUG
let channel = ChannelPaths.Channel.dev
#else
let channel = ChannelPaths.Channel.release
#endif
let socket = ChannelPaths.channelDir(home: home, channel: channel) + "/tarmacd.sock"
```

One `channel_dir` (not separate `socket`/`state` functions) keeps the path join in a single audited place and matches the colocation argument below — socket and state are just two filenames under the same per-channel dir. So that override-vs-default precedence is itself unit-testable in the one shared place (not re-implemented in each impure shell), `tarmac-protocol` also exposes thin override-aware wrappers `resolve_socket_path(over, home, channel)` / `resolve_state_path(over, home, channel)` that early-return a non-empty `over` verbatim and otherwise delegate to `channel_dir(home, channel).join(..)`; `channel_dir` stays the inner directory helper (no override). The spec's `(override, home, channel)` resolver seam is exactly these wrappers — it mirrors 2606.0002 and keeps S3/S4/S9 unit-testable in both build configs. Routing through `Channel` (rather than sprinkling `cfg!(debug_assertions)` through the path code) is deliberate: it decouples the channel decision from incidental `debug_assertions` semantics and makes the dev/release choice a **single audited point** per binary. We also add an inline comment in `core/Cargo.toml` forbidding `[profile.*]` overrides of `debug-assertions`, because such an override would silently flip the channel without touching the path code at all (see Risks).

### Why this over the alternatives

We evaluated four approaches against constraints (1)–(6); the per-alternative reasoning below ranks them C ≫ A > B > D. The deciding axis is constraint (5): only C resolves the channel on the standalone CLI without an injected variable.

- **A — App injects `TARMAC_SOCKET`/`TARMAC_STATE` before spawn (zero Rust change).** The app picks one value for its connect target *and* the spawned daemon's env, and the in-pty CLI inherits it. **Rejected:** the standalone cask CLI (constraint 5) has *no* inherited env, so it falls back to the default — correct *only* by a coincidence that the release default equals what the app injected. Two independently-edited constants must stay equal forever. It also threads correctness through a 3-hop env chain (app → daemon → pty → CLI) and makes the app the sole source of truth (launch the app any other way and it breaks). More surface, env-coupled, fragile.
- **B — A new `TARMAC_CHANNEL` env var read by all three.** Conceptually clean and future-proof for a third channel, but **something must set it**, and nothing sets it on the hardest path: the standalone CLI has no `TARMAC_CHANNEL`, so it must default — the same coincidence-of-constants fragility as A, *plus* a brand-new env name interacting with the existing override precedence. Strictly more machinery than C for the same result. **Rejected.**
- **D — Hybrid: Swift detects channel via bundle path, Rust via build-config.** Two *different* discriminators that must agree; bundle-path detection is brittle (the unsigned `make bundle` local `.app`, symlinked launches, layout changes). It defends the identical-resolution invariant by coincidence of two unrelated heuristics. No upside over C. **Rejected.**
- **C — Build-config discriminator (chosen).** One signal, shared by all three, fixed at compile time and impossible to lose in transit. Zero env, zero migration, standalone CLI included. The only approach with **no env-dependent failure mode**.

**The trade-off in one sentence:** we bind the channel at *compile time* rather than selecting it at runtime — accepting that a single binary can only ever speak its own channel (you cannot point a debug daemon at the release channel, or vice-versa, except via the explicit `TARMAC_SOCKET`/`TARMAC_STATE` override) in exchange for a channel that is impossible to lose, mismatch, or forget to set across the three independently-built binaries. The two `TARMAC_DAEMON` / mixed-config footguns in the Risks table are the cost side of that trade.

## Non-goals (explicit)

The **build-config layer (C)** isolates exactly **one** dev channel from **one** release channel — and nothing finer. On its own it does **not** isolate two concurrent dev worktrees or two `make run`s of one checkout: both are debug builds → both resolve the same `dev/` path. That finer isolation is handled by the **separate `make run` per-worktree layer** (see "Resolved" below), which sets a per-worktree `TARMAC_SOCKET`/`TARMAC_STATE` on the existing verbatim-override path. The two layers are independent: C is the always-on default; the Makefile layer is a dev-only override on top.

Reaching a per-worktree dev daemon from a **bare shell** `tarmac` (one with no inherited `TARMAC_SOCKET`) is likewise an explicit non-goal: a bare-shell debug CLI resolves the *plain* `dev/` path and so won't find a worktree's daemon. The required diagnostic (resolved path + channel) only makes that miss **legible** — it is not a fix. Drive the CLI from the in-app terminal, or export that worktree's `TARMAC_SOCKET`.

The general escape hatch for any finer isolation remains the existing **`TARMAC_SOCKET` + `TARMAC_STATE` override**, which always wins verbatim.

## Resolved: per-worktree `make run` namespacing (decision = include it)

**Decision (EP, 2026-06-16): include per-worktree namespacing in this change.** You actively use multiple worktrees ([[whiteboard-ux-model]] work, `cjk`, `osc-title`, this one); the core build-config fix gives them all one shared `dev/` channel, so two simultaneous `make run`s collide (second daemon hits the singleton, second app attaches to the first, sharing one `dev/state.json`). `make run` therefore derives a **stable per-worktree** `TARMAC_SOCKET`/`TARMAC_STATE` from `$ROOT`, layered *under* the `dev/` channel — riding the existing verbatim-override path, **no app/daemon/CLI code change**:

```make
# stable 8-hex per-worktree id from $ROOT, hashed ONCE into a recipe-local var, layered UNDER the dev/ channel
run: core app
	wt="$$HOME/Library/Application Support/tarmac/dev/wt-$$(echo -n "$(ROOT)" | shasum | head -c8)"; \
	TARMAC_SOCKET="$$wt/tarmacd.sock" \
	TARMAC_STATE="$$wt/state.json" \
	TARMAC_DAEMON="$(ROOT)/core/target/debug/tarmacd" \
	PATH="$(ROOT)/core/target/debug:$$PATH" \
	"$(ROOT)/app/.build/debug/TarmacApp"
```

Two consequences to handle in the spec:

1. **Byte budget.** The per-worktree segment is `dev/wt-XXXXXXXX/` = 16 bytes (vs 4 for plain `dev/`). Worst-case dev socket suffix after `$HOME` is now ~64 bytes; with a ≤39-byte `$HOME` it stays under the 103-byte `sockaddr_un` cap. This is a `make run`-only path (developer machines), and the `TARMAC_SOCKET` it sets wins verbatim — but the boundary unit test must cover the per-worktree-length case, and `make run` overflow should surface a clear error (not a raw `socketPathTooLong`).
2. **The diagnostic is now load-bearing, not optional.** Because each worktree's dev daemon lives at a *different* per-worktree path, a bare-shell debug `tarmac open` (no inherited env) self-resolves to the *plain* `dev/` path and reaches **no** daemon. The in-pty CLI is fine — it inherits `TARMAC_SOCKET` from its worktree's daemon. So the "print resolved socket path + channel on connect failure" diagnostic (Risks table, rows 2–3) is **promoted from optional to required**: it is the only thing that makes the bare-shell miss self-explanatory rather than a silent no-op. Drive the dev app from a bare shell via the in-app terminal or by exporting that worktree's `TARMAC_SOCKET`.

The core build-config fix and this `make run` enhancement are independent layers — the Makefile change adds zero risk to the core path resolvers (it only sets env vars they already honor).

## Scope of change

- **Rust** `tarmac-protocol`: pure `channel_dir(home, ch)` + `enum Channel`, with the table-driven unit tests (boundary/`sockaddr_un` budget, `release default == legacy path`). Both binaries already depend on this crate, so the derivation and its tests exist **once**.
- **Rust** `tarmacd/src/main.rs`: `socket_path()`/`state_path()` call `channel_dir(..).join(..)` after the env early-return.
- **Rust** `tarmac-cli/src/main.rs`: `socket_path()` calls `channel_dir(..).join("tarmacd.sock")` (CLI has no state); update the `HELP` text (`:18`).
- **Swift** `TarmacKit`: new pure `ChannelPaths.channelDir`; `DaemonClient.resolveSocketPath()` (and a new state derivation, if the app ever needs it) join onto it behind `#if DEBUG`. Table-driven `TarmacKitTests`.
- **Make** `Makefile` `run` target: derive a stable per-worktree `TARMAC_SOCKET`/`TARMAC_STATE` from `$ROOT` (env-only; rides the verbatim-override path).
- **Diagnostic** (now required, not optional): the daemon logs its resolved channel + socket path at startup; `connectFailed` (Swift `DaemonClient`) and the CLI's "no daemon running" message name the **resolved socket path + channel**.
- **Docs** `architecture.md:81`, `protocol.md` transport: the path is now per-channel; the invariant holds per channel.
- **No change** to `claim_socket` (per-socket-path, channel-agnostic — `create_dir_all` makes `dev/` and the per-worktree dir automatically), the cask `zap` (already trashes `tarmac/` recursively), the protocol, or the override branch.

## Success criteria

1. A `make run` dev build and the released `.app` run **simultaneously**, each with its own daemon, on distinct sockets, writing distinct `state.json` files — neither sees the other's boards.
2. Release builds resolve **byte-for-byte** today's flat path (a unit test asserts `release default == legacy path`); existing users need zero migration.
3. Explicit `TARMAC_SOCKET` / `TARMAC_STATE` override wins verbatim **in both build configs** (unit test); the integration harness, `m0` tests, `socket_claiming`, and `e2e.sh` stay green untouched.
4. The dev path stays within the `sockaddr_un` budget (boundary unit test), including the per-worktree `dev/wt-XXXXXXXX/` segment length.
5. All three binaries compute the **same** path for the same channel (recommended integration guard; single-daemon assertion).
6. Two `make run`s from **different worktrees** run simultaneously without colliding (each gets its own per-worktree dev socket + state); a bare-shell debug `tarmac` that can't reach a daemon prints the **resolved path + channel** rather than a silent no-op.
