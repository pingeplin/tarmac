# 2606.0002 macOS Distribution: a Self-Hosted Homebrew Tap plus a Notarized .dmg

**Date:** 2026-06-16
**Status:** draft
**Author:** EP Lin
**Doc type:** RFC

## TL;DR

Distribute Tarmac through a **self-hosted Homebrew tap** whose cask installs `Tarmac.app` to `/Applications` *and* symlinks the `tarmac` CLI onto `PATH`, while publishing the **same notarized `.dmg`** as a direct download on GitHub Releases. The two channels are non-exclusive and share one upstream artifact. The "Homebrew vs. single file" question is real but secondary; the binding decision underneath it is that we must **bundle, code-sign, and notarize** the app regardless of channel, and only Homebrew can put the `tarmac` CLI on `PATH` without making every user run a manual command.

We choose the cask over a bare `.dmg`-only release because Tarmac's defining promise ‚Äî `tarmac open <path>` works from any terminal, Makefile, hook, or CI run ‚Äî depends on the CLI being resolvable on `PATH`, and a dragged-in `.app` runs no install step that could put it there. We keep the direct `.dmg` because it costs nothing extra (it's the cask's own artifact) and serves casual testers with a single link.

## Problem

Tarmac currently has no distribution story. `make run` is the only way to launch it, and it leans on three machine-specific facts injected by the Makefile:

```make
run: core app
	TARMAC_DAEMON="$(ROOT)/core/target/debug/tarmacd" \
	PATH="$(ROOT)/core/target/debug:$$PATH" \
	"$(ROOT)/app/.build/debug/TarmacApp"
```

To hand Tarmac to anyone who is not building from source, we need an artifact a stranger can install and launch, where:

1. The GUI app starts by double-click (no terminal, no `make`).
2. The bundled `tarmacd` daemon is found and auto-spawned without an env var pointing at a debug build dir.
3. The `tarmac` CLI is on `PATH`, so `tarmac open <path>` works from the user's shells ‚Äî including Tarmac's *own* terminals, where agents and Makefiles call it. This is not a nicety; it is the product.

The evidence that (2) and (3) are unmet today is in the code, not hypothetical:

- `DaemonClient.connect()` **throws** `connectFailed` when `TARMAC_DAEMON` is unset or empty (`app/Sources/TarmacKit/DaemonClient.swift:56`). A bundled app with no env var cannot start its daemon.
- `DaemonClient.spawnDaemon()` runs `Process()` with no `environment` override (`:200`), so the daemon ‚Äî and every PTY it spawns ‚Äî inherits the app process's `PATH`. A Finder-launched `.app` gets the minimal launchd `PATH` (`/usr/bin:/bin:/usr/sbin:/sbin`); it does **not** inherit the login shell's `PATH`, and it does **not** include `/opt/homebrew/bin` or any CLI dir. So even after a cask symlinks `tarmac` into `/opt/homebrew/bin`, `tarmac open` fails *inside Tarmac's own terminals* unless the app injects that directory.

## Constraints

- **Solo developer, prototype stage.** No release engineer, no CI fleet assumed. Per-release effort must be near-zero after setup.
- **Two architectures.** Ship for Apple Silicon and Intel ‚Äî universal binaries for both the Swift app and the two Rust binaries.
- **Three programs, one product.** `TarmacApp` (Swift GUI), `tarmacd` (Rust daemon), `tarmac` (Rust CLI) must travel and install together.
- **SwiftPM, not Xcode.** `swift build` yields a bare Mach-O at `app/.build/debug/TarmacApp` ‚Äî no `Info.plist`, no bundle, no signature. There is no Xcode project to lean on for bundling/signing.
- **No paid Apple account or GitHub remote yet.** Both are prerequisites we must stand up; neither exists in the repo today.
- **Gatekeeper is non-negotiable on recent macOS.** On Sequoia (15) the right-click‚ÜíOpen bypass was removed; an unsigned, un-notarized download forces the user through *System Settings ‚Üí Privacy & Security ‚Üí Open Anyway* plus an admin prompt on first launch.

## The work that both channels share (and that the choice is downstream of)

Before any "Homebrew vs. single file" question can matter, the same pipeline must exist under either answer:

1. **Bundle.** Assemble a real `Tarmac.app/Contents/{MacOS,Resources}` with an `Info.plist` (`CFBundleExecutable=TarmacApp`, `CFBundleIdentifier`, `LSMinimumSystemVersion=14.0`, `LSUIElement` left **false** ‚Äî it is a windowed cockpit). The SwiftPM resource bundle `Tarmac_TarmacApp.bundle` (holding `DocTemplate.html`) must be copied into `Contents/Resources`, or `Bundle.module` resolution fails and doc cards silently degrade to plain text.
2. **Embed.** Place universal `tarmacd` and `tarmac` inside the bundle.
3. **Sign, inside-out.** Sign each Mach-O (`tarmacd`, `tarmac`, `TarmacApp`) then the `.app`, each with a Developer ID Application certificate and `-o runtime --timestamp`. Not `--deep` (deprecated since macOS 13). The separately-signable items are exactly three; SwiftTerm is statically linked and its bundle holds only a `Shaders.metal` resource, not a signable binary.
4. **Notarize + staple.** `xcrun notarytool submit --wait`, then `xcrun stapler staple` the `.app`/`.dmg` (a `.zip` can be submitted but not stapled).
5. **Package + publish.** Wrap in a `.dmg`, attach to a GitHub Release, record its `sha256`.

This pipeline costs the **$99/yr Apple Developer Program** and is **identical** regardless of channel. Homebrew does not exempt us from it: as of recent Homebrew releases, casks are quarantined like any download, and the official cask repo will disable Gatekeeper-failing casks beginning September 2026. The takeaway: **signing/notarization is the real work, not the differentiator.** Whoever tells you "just use Homebrew so you can skip signing" is wrong.

## Proposed approach

Build the notarized `.dmg` as the single upstream artifact, then layer a self-hosted Homebrew tap on top of it.

- **Tap.** `brew tap-new pingeplin/tarmac` ‚Üí a public GitHub repo `homebrew-tarmac` with `Casks/tarmac.rb` (the repo MUST be named `homebrew-tarmac`: `brew install <user>/<tap>/<cask>` resolves the tap to `github.com/<user>/homebrew-<tap>`). Users install with one auto-tapping line:
  ```sh
  brew install pingeplin/tarmac/tarmac
  ```
- **Cask, dual-stanza.** One cask delivers both deliverables:
  ```ruby
  app    "Tarmac.app"
  binary "tarmac"          # the standalone universal CLI, symlinked onto PATH
  ```
  We ship `tarmac` as a top-level binary in the `.dmg` that the cask links directly into `$(brew --prefix)/bin`, rather than symlinking into `Tarmac.app/Contents/...` (an in-bundle target can need a shim wrapper). The same `tarmac` binary is also embedded in the app bundle for the app's own `PATH` injection (see below).
- **Direct download.** The identical stapled `.dmg` is attached to each GitHub Release for testers who don't use Homebrew.
- **Updates.** `brew upgrade` via `version` + `sha256` bumps, with `livecheck` tracking GitHub Release assets. No Sparkle on day one.

### The decisive factor: the CLI on PATH

This is the one axis on which the channels genuinely differ, and it decides the call. A cask's `binary` stanza runs an install action that symlinks `tarmac` onto `PATH` automatically. A dragged-in `.app` or mounted `.dmg` runs **no** install action and therefore cannot register the CLI; the user would have to `ln -s` it by hand, and until they did, `tarmac open` would fail for every agent, Makefile, and CI job ‚Äî i.e., the product wouldn't work as advertised. Matching Homebrew's behavior without Homebrew means shipping a signed `.pkg` with a root postinstall script, which is *more* machinery than a cask for a worse first-run experience.

### Two app-code changes this requires (the spec'd slice)

Packaging is mostly scripts, but it forces two genuine behavior changes in the app. These ‚Äî and only these ‚Äî go through the `/spec ‚Üí implement ‚Üí /verify` gate, because they are pure logic extractable into `TarmacKit` and unit-testable there (consistent with the project's "pure logic lives in TarmacKit; the app layer is GUI-verified" convention):

1. **Default the daemon path.** When `TARMAC_DAEMON` is unset/empty, resolve a bundled daemon path relative to the app bundle (e.g. `Bundle.main.bundleURL` ‚Üí `Contents/MacOS/tarmacd`) and spawn that, instead of throwing. An explicitly set `TARMAC_DAEMON` still wins (so `make run` is unaffected); if neither the env var nor a bundled binary resolves, keep today's helpful error. Pure function: `(env, bundleURL) ‚Üí daemonPath?`.
2. **Inject the CLI dir into the spawned daemon's PATH.** In `spawnDaemon()`, set `proc.environment` to the inherited environment with the bundled CLI directory prepended to `PATH`, so the daemon ‚Äî and the PTYs it spawns ‚Äî can resolve `tarmac`. Pure function: `(basePATH, cliDir) ‚Üí newPATH` (prepend, no duplicate).

Everything else ‚Äî `bundle.sh`, the codesign/notarytool/stapler/dmg pipeline, the tap and cask ‚Äî is implementation verified by **downloading the artifact on a clean machine and watching it launch cleanly and resolve `tarmac`**, not by a unit suite. Forcing those through the test gate would only manufacture vacuous tests.

## Alternatives considered

**A. Single-file (`.dmg`/`.zip`) direct download only.** Rejected as the *primary* channel. It is the cheapest to host and fine for the first "send a link to three people" moment, but it structurally cannot put `tarmac` on `PATH`, which is the core promise. It also gives no update channel. We keep it as a *secondary* channel precisely because it's free ‚Äî the cask's own `.dmg`.

**B. Signed `.pkg` installer with a postinstall symlink.** A `.pkg` *can* run a root script that symlinks the CLI, matching Homebrew's PATH behavior without Homebrew. Rejected because it needs a separate *Developer ID Installer* certificate, its own notarization, and more packaging machinery than a cask, while delivering a heavier, less dev-native install (`tarmac`'s audience already has `brew`). It also has no clean uninstall: a `.pkg` postinstall symlink into `/usr/local/bin` is not tracked by the installer receipt, so there is no `brew uninstall --cask`-equivalent that removes both the app and the symlink ‚Äî the user is left to delete the dangling link by hand. (The cert + notarization burden alone already carries this rejection; the uninstall gap is a secondary cost, not the load-bearing one.) Revisit only if we ever need to target non-developer users who lack Homebrew.

**C. Official `homebrew/homebrew-cask` repo.** Rejected (for now) on eligibility, not preference. The official repo's notability bar ‚Äî on the order of 75‚òÖ / 30 forks / 30 watchers, with the thresholds raised substantially for self-submission ‚Äî is out of reach for an unannounced prototype, and the same repo will disable Gatekeeper-failing casks starting September 2026. A self-hosted tap is explicitly exempt from those signing checks and has near-zero setup cost. Re-evaluate if Tarmac ever clears the bar and we want broad discoverability.

**D. Sparkle in-app auto-update.** Deferred, not rejected. Sparkle multiplies the signing surface (its own framework/XPC service), needs an EdDSA-signed appcast regenerated per release, and still wants Developer ID + notarization. `brew upgrade` covers updates adequately for a prototype. Add Sparkle only if update friction is observed to actually hurt non-Homebrew users.

## Trade-offs we accept

We accept a **non-discoverable** distribution channel (a self-hosted tap isn't searchable like the official repo) and **self-owned release bumps** (third-party taps get no BrewTestBot autobump, so each release's `version`+`sha256` is ours to update ‚Äî by hand on day one, with a small CI step or GoReleaser as a later optimization if the manual bump proves error-prone) in exchange for shipping *today* without clearing an official-repo bar, while still solving the load-bearing CLI-on-PATH requirement in a single `brew install`. We also accept paying $99/yr ‚Äî but that cost is forced by Gatekeeper under every channel, so it is not a concession unique to this choice.

## Riskiest assumption

That a Finder-launched, bundled, notarized `Tarmac.app` can reliably **locate and spawn its embedded `tarmacd` by a bundle-relative path** and **hand its PTYs a `PATH` that resolves `tarmac`** ‚Äî i.e., that the two code changes above are sufficient and that App Translocation won't move the bundle to a random read-only path that breaks relative resolution. Stapling and shipping a signed `.dmg` (rather than a bare `.zip`) is the documented mitigation for translocation, which is why the `.dmg` is the artifact of record. The assumption is *broken* if, on a clean machine, the bundle-relative `tarmacd` path fails to resolve after a Finder launch (translocation moved the bundle, or `Bundle.main.bundleURL` points at a read-only shadow) ‚Äî in which case the fallback is an absolute install-location anchor (e.g. resolving via `/Applications/Tarmac.app`) or a `.pkg` postinstall, both heavier than the bundle-relative path this design bets on. If this assumption is wrong, the whole "double-click and it works" story collapses regardless of channel, so it is the first thing to validate on a clean machine ‚Äî before any tap exists.

## Success criteria

- On a **clean macOS machine** (both Apple Silicon and Intel) with no Tarmac env vars, the user runs `brew install pingeplin/tarmac/tarmac`, launches Tarmac from `/Applications` by double-click, and it opens with **no Gatekeeper prompt**.
- The daemon auto-spawns with **no `TARMAC_DAEMON` set**, and a doc card renders from `DocTemplate.html` (proving the resource bundle is found).
- In a fresh login shell *and* inside one of Tarmac's own terminals, `tarmac open <path>` surfaces a card ‚Äî proving the CLI is on `PATH` both ways.
- A new tagged release is installable via `brew upgrade` with only a `version`+`sha256` change to the cask.

## Out of scope

- Sparkle / in-app auto-update (Alternative D ‚Äî deferred).
- Submission to the official `homebrew-cask` repo (Alternative C ‚Äî gated on notability).
- A `.pkg` installer (Alternative B ‚Äî only if we target non-Homebrew users).
- App Store distribution (different signing/sandboxing regime entirely).
- Windows/Linux distribution (Tarmac is macOS-only).
- The internals of the signing/notarization shell pipeline ‚Äî captured here as a sequence, detailed in the implementation, not specified as testable behavior.
