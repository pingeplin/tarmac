# 2606.0003 Defer the Card Restack Until Mouse-Up So Terminal Selection Survives Click-to-Focus

**Date:** 2026-06-16
**Status:** implemented
**Author:** EP Lin
**Doc type:** ADR

## TL;DR

Click-to-focus tears every card's view out of the hierarchy (`removeFromSuperview` + re-add) one runloop tick after a press ŌĆö exactly as a terminal text-selection drag begins ŌĆö which severs the drag and makes selecting terminal text impossible. We **defer the view-hierarchy reorder while any mouse button is physically down** and replay it on mouse-up, mirroring the existing `isGesturing` guard that already protects card move/resize drags. The `z` bump that establishes stacking order still happens immediately, so order stays logically correct; only the harmful `removeFromSuperview` is postponed. We accept a one-runloop visual delay in raising a *background* card to the front (it now pops up on release rather than on press) in exchange for selection that doesn't break.

## Context

The reported symptom: **you cannot select text inside a terminal card with the mouse.** Click-drag over the SwiftTerm grid highlights nothing. This reproduces on `main` ŌĆö it is a pre-existing app-level bug, unrelated to any SwiftTerm version bump.

SwiftTerm does drag-selection the ordinary AppKit way: `mouseDown` arms the selection and each following `mouseDragged` extends it. That only works if the SwiftTerm view keeps receiving the events that follow its `mouseDown`. `removeFromSuperview` during an active mouse-tracking session severs tracking ŌĆö the view stops getting `mouseDragged`/`mouseUp`, and the selection never extends.

Our click-to-focus path does exactly that, at exactly the wrong moment:

1. Press inside the terminal ŌåÆ SwiftTerm gets `mouseDown` and arms its selection.
2. The click-focus monitor (`AppController.start()`, matching `.leftMouseDown`) schedules `handleClickFocus` for the **next runloop tick** and returns the event unswallowed. The defer is deliberate and correct ŌĆö its purpose is to let terminal selection start normally. The problem is *what it defers into*.
3. One tick later ŌĆö as the drag begins ŌĆö `handleClickFocus` ŌåÆ `focus()` ŌåÆ `bringToFront(id)` ŌåÆ `raiseToFront(card)`.
4. `raiseToFront` bumps the card's `z` **even when it is already frontmost** (the guard was `if card.worldFrame.z <= maxZ` ŌĆö `<=`, so a top card still gets bumped), then calls `restack()`.
5. `restack()` does `removeFromSuperview()` + re-add on **every** card, including the one SwiftTerm is mid-selection on. Tracking severs. Selection dies.

Two facts about the scope make this worse than it sounds:

- **It fires on every terminal click**, not just the first. Because step 4 bumps `z` even for an already-frontmost card, the restack runs even on the currently-focused terminal and even when it is the only card on the board. That matches the "I can't select at all" report rather than "only on first click."
- **The codebase already knew about this hazard.** `raiseToFront` carried `guard !isGesturing else { return }` and a comment documenting "don't restack mid-drag." But `isGesturing` is true only during a card **move/resize** (which sets `gesturingID` via `beginMove`/`beginResize`). A terminal **selection** drag lives entirely inside SwiftTerm, never touches the card gesture system, so `gesturingID` stays nil and the guard waves it straight through. The fix closes a known gap, it doesn't discover a new mechanism.

There is no unit seam for AppKit mouse tracking ŌĆö per the project's testing reality, only TarmacKit is unit-tested; the app layer is verified by hand and by review. So this decision has to be defensible on its reasoning, not on a passing test.

## Decision

In `raiseToFront`, after the existing `isGesturing` short-circuit, add a second guard: if `NSEvent.pressedMouseButtons != 0`, set a `pendingRestack` flag and return *before* `restack()`. The `z` bump above the guard still runs, so stacking order is already correct in the model ŌĆö only its replay into the view hierarchy is postponed.

Replay happens on button release. A fourth local event monitor (`.leftMouseUp`, an `NSEvent.addLocalMonitorForEvents` like the other three) calls `board.flushPendingRestack()`, which runs the deferred `restack()` once and clears the flag. The monitor **never swallows** the event ŌĆö the selection's own `mouseUp` must still reach SwiftTerm to finalize the highlight.

This is the same shape as the `isGesturing` guard it sits beside: "a drag may be tracking a view; don't pull that view out of the hierarchy until the drag ends." It just covers the second kind of drag ŌĆö the one SwiftTerm owns ŌĆö that the original guard couldn't see.

## Alternatives considered

**Stack via `layer.zPosition` instead of reordering subviews.** This would avoid `removeFromSuperview` entirely ŌĆö the most direct way to never sever tracking. Rejected because AppKit **hit-testing follows subview order, not `zPosition`**. Overlapping cards would render in one order and hit-test in another, so clicks would land on the wrong card. Subview order has to stay authoritative; we can only defer *when* we reorder, not stop reordering.

**Tighten the z-bump from `<=` to `<` so an already-frontmost card never restacks.** Cheaper, and it fixes the most common case (clicking the terminal you're already in). Rejected as a *complete* fix because it does nothing for selecting in a **background** terminal: there the card genuinely isn't frontmost, the restack is legitimately wanted, and it would still fire mid-drag and sever the selection. It treats a symptom of step 4 while leaving the real hazard in step 5. (We did still want the bump to stop being gratuitous, but the deferral is what actually makes selection safe in every case.)

**Suppress click-to-focus entirely for terminal cards.** Would dodge the restack, but it breaks the board's single-click-to-focus model for the most-used card type and couples the focus system to card kind. Too broad a behavior change to fix a reorder-timing bug.

## Consequences / trade-offs

We accept that **raising a background card to the front is now deferred to mouse-up**. For a plain click the gap is one runloop tick and imperceptible. The visible case is click-*dragging* on a not-yet-focused card: the card stays behind its neighbors until you release, then pops forward. We judge this a fair price ŌĆö and arguably more correct, since the user is mid-gesture ŌĆö for selection that survives.

Costs added: one more local event monitor (now four, all armed in `start()` and torn down in `shutdown()`) and one piece of mutable view state (`pendingRestack`). Both are small and local to the existing monitor/gesture machinery.

The move/resize path is unaffected: `isGesturing` still short-circuits first, so those drags never reach the new guard.

**First-responder ordering ŌĆö a consequence we had to repair.** `focus()` deliberately runs `restack()` (via `bringToFront`) *before* `setPrime` calls `makeFirstResponder`, so the responder assignment is the last word. Deferring the restack to mouse-up inverts that: the restack now runs *after* `focus()` has already made the selected terminal first responder, and `removeFromSuperview` on its card resigns it. The symptom is subtle ŌĆö selection works, but the terminal silently loses keyboard input and `ŌīśC` copy right after a select-drag, because the Edit-menu `copy:`/`paste:` actions have no first responder to land on. The deferred flush therefore has to capture the first responder and restore it across the restack churn (`flushPendingRestack`). The lesson generalizes: any code that moves a hierarchy-mutating step later in time must also preserve whatever responder state the original ordering guaranteed.

## Riskiest assumption

**That the deferred `restack()` always gets flushed** ŌĆö i.e., that a `.leftMouseUp` reliably reaches the local monitor for every press that set `pendingRestack`. A *local* monitor only sees events routed to this app, so the concrete miss-case is real: if the button is released while the pointer is over another app (the press started in-window but the drag exited), or the event is consumed upstream, our monitor never fires, the flag stays set, and the view-hierarchy order stays stale until the *next* `raiseToFront` flushes it implicitly. This is low-severity by construction ŌĆö `z` is already logically correct, so the model isn't wrong, only the subview order lags ŌĆö but it is the thing that could surprise a future reader, so it's stated rather than hidden. If it ever bites, the fix is to also flush on focus changes or window deactivation, not to undo the deferral.

## How we verified

By hand (no unit seam exists for this layer):

1. `make run`, open a terminal card at a shell prompt (no mouse-reporting TUI), click-drag across text ŌåÆ it highlights and stays highlighted.
2. `ŌīśC`, paste elsewhere ŌåÆ confirms the selection is real, not just drawn.
3. Regression checks the deferral must not break:
   - Click a background card ŌåÆ it still comes to the front (on release).
   - Header-drag to move, corner-drag to resize ŌåÆ unaffected (they short-circuit on `isGesturing` first).
   - Double-click (word) and triple-click (line) selection still work.
   - Click an already-focused terminal, then immediately drag-select ŌåÆ works.

`make` (Rust core) and `make app` (Swift) both build clean with the change.

## Out of scope

Selection is still ŌĆö correctly ŌĆö suppressed when the running program has **mouse reporting** on (vim/tmux/`less` with mouse). SwiftTerm forwards those drags to the pty by design; that is expected terminal behavior, not part of this bug, and the fix does not touch it.
