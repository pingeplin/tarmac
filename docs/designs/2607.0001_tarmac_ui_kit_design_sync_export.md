# 2607.0001 Tarmac UI Kit: a standalone design-sync export

**Date:** 2026-07-01
**Status:** approved
**Author:** EP Lin
**Doc type:** Mini RFC
**Issue:** #47 (extract a consumable "Tarmac UI kit" and sync it to claude.ai/design)

## TL;DR

Package the already-pure presentational slice of the app — the Breeze tokens, 8 `src/ui/` components (TitleBarChip is dropped as dead code), and `cards/{CardShell,DocCard}` — as a **standalone, browser-loadable kit** that `/design-sync` uploads to a Claude Design project, so the design agent composes with our real compiled parts instead of generic look-alikes. The engineering is *packaging, not decoupling*: the code is already prop-driven with no IPC/Tauri/xterm coupling. The work is (1) a second, esbuild-scripted build target that emits an IIFE bundle + one stylesheet + fonts into `desktop/dist-kit/` **without touching** the app's `tsc && vite build`; (2) splitting `theme.css` into `@import` partials so the kit consumes only the token+card+chrome closure while the app keeps rendering byte-identically; (3) bundling IBM Plex Mono (today assumed system-installed); (4) neutralizing three static-preview gotchas so a component renders correctly with **no live daemon**; and (5) emitting one `@dsCard`-marked preview HTML per component for the DesignSync manifest. The load-bearing bet: if a standalone preview renders wrong (collapsed prose, wrong font), that error propagates into every design the agent generates — so preview fidelity is the whole point, not a nicety.

## Problem

Designs explored in Claude Design use generic components, so what comes back drifts from our real UI when implemented — every generated design currently needs a manual re-implementation pass against the real components before it's shippable, which is the cost this kit buys down. The fix is to give the design agent our actual compiled components + tokens. Scoping already verified the target slice is import-clean: none of the 8 live `ui/` components, nor `CardShell`/`DocCard`, reach into `ipc/`, `board/`, `cards/TerminalCard`, `@tauri-apps/*`, or `xterm` — their only non-React edges are a handful of pure, dependency-free `kit/*.ts` modules plus `theme.ts` and `marked`. So there is nothing to *decouple*. What's missing is a **delivery format**: the components only exist today inside the Vite app graph, styled by a 1060-line `theme.css` entangled with board-engine and terminal rules, using a chrome font the repo never actually ships, and depending on three runtime CSS-var writers (`BoardEngine`, `docCardVars()`, the `.board` ancestor) that don't exist when a component is rendered in isolation.

## Constraints

- **`make app` and `make core` must be untouched.** The Tauri app builds via `tsc && vite build` → `dist/`, and `tauri.conf.json` sets `frontendDist: "../dist"`. Anything the kit build writes into `desktop/dist/` would collide with the app bundle. `make` must still pass and existing app tests must be unaffected.
- **Wire protocol / daemon are irrelevant here** — this is a frontend-only export with no socket, no `Msg` change, no conformance-vector churn. It aligns with Tarmac's "every mark backed by an OS fact" ethos only loosely; it's design tooling, not a runtime surface.
- **One source of truth for CSS.** The app and the kit must render from the *same* rules; duplicating `.card`/token definitions would guarantee drift, which is the exact problem we're solving.
- **DesignSync's contract is fixed:** the Design System pane builds its card index from each preview HTML's first-line `<!-- @dsCard group="…" -->` comment (compiled into `_ds_manifest.json` by the app's self-check). Explicit `register_assets` is legacy. So the kit's job is to emit marker-carrying preview HTML plus the JS/CSS/font assets those previews load.

## Decisions

### (a) Build tool: a standalone esbuild script, output to `dist-kit/`

Add `esbuild` as an explicit devDependency (it's already in the tree transitively under Vite) and a Node script `desktop/scripts/build-kit.mjs` that calls esbuild's JS API, wired to a new `npm run build:kit` and a new `.PHONY: kit` Makefile target — never folded into `app`.

- **Format:** a single IIFE bundle `dist-kit/tarmac-kit.js` exposing `window.TarmacKit = { …8 ui components…, CardShell, DocCard, mount }`, with **React, react-dom, and marked inlined** (not marked external). Off-machine previews get one `<script>` and no import map.
- **Output dir:** `desktop/dist-kit/`, **not** `dist/`.

> **Trade-off:** a scripted esbuild IIFE with React inlined is chosen over Vite library mode because DesignSync previews need one self-contained, global-exposing file loadable from a bare HTML page, and a separate script leaves the app's `tsc && vite build` and Tauri's `frontendDist` completely alone — at the cost of a second, hand-maintained build entry point that isn't exercised by `make app`. (The ticket says "emit dist/"; we deliberately read that as a *kit* dist directory and name it `dist-kit/` to avoid the app-bundle collision.)

### (b) Split `theme.css` by `@import`-closure membership, not by comment section

Move every rule out of `theme.css` into partials under `src/theme/`, and reduce `theme.css` to an ordered list of `@import`s **in the original top-to-bottom order** (so the cascade and specificity ties resolve identically). Partitions:

- `tokens.css` — the ~30 genuine design tokens, `--oversample-k`, and the three engine-var **defaults** (`--zoom:1; --world-tx:0px; --world-ty:0px;`) that already live in `:root` today, plus the IBM Plex Mono `@font-face` (see (c)).
- `card.css` — `.card` + state classes, `.card-header` children, `.card-handle*`, `.repo-dot` (kept unscoped, as today), and all `.doc-card`/`.doc-*` prose rules.
- `chrome.css` — the 8 in-scope `ui/` component rules, including `.dock-pane`/`.dock-header`/`.dock-body` *box* styling (DockPane is in scope).
- `app-only.css` — board-engine/world-layer/edge rules, JetBrains `@font-face`, and every terminal/reparent rule: both `.term-host` definitions (the base one at 334–342 and the `.dock-body .term-host` override), `.card.docked`, `.term-raster-*`, `.term-dock-ghost`.

The kit builds from `tokens.css + card.css + chrome.css` (+ the preview-defaults from (d)); the app's `theme.css` `@import`s **all** partials, so the app is unchanged.

> **Trade-off:** we split by closure membership into shared partials that both `theme.css` and the kit stylesheet `@import`, rather than duplicating rules or splitting mechanically by the section comments, so every shared selector has exactly one definition and app rendering stays byte-identical — at the cost of hand-curating the ~6 cross-cutting selectors the research flagged (`.term-host` defined twice, `.card.docked` living in the terminal block, the parallel `.doc-card.prime/.lifting/.fresh` shadow system, `.repo-dot`'s deliberate un-scoping) that don't fall on the comment boundaries and would be misfiled by a naive cut.

### (c) Bundle IBM Plex Mono as real font files

Vendor IBM Plex Mono **Regular (400)** and **Bold (700)** as `.woff2` (OFL-licensed, redistributable) into `desktop/public/fonts/`, and add their `@font-face` rules to `tokens.css` with `src: url('/fonts/ibm-plex-mono-*.woff2')`. Because the rule lives in the shared token partial, both the kit *and* the shipping app gain it.

> **Trade-off:** we bundle Plex into the shared `tokens.css` `@font-face` (not a kit-only file) because doing so also closes a latent bug in the shipping app — the chrome font is currently assumed system-installed and silently falls through to `SFMono` on Macs without it — accepting ~2 small `.woff2` files (~120 KB) added to the Tauri webview assets. `.woff2` over the `.ttf` used for JetBrains keeps the kit download small; JetBrains itself is **not** bundled into the kit, since no in-scope component renders a terminal surface.

### (d) Neutralize the three static-preview gotchas — exact file changes

1. **Engine vars (`--zoom`/`--world-tx`/`--world-ty`).** *No change needed beyond the split.* These already have `:root` defaults of `1`/`0px`/`0px` in `theme.css:46–48`; carrying them into `tokens.css` (which the kit stylesheet imports) means a preview with no `BoardEngine` writer renders unzoomed/untranslated, silently and correctly.
2. **DocCard prose collapse (`--card-w`/`--card-h`).** These have **no** `:root` fallback, so without `docCardVars()`'s inline write the `.doc-prose-sizer` width resolves via invalid `calc()` to `auto` and prose collapses. Fix: a **kit-only** partial `src/theme/preview-defaults.css` sets `:root { --card-w: 360px; --card-h: 520px; --card-x: 0px; --card-y: 0px; }`. It is imported by the kit stylesheet **only**, never by the app's `theme.css`, so the app's `:root` is unchanged and its per-card inline vars still win where they exist; standalone DocCard prose gets a real non-zero width.
3. **DocCard needs `.board` + `marked`.** `marked` is inlined into the IIFE by esbuild (decision (a)), so `marked.parse` just works. The `.board` ancestor is supplied by the **DocCard preview HTML wrapper** (`<div class="board"><div id="mount"></div></div>`); `DocCard`'s `closest('.board')` MutationObserver then finds an ancestor and — because nothing mutates `--zoom` — simply never fires, which is the correct static behavior (initial layout renders; no re-layout).

Files touched for (d): new `src/theme/preview-defaults.css`; the DocCard preview template (generated by `build-kit.mjs`) carries the `.board` wrapper; no source-component edits.

`preview-defaults.css` also restates a `body { font-family/color/background }` rule (kit-only, mirroring the `html,body,#root` setter that the split correctly files under `app-only.css` as page-shell scaffolding): a bare preview page has no `#root` element to inherit the chrome font from, so without this restatement every preview would render in the browser's serif default instead of Plex.

### (e) What the kit emits for `/design-sync`

`npm run build:kit` writes `desktop/dist-kit/`:

```
dist-kit/
  tarmac-kit.js                 # IIFE → window.TarmacKit (components + mount), React+marked inlined
  tarmac-kit.css                # compiled @import closure: tokens + card + chrome + preview-defaults
  fonts/                        # ibm-plex-mono-{regular,bold}.woff2
  components/<name>/index.html  # one per component, first line = <!-- @dsCard group="…" -->
```

Each `components/<name>/index.html` is tiny and declarative: first-line `@dsCard` marker, then `<link rel=stylesheet href=../../tarmac-kit.css>` + `<script src=../../tarmac-kit.js>`, a mount node (wrapped in `.board` for DocCard), and one line calling `TarmacKit.mount("<name>", <sampleProps>, el)`. Sample props are fixtures authored from each component's documented props interface — this fixture set is the real bulk of the work. Groups: `Cards` (CardShell, DocCard), `Overlays` (BoardSwitcher, ToastOverlay, OffscreenHints, MinimapOverlay, CycleHud), `Chrome` (StatusBar, ZoomControl, DockPane).

> **Trade-off:** we rely on the `@dsCard` first-line marker → `_ds_manifest.json` compilation path and skip explicit `register_assets`, because that's the tool's current, non-legacy contract and keeps the kit output a plain static bundle with no registration step — accepting that every preview *must* carry a well-formed first-line comment or it silently won't index.

## Alternatives rejected

- **Vite library mode (`build.lib`).** Rejected: it's designed for ESM/CJS npm packages consumed by another bundler, not a single global-exposing browser file with React inlined, and mode-gating it into the existing `vite.config.ts` risks perturbing the app build we're required to leave alone. A standalone esbuild script is a clean, separate seam.
- **Duplicate the token/card CSS into a kit-only stylesheet.** Rejected: two copies of `.card`/tokens is exactly the drift the project exists to kill; the `@import`-partial split gives the kit its subset with zero duplication.
- **Add `--card-w`/`--card-h` defaults to the shared `tokens.css`.** Rejected in favor of a kit-only `preview-defaults.css`: keeping the neutralizing fallbacks out of the app's `:root` is the strongest guarantee that "existing app rendering unaffected" holds literally, and the previews are the only consumer that needs them.
- **Ship a runtime harness that fakes a `BoardEngine`** to drive `--zoom` on previews. Rejected: over-engineered — static previews want a fixed, correct frame, not animation; the `:root` defaults already deliver that.

## Assumptions and trade-offs

- **Riskiest assumption — preview fidelity.** The whole value rests on each standalone preview rendering *pixel-faithful to the app* (correct font, tokens, non-zero prose). If a preview renders wrong, that error propagates into every design the agent generates from the kit — worse than not syncing at all. The three neutralizations in (d) plus the font bundle in (c) exist specifically to hold this; the acceptance gate must verify DocCard prose at non-zero width, not just "it renders."
- **DockPane's contract stays imperative.** DockPane's own chrome ships, but its `.dock-body` is, in the app, a bare DOM node the caller reparents an xterm host into. The preview renders the empty pane; consumers must know `.dock-body` is not for React children. Documented, not enforced.
- **MinimapOverlay isn't purely CSS-themeable.** It computes SVG fills from `theme.ts`'s `palette` in JS (SVG can't read CSS vars), so reskinning the minimap requires overriding the bundled `palette`, not just CSS custom properties. Accepted as a known kit limitation; the default palette travels in the bundle so the preview is correct out of the box.
- **TitleBarChip is dropped.** It's imported nowhere and its only class `.title-chip` doesn't exist in `theme.css`; exporting unstyled dead code would mislead the design agent. If it's ever revived, add it back with real CSS first.
- **Second build entry is unversioned by CI.** There's no CI in this repo, and `make` won't run `build:kit`, so the kit build can rot without anyone noticing until a sync fails. Accepted for now; `make kit` is the manual gate.

## Success criteria (observable)

- `make` still passes; `make app` output and `desktop` + app-cargo tests are unchanged (the CSS split is `@import`-only and preserves cascade order; no component source is edited except optional fixture files).
- `npm run build:kit` (and `make kit`) emit `dist-kit/` containing `tarmac-kit.js`, `tarmac-kit.css`, `fonts/`, and one `@dsCard`-marked HTML per component; the stylesheet's `@import` closure contains **only** tokens + card + chrome (+ preview-defaults) — no board-engine or terminal rules.
- Each in-scope component renders standalone from props with **no live daemon**, correct IBM Plex Mono chrome, and correct tokens — including DocCard prose at non-zero width.
- `/design-sync` consumes `dist-kit/`, the app's self-check compiles `_ds_manifest.json` from the `@dsCard` markers, and the kit uploads to a Claude Design project.

## Out of scope

- `cards/TerminalCard`, `board/Board` + `BoardEngine`, and `App.tsx` — behavioral, not statically renderable from props.
- Any daemon/protocol/socket change.
- Wiring `build:kit` into `make app`/`make core` or into CI.
- A published npm package (ESM/CJS) — the target is a browser bundle for DesignSync, not a dependency for other repos.
- Reworking MinimapOverlay to be CSS-themeable, or reviving TitleBarChip.

## Likely PR phasing

1. **CSS split** — move rules into `src/theme/*` partials, reduce `theme.css` to ordered `@import`s; verify app renders identically. (No behavior change; smallest reviewable unit.)
2. **Font bundle** — vendor Plex `.woff2` + `@font-face` in `tokens.css`.
3. **Build target** — `build-kit.mjs`, `esbuild` devDep, `mount()` export, `npm run build:kit`, `make kit`, `preview-defaults.css`.
4. **Previews + fixtures** — per-component sample props and `@dsCard` HTML templates; first `/design-sync` upload.
