# 2606.0005 Doc Card Close: Teardown Across Glass and Daemon

**Date:** 2026-06-30
**Status:** implemented
**Author:** EP Lin
**Doc type:** RFC
**Issue:** #34 (close button on doc card is a no-op)

## TL;DR

The close ✕ on a doc card does nothing, and even when the card *does* leave the board (via ⌘W) the daemon never forgets the doc. Two layered bugs: a frontend event no-op (the ✕'s `onClick` never fires because the header captures the pointer first), and a daemon state leak (no protocol message tells the daemon to close a doc, so `Registry.docs`/`dock` keep the entry, the notify watcher keeps firing, and the doc reappears after restart). Fix both: `stopPropagation` on the ✕'s `pointerdown` so the click lands, and a new additive `Msg::DocClose { path }` that prunes registry state, persists, and unwatches the parent dir. "Closed" must mean the card is gone *and* the OS facts behind it (the watch, the persisted entry) are gone too.

## Problem

### Bug 1 — frontend no-op (the reported symptom)

`DocCard.tsx` renders the close control as `<span className="close" onClick={props.onClose}>`. But `CardShell.tsx`'s `.card-header` has an `onPointerDown` that, on a left press (`e.button === 0`, `onMove` set — both true here), calls `setPointerCapture(e.pointerId)` on the header element (the drag-to-move handle). The ✕ span is a descendant of the header, so on press the pointer is captured by the header; in WebKit the subsequent `click` is dispatched at the capture target (`.card-header`), not at the `.close` descendant. `onClose` never fires. The ⌘W keyboard path calls the same `removeDoc` and works — only the mouse button is dead.

### Bug 2 — daemon state never torn down

There is no `DocClose`/`DocRemove` variant in the `Msg` enum; `Layout` is the only app→daemon channel for dock/tile state. Closing a card (even via ⌘W) only mutates frontend `cards`/`dockOrder`; it cannot express "forget this doc" to the daemon. Worse, `Registry::apply_layout` deliberately *re-appends* any registered doc missing from the incoming dock snapshot back onto `self.dock`, and never deletes from `self.docs`. So:

- The closed doc survives in `Registry.docs`/`dock` and reappears in the next `Restore`.
- `ensure_watched` is additive-only — there is no `unwatch()` anywhere — so `watch_loop` keeps emitting `FileEvent` for the closed doc's parent dir, since `reg.docs.contains_key` still matches.
- Minor app-side leak: `BoardState.docMeta` and `docContents` cache entries are not pruned on `removeDoc`.

The SVG provenance edge is *not* a bug — `EdgeLayer` derives edge keys from live card models, so React drops the edge automatically when the card leaves `cards`.

## Decision

Fix both layers; make `DocClose` a first-class, persisted, additive protocol message.

### Frontend — make the click land

On the `.close` span, handle `onPointerDown` with `e.stopPropagation()` (and `preventDefault`), then fire the close. Stopping propagation on the span means the event never reaches the header's `onPointerDown`, so the header never captures the pointer and the close fires reliably.

**Alternative considered:** inspect `e.target` inside `CardShell`'s `onHeaderPointerDown` and skip `setPointerCapture` when the target is (or is within) `.close`. Rejected: it pushes knowledge of the ✕ up into the generic shell, couples the header to a specific child, and is fragile to future header controls. `stopPropagation` on the ✕ is local to the thing that owns the behavior and reads as "this control consumes its own press."

### Protocol — add `Msg::DocClose { path }`

Additive new variant carrying the doc's absolute path (the registry key). Per the additive-only contract: new message type, no existing key changed, unknown types ignored by older peers. Encode with `rmp_serde::to_vec_named`. **Add a new conformance vector** for `DocClose` — per `docs/protocol.md`, vectors assert decode-equivalence (not byte-exactness) and a new message type leaves the existing V1–V9 vectors unchanged; add the next vector (V10) for the new variant (the desktop backend path-deps the same crate, so there is no second codec to sync). The `socket_path()` duplication across `tarmacd` and `tarmac-cli` is untouched here (no CLI verb added).

### Daemon — handle `DocClose` in `conn.rs`

On receipt:
1. Remove `path` from `Registry.docs` and `Registry.dock`.
2. Mark the registry dirty → persist `state.json`.
3. Unwatch the parent dir **iff** no other open doc shares it (compute remaining `docs` whose parent equals this path's parent before calling `unwatch()`). This adds the first `unwatch()` caller; keep it symmetric with `ensure_watched` and respect the lock discipline — the watcher `Mutex` must not be held across an `.await`.

`DocClose { path }` is **idempotent**: an unknown/already-removed path is a silent no-op (no error, no persist), matching "already torn down" semantics and tolerating double-clicks, a stale frontend, or a `DocClose` racing `Restore`.

### Frontend cleanup

In `removeDoc`, also prune `docMeta` and `docContents` for the closed path, and send the `DocClose` request over the daemon IPC (new `doc_close` Tauri command + `daemon.ts` sender). The ✕ button and ⌘W both route through `removeDoc`, so both paths get full teardown.

## Alternatives rejected

- **Frontend-only fix** (just make the ✕ fire `removeDoc`). Rejected: it satisfies the visible symptom but not the issue's persistence/watch-cleanup requirement — the doc still reappears on restart and still leaks `FileEvent`s.
- **Overload `Msg::Layout` to express removal** (omit the doc from the dock snapshot and let the daemon infer deletion). Rejected: `apply_layout` *deliberately* re-appends docs missing from the snapshot (so a transient partial layout never drops a registered doc). Removal-by-absence directly conflicts with that semantics; a dedicated explicit message is unambiguous and leaves `apply_layout` intact.

## Assumptions and trade-offs

- **Path is the stable registry key.** `DocClose { path }` assumes the absolute path uniquely identifies the doc in `Registry.docs` — consistent with how `open` registers and how watching is keyed by parent dir.
- **Additive bump cost.** A new `Msg` variant forces a conformance-vector regen (mandatory, gated test). Cheap and one-time, but non-zero; accepted because expressing close as data is the only contract-clean way to persist the teardown.
- **Unwatch is conditional, not eager.** Multiple docs can share a parent dir; unwatching unconditionally would silence a still-open sibling. The "iff no sibling shares the parent" check is the correctness pivot.

## Success criteria (observable OS facts)

After clicking the ✕ (or ⌘W):
- The card is gone from the board, and its dashed provenance edge is gone.
- No further `FileEvent`s are emitted for the closed doc's path (the parent dir is unwatched when no sibling needs it).
- The path is absent from `state.json`.
- After a daemon restart + `Restore`, the doc does **not** reappear.
